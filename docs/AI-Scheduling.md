# AI 发言调度机制

## 目标

AI 发言调度的目标不是让 AI 高频填充对话，而是让 AI 在合适的时机像真人一样发言：

- 信息少时可以沉默。
- 被点名或被质疑时可以较快回应。
- 普通接话可以有自然延迟。
- 冷场时可以晚一点补话。
- 生成期间如果出现新消息，不能把基于旧上下文的发言直接发出去。

## 为什么不使用固定概率

旧规则是固定 `4-10s` 检查一次，并以 `55%` 概率尝试发言。这种方式能跑通 MVP，但会带来几个问题：

- AI 容易抢话，尤其是小局里会显得系统在推进对话。
- 低信息场景也会触发模型，容易生成“先看看大家反应”这类机械发言。
- 节奏稳定且可预测，不像真人。
- 两个 AI 容易连续帮腔。
- 发言时机只由随机数决定，没有利用上下文。

## 为什么不做 1 秒模型轮询

即使不考虑模型成本，也不建议每秒调用模型判断是否发言：

- LLM 不适合作为实时调度器；时间、并发、冷却和阶段约束仍需要工程层保证。
- 模型有返回延迟，返回时上下文可能已经过期。
- “是否允许问 AI”包含产品规则，不完全是语义判断。
- 高频询问会让模型倾向于不断寻找发言理由。

更合理的边界是：

```text
工程层：决定什么时候允许询问 AI，并保证状态合法。
AI 层：决定现在要不要说、目标反应时间、下次观察时间，以及如果说该怎么说。
```

## 第一版方案

第一版尽量减少工程化控制，只保留硬约束和上下文有效性检查。

工程层保留：

- 当前必须是 `discussion` 阶段。
- AI 玩家必须存活。
- 同一房间同一时间只允许一个 AI 生成发言。
- AI 自己必须满足发言冷却。
- AI 返回的时间要做上下限裁剪。
- 模型生成期间或等待发送期间有新消息，则丢弃旧回答并短延迟后重新观察。

AI 策略层负责：

- 判断现在是否发言。
- 如果发言，给出 `targetResponseDelayMs`，表示从看到当前局势到最终发言出现的目标总反应时间。
- 给出 `nextCheckAfterMs`，表示本次决策后多久再次观察局势。
- 生成表达层需要的结构化策略。

## 策略层输出

发言：

```json
{
  "type": "speak",
  "targetResponseDelayMs": 2500,
  "nextCheckAfterMs": 10000,
  "strategy": {
    "replyTo": "3号说我机械",
    "speechAct": "防守反问",
    "publicPoint": "我只是催2号说句话，不足以说明机械",
    "tone": "有点不服，但别长篇解释",
    "maxSentences": 2,
    "constraints": ["不要直接否认 AI 身份", "不要同时点评多人"],
    "avoidPhrases": ["先看看大家反应", "带节奏", "有点可疑"]
  }
}
```

跳过：

```json
{
  "type": "skip",
  "reason": "信息太少，先不说",
  "nextCheckAfterMs": 12000
}
```

## 时间处理

`targetResponseDelayMs` 不是模型返回后的额外等待，而是目标总反应时间。

工程层会记录调用开始时间：

```text
remainingDelayMs = targetResponseDelayMs - modelElapsedMs
```

如果模型已经耗时超过目标时间，则不再额外等待。

第一版裁剪范围：

- `nextCheckAfterMs`: `1000-30000ms`
- `targetResponseDelayMs`: `800-20000ms`
- 旧上下文重试延迟：`500-1500ms`

## 旧上下文处理

调用模型前记录上下文版本：

```text
roundNo
messageCount
lastMessageId
voteCount
```

模型返回后检查一次，真正发送前再检查一次。

第一版规则简单处理：只要生成期间或等待发送期间出现新消息或轮次变化，就丢弃旧回答，并在 `500-1500ms` 后重新观察。这样可以避免 AI 无视刚刚的新发言。

## 后续可优化

- 区分“无信息新消息”和“实质新消息”，无信息内容可以保留旧回答。
- 被直接点名时优先选择被点名 AI，而不是随机候选 AI。
- 让策略层返回 `urgency`，用于更稳定地控制下次观察时间。
- 增加 AI 群体级别的发言比例约束，但这属于更强工程化控制，暂不放入第一版。
