## 1. 背景

“谁是AI”是一款真人玩家与隐藏 AI 玩家同场参与的社交推理游戏。普通对局的核心目标是：

- 真人阵营通过公开聊天和投票找出隐藏 AI。
- AI 阵营在不暴露身份的前提下存活到终局。

本文件定义的是**普通对局的产品规则**，不展开 AI 提示词、调度实现和调试房机制。

## 2. 范围

本文覆盖：

- 普通对局的开局条件
- 轮次与阶段
- 发言与投票规则
- 淘汰、胜负和奖励规则
- 与玩法直接相关的边界条件

本文不覆盖：

- 调试用自动对抗房
- AI 提示词、人格、缓存和模型调用细节
- 复盘、评估闭环和运维流程

## 3. 术语与定义

| 术语 | 定义 |
| --- | --- |
| 普通对局 | 面向真实玩家的标准房间，不包含调试用自动对抗逻辑 |
| 真人玩家 | `type === "human"` 且由真实用户通过 Socket 交互的玩家 |
| AI 玩家 | `type === "ai"` 的隐藏身份玩家，由模型自动驱动 |
| 讨论阶段 | `phase === "discussion"`，玩家公开发言的阶段 |
| 投票阶段 | `phase === "voting"`，玩家提交淘汰目标的阶段 |
| 结算阶段 | `phase === "resolving"`，服务端统计票数并决定是否淘汰玩家的阶段 |
| 存活玩家 | `status === "alive"` 的玩家 |
| 出局玩家 | `status === "eliminated"` 的玩家 |

## 4. 规则与需求

### 4.1 房间与开局

| 规则组 | 内容 |
| --- | --- |
| `START-ACCESS` | 普通对局等待房最多允许 `5` 名真人玩家加入；开局至少需要 `1` 名真人玩家；普通对局只能由房主发起开局。 |
| `START-SETUP` | 开局时，系统会自动补齐 AI 玩家，使普通对局总 AI 数量达到 `2` 名；开局后会对房间内玩家重新洗牌并重新分配座位号。 |

### 4.2 轮次与阶段

| 规则组 | 内容 |
| --- | --- |
| `ROUND-LIMITS` | 一局游戏最多进行 `4` 轮。 |
| `PHASE-FLOW` | 游戏阶段按 `waiting -> discussion -> voting -> resolving -> discussion ... -> game_over` 推进；第 `1` 轮开始时房间状态切换为 `playing`，阶段切换为 `discussion`；若本轮未结束且未产生胜者，系统会在结算后进入下一轮讨论阶段。 |
| `PHASE-DURATIONS` | 单轮讨论阶段默认时长为 `300_000ms`（5 分钟）；普通房仅允许在开局前修改该时长，且时长下限为 `60_000ms`；投票阶段时长固定为 `60_000ms`。 |

### 4.3 发言规则

| 规则组 | 内容 |
| --- | --- |
| `SPEAK-ACCESS` | 只有存活玩家才能发言；公开聊天只允许在讨论阶段发生；同一玩家两次发言之间必须满足 `15_000ms` 冷却时间。 |
| `SPEAK-CONTENT` | 单条消息长度上限为 `240` 字符，超出部分会被截断；普通对局中的 AI 玩家适用同一消息长度限制。 |

### 4.4 投票规则

| 规则组 | 内容 |
| --- | --- |
| `VOTE-ACCESS` | 只有存活玩家才能投票；投票只允许在投票阶段发生；每名存活玩家在每轮最多投 `1` 票。 |
| `VOTE-TARGETING` | 投票目标必须是除自己之外的其他存活玩家。 |
| `VOTE-BLIND-SIMULTANEOUS` | 当前轮投票采用同时盲投规则；投票阶段内不公开实时票型。 |

### 4.5 淘汰与信息公开

| 规则组 | 内容 |
| --- | --- |
| `ELIM-RESOLUTION` | 每轮投票结束后，得票数最高的存活玩家出局；若最高票平票，则本轮无人出局；若本轮无人投票，则本轮无人出局。 |
| `ELIM-EFFECTS` | 出局玩家不再参与后续发言和投票；普通对局进行中不公开 AI 身份，身份在游戏结束后的快照中才会揭示。 |

### 4.6 胜负规则

| 规则组 | 内容 |
| --- | --- |
| `WIN-CONDITIONS` | 若所有 AI 玩家都已出局，则真人阵营获胜；若所有真人玩家都已出局，则 AI 阵营获胜；若达到最大轮数时仍有 AI 玩家存活，则 AI 阵营获胜。 |

### 4.7 奖励规则

| 规则组 | 内容 |
| --- | --- |
| `REWARD-ELIGIBILITY` | 奖励池固定为 `2000` 积分；只有真人阵营获胜时才触发积分发放；只有存活且绑定账号的真人玩家有资格获得积分。 |
| `REWARD-DISTRIBUTION` | 奖励按合格真人玩家均分；若存在余数，按座位号升序依次补 `1` 分；AI 获胜或无合格真人玩家时不发放积分。 |

## 5. 边界条件与异常情况

| 编号 | 场景 | 处理规则 |
| --- | --- | --- |
| `EDGE-WAITING-LIFECYCLE` | 等待房真人掉线、最后一名真人离开 | 真人掉线后若 `30_000ms` 宽限期内未重连，则从等待房移除；等待房最后一名真人离开时删除房间。 |
| `EDGE-PLAY-DISCONNECT-KEEP-PLAYER` | 对局中真人掉线 | 不自动出局，不自动托管；玩家保留在房间中，后续可重连 |
| `EDGE-SPEAK-REJECTION` | 空消息、错误阶段、冷却未结束 | 拒绝发送。 |
| `EDGE-VOTE-REJECTION` | 非投票阶段投票、自投、重复投票、投已出局玩家 | 拒绝提交。 |

## 6. 非目标

以下内容不属于本文件定义范围：

- AI 发言时机、人格、提示词和模型协议
- 自动对抗调试房的阵营构成与模拟真人规则
- 积分流水账、事务化结算和排行榜
- 赛后复盘、评估闭环和调优策略

## 7. 验收标准

| 验收项 | 对应规则 | 验收标准 |
| --- | --- | --- |
| 开局条件 | `START-ACCESS` `START-SETUP` | 普通房至少有 1 名真人且由房主发起时才能开局；开局后自动补齐到 2 名 AI，并重新分配座位号。 |
| 阶段推进 | `ROUND-LIMITS` `PHASE-FLOW` `PHASE-DURATIONS` | 游戏能按 `discussion -> voting -> resolving` 正常循环，并在终局进入 `game_over`。 |
| 发言限制 | `SPEAK-ACCESS` `SPEAK-CONTENT` | 非讨论阶段不能发言；冷却未结束不能发言；超长消息被截断到 `240` 字符。 |
| 投票限制 | `VOTE-ACCESS` `VOTE-TARGETING` `VOTE-BLIND-SIMULTANEOUS` | 每轮每人只能投 `1` 次，不能投自己，只能投其他存活玩家，投票阶段不公开实时票型。 |
| 淘汰规则 | `ELIM-RESOLUTION` `ELIM-EFFECTS` | 最高票玩家被淘汰；平票或无人投票时无人出局；出局玩家不能继续行动。 |
| 胜负规则 | `WIN-CONDITIONS` | 全部 AI 出局则真人胜；全部真人出局则 AI 胜；第 `4` 轮结束仍有 AI 存活则 AI 胜。 |
| 奖励规则 | `REWARD-ELIGIBILITY` `REWARD-DISTRIBUTION` | 仅真人胜时向存活且绑定账号的真人发放 `2000` 积分，并按规则均分。 |
