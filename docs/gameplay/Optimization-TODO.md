## 1. 普通产品对局 · 发言调度

> 来源:2026-07-02 对 [AI-Speak-Scheduling.md](./AI-Speak-Scheduling.md) 的机制评审。红线:任何改动都不要回退到「策略层→表达层」双层链路、固定概率或 1s 模型轮询(§3/§8 已论证)。

### 1.1 被点名/被质疑时的即时反应 · **P0** · 待办

**问题**:被 `"2号你说说"` 点名时,AI 有两层延迟叠加——(a) `sendChat`(`game.service.ts:685`)落库后不唤醒调度器,要等最多 10s 的下一次心跳;(b) `selectByRoundFreshness`(`game.service.ts:2116`)完全不感知谁被点名,甚至可能选中另一个 AI 去接话。被围攻时慢半拍 + 答非所问,是最刺眼的 AI tell。

**方向**:
- 廉价版:`selectSpeechPlayer`(`game.service.ts:1964`)加一档最高优先级——扫描最近未被回应的消息,若点名了某存活 AI 则优先选它,可对其放宽/豁免 15s 冷却。改动局限在选择函数,风险低。
- 完整版:`sendChat` 落库后若消息点名 AI 且当前无生成在飞,触发一次带 2–4s 抖动的提前观察,不再干等心跳;仍需走串行锁与冷却防刷屏。

**关联**:AI-Speak-Scheduling.md §8「被直接点名时优先选择被点名 AI」。

### 1.2 固定心跳 → 事件驱动 + 抖动 + 随活跃度自适应 · **P1** · 待办

**问题**:当前节奏近似节拍器(首观察 10s、下次观察恒 10s、冷却恒 15s、退避恒 8s),唯一随机来自模型耗时与文本长度,且**不随人类聊天快慢变化**。粗算(5 分钟局、2 AI、低 skip)约每 13–16s 合计一条、一轮约 20 条 AI 发言。

**方向**(可叠加,按成本递增):
- 给心跳加抖动(如 7–14s 取代恒定 10s),打破可预测节奏。近零成本。
- 按人类近 N 秒消息速率缩放 `nextCheckAfterMs`:人多话密时少插嘴,冷场时略主动补话。契合 §2「冷场晚点补话/信息少沉默」与 §8「群体级发言比例约束」。
- 事件驱动 + 心跳保底:心跳作下限,人类消息可提前唤醒调度(与 1.1 完整版共用唤醒通道)。

**关联**:AI-Speak-Scheduling.md §5.6、§8。

### 1.3 双 AI 协同盲区(抱团/复读 tell) · **P2** · 待办

**问题**:选择逻辑不看另一个 AI 刚说了什么、针对谁。两个 AI 可能连续帮腔同一人或先后表达高度一致的怀疑,产生「这俩总站一边」的可疑对齐。

**方向**:加轻规则——若 AI-A 刚附和/攻击某人,本次观察降权 AI-B 立刻做同样的事。单层机制下无 `speechAct` 结构,只能用启发式判断上一条 AI 发言指向,复杂度中等。

### 1.4 参与度地板 / 沉默即 tell · **P2** · 待办(需先观测)

**问题**:狼人杀类游戏里整轮不说话本身是破绽,但当前无最低参与保障;两个 AI 若都持续 skip,可能整轮近乎失声。

**方向**:软约束——某 AI 本轮未发言且时间过半时,优先选它 / 通过提示词降低其 skip 倾向。价值取决于实测 skip 率,建议先观测再决定阈值。

### 1.5 上下文失效判断的极端场景 · **P2** · 已搁置(有意取舍)

**问题**:讨论期实际只有 `roundNo`/阶段变化会丢弃发言,新聊天一律不丢弃;快节奏房里可能落库一条「基于十几秒前上下文」的发言。

**现状结论**:AI-Speak-Scheduling.md §6 已明确接受该取舍(小局节奏优先,交离线尺子衡量)。**暂不动**;仅当 1.1/1.2 上线后仍观察到明显答非所问,再考虑「实质新消息 vs 噪声」的区分(§8 亦列)。

## 2. 离线沙盒 · 发言调度

> 来源:2026-07-02 对 [AI-Speak-Scheduling.md](./AI-Speak-Scheduling.md) §7 的机制评审。评价视角是「测量仪器」——判据是可复现性 / 无混淆变量 / 对被测分布的保真度,不是人类相似度。骨架(串行 await、跨 pass 轮转、无节流、探测确定性改派、复用生产 `generateSpeech`)是对的,不需重构;下列为仪器校准级泄漏。A、B 建议在自对弈优化循环真正跑起来**之前**修——仪器有偏会让整条优化朝偏差过拟合。

### 2.1 单局发言密度被模型延迟混淆 · **P1** · 待办

**问题**:每轮 pass 数 = `讨论时长 ÷ 每个 pass 墙钟耗时`,而每个 pass 耗时 = 所有 model-driven 玩家模型延迟之和(顺序 await、无节流、**无 pass 上限**——`completeSandboxSpeechPass`(`game.service.ts:1689`)只判 `phaseEndsAt` 到期)。后果:transcript 密度不是受控参数,而是延迟副产物。快的 AI → 单位时间更多 pass → 每轮更多发言/探测;慢的 AI → 更少。**比较延迟不同的两个 AI 版本时,它们面对的是结构不同的对局**(发言轮数、被追问次数都不同),污染 win-rate / human-likeness 对比。

**方向**:把「每轮交换次数」变成显式受控量——加每轮 pass 上限(如 `sandboxMaxPassesPerRound`),或把讨论结束条件从「墙钟到期」改为「达到 N 个 pass」。使密度与延迟解耦、跨版本可比。

### 2.2 首个 startOffset 改为基于场景 seed 的确定性随机 · **P1** · 待办

**问题**:`prepareSandboxSpeechState`(`game.service.ts:1660`)首个 `startOffset` 用 `Math.random()`,破坏可复现性。发言顺序实质影响结果(谁先定调、谁在反应谁),而可复现是盲测地基;当前探测是 seeded-确定性、发言顺序却是真随机,**确定性哲学不一致**。

**方向**:改为基于 `sandboxSeed / run / round` 的 seeded 随机——既复现又无固定座位偏差。与 2.1 同为「同种子可复现」这条自对弈原则的前提,建议一起做。

**备注**:2026-07-02 从 P2 提升到 P1(仪器视角下可复现是地基,非锦上添花)。

**关联**:AI-Speak-Scheduling.md §8。

### 2.3 密集轮转 vs 生产稀疏抢话 → 分布偏差 · **P2** · 已知取舍待记录

**问题**:沙盒里每人每 pass 都说(除非 skip),节奏密集、回合制;生产里 AI 每 ~13–16s 才说一次、稀疏、实时交错。这是 §7 刻意的「控制换保真」取舍,不是 bug;但代价是**在此结构上优化出的提示词可能对一个生产中不存在的会话结构过拟合**——也是「把 win-rate/human-likeness 当代理指标」的根因。

**方向**:不「修」,而是(1) 文档显式声明该偏差与其对指标可迁移性的影响;(2) 可选——给 detective/filler 引入概率性 skip,让 transcript 更接近生产稀疏度。

### 2.4 skip 无退避导致重复追问浪费 · **P2** · 待办

**问题**:沙盒里 skip 只标记 considered(`markSandboxSpeechConsidered`),下个 pass 照样再选中。若 ai_under_test 在受控场景里反复 skip,就每 pass 被重问一次(烧模型调用);detective/filler 持续 skip 会让记录变薄。

**方向**:加轻量守卫——同一玩家本轮连续 skip N 次后本轮不再问,降成本、避免空转。
