## 一、问题定义与第一性原则(整套设计的根)

**产品**:多人实时聊天游戏《谁是AI》。房间 3–5 人含至少 1 个 AI,真人要找出 AI,AI 要藏住;讨论自由聊天,投票同时盲投,每轮票最多者淘汰、平票不淘汰,4 轮后 AI 存活则 AI 胜。

整套设计建立在一个**核心判断**上,它决定了后面所有取舍:

> **AI 输,不是因为"话不像人",而是因为"它在被动回应,而真人在主动博弈"。**

大模型的默认状态是"乐于助人的客服":完整、中立、礼貌、有问必答。而这局里所有真人都是自私、片面、爱算计、有点懒、只想保自己票别人的。一个语言再自然、但只会好好聊天有求必应的玩家,几轮必死。由此推出三条第一性原则,贯穿全系统:

1. **拟人化 = 行为与博弈参与,而非语言修饰。** 暴露 AI 的主要是行为模式(是否主动起话题、是否怀疑别人、是否参与投票博弈),不是单句文本质量。所以优化重心在"怎么玩这局",不是"怎么把字打得碎"。
2. **目标是"每轮不当最可疑的人",而非"赢得辩论/找出谁"。** 胜利条件很宽松(撑过 4 轮),所以 AI 每轮唯一任务是不成为票最高者——不显眼、把火引向真人。这个目标框定法,比任何拟人技巧都重要。
3. **三个致命指纹必须压住**:① 完整/中立/有条理(列点、"首先其次"、面面俱到);② 礼貌/热心/有求必应;③ 随叫随到展示能力(答对测试题=自杀)。对应的正面状态是:有立场、有偏见、有情绪、有盲区、有点懒。

> 这三条不是"提示词技巧",而是**系统的设计公理**。裁判量表的维度、场景库的探测类型、优化器的目标,全都是这三条的下游产物。改动若动摇了这三条,要在第八章取舍记录里显式重评。

## 二、整体架构总览

系统分两半,靠"真人对局"这条真相信号闭合:

```
┌────────────────────── 产品运行时(对真人) ──────────────────────┐
│  真人玩家  ⇄  AI玩家(由【提示词版本 + 人设卡 + 选定大模型】驱动)   │
│                        │ 真人对局全量记录(转录+票+结果+版本)        │
└────────────────────────┼─────────────────────────────────────────┘
        真相信号/失败回灌 ▼                              ▲ 部署当前最优版本
┌────────────────────────┼──────────────────────────────┼──────────┐
│            离线优化沙盒(自动演化提示词)                │          │
│  场景库 ─► 对局引擎 ─► 裁判评分 ─► 优化器 ─► 子版本 ─► 配对评测/验收 │
│    ▲                                                      │        │
│    └──────── 真人失败回灌(长场景库) + 真人校准(验对齐) ◄┘        │
└────────────────────────────────────────────────────────────────────┘
```

- **左/上半:产品运行时**。真正对战真人的 AI 玩家,由〔提示词〕里的提示词版本 + 人设卡 + 选定大模型共同驱动。**被优化的对象就是这个"提示词版本"。**
- **右/下半:离线优化沙盒**。让 LLM 自对弈、裁判打分、优化器改提示词、再配对评测验收,自动迭代出更好的版本。
- **闭合两半的是真人对局**:它既提供真相信号校准沙盒(防自欺),又把真实失败回灌进场景库(让沙盒持续逼近真人)。

为什么要这个两半结构:**拟人化是个评判主观、没有现成信号的难题,凭感觉改提示词不可能稳定迭代。** 而这个游戏运气好——它自带客观真值(被票出去没有)。把整套优化建立在这个信号上,才有"稳定演进"可言。这正是引入沙盒的根本理由(详见第四章)。

## 三、第一部分:AI 玩家的拟人化设计(被优化的本体)

> 详见〔提示词〕全文。这里只讲为什么这样设计。

**为什么用"行为/人设"而非"语言技巧"来做拟人化**:见第一章原则 1。所以〔提示词〕的系统提示词主体不是教它"怎么打字",而是教它"以一个想活下去的骗子身份玩这局"——把目标、心态、博弈打法、抗测试讲清楚,效果远好于"请像人一样聊天"。

四个关键部件及其设计意图:

- **系统提示词**(〔提示词〕§一):核心是**目标框定 + 负面清单 + 博弈参与 + 抗测试**。负面清单(禁止完整/礼貌/展示能力)直接对应三大指纹;"主动指认别人是 AI"被列为最强单招,因为它**一石三鸟**——直接做了真人在做的事(显得不像 AI)、把火引开、且用"凭感觉"的理由(避免侦探式严密推理这一新指纹)。
- **人设卡 ×4**(〔提示词〕§三:阿条/酸梅/布丁/探长):**为什么要人设,且要"自带掩护"的人设**——给行为一个稳定一致的来源,并让"不答测试题=懒""片面=性格""怼人指认=毒舌"全部合情合理,不需要刻意演。这是把抽象原则落成可执行性格的手段。
- **投票提示词**(〔提示词〕§五):**为什么单独成一套且强调"投票=自保不是论证"**——因为 AI 胜利条件是撑到第 4 轮,平票无人淘汰对它无害,唯一危险是票集中到自己身上;所以投票目标是"帮票集中到某个真人",而非"把谁找出来"。这与讨论阶段是不同的优化目标,故分开调用。
- **运行时上下文 + 调参**(〔提示词〕§二、§四):每局换人设、temperature 调高、允许沉默——都是为了减少"标准答案感"、增加个性与随机性。

> **为什么把它单列为"被优化的本体"**:整个沙盒存在的意义,就是把这份提示词版本越改越好。它是系统的产物,也是系统的输入。

## 四、第二部分:为什么需要离线沙盒

三个现实问题逼出了沙盒(这三点是用户在演进中逐步提出的,沙盒正是对它们的系统性回答):

1. **没有度量就没有迭代**:拟人化评判主观,凭感觉改提示词是抓瞎。→ 需要客观信号 + 自动评分。
2. **三种波动各需对策**:不同大模型默认人格不同(同一提示词换模型必须重测,提示词+模型要绑定评测,甚至要维护按模型的变体);同模型同提示词不同对局输出也不同(绝不能一局定胜负);对手强弱/人数等混杂因素会干扰。→ 需要大规模对局 + 配对比较 + 控制变量。
3. **想比"等真人来玩"快上百倍** → 需要让 LLM 自己扮演玩家来离线跑。

沙盒采用的范式是 **自对弈(self-play)+ LLM 当优化器的进化式提示词搜索**(与 OPRO / EvoPrompt / DSPy、以及 AlphaGo 式自我博弈同源)。它的骨架与六模块见〔沙盒〕"模块总览与数据流"。

## 五、沙盒各模块:设计原因与串联

> 每个模块"做什么"从简,"为什么这样设计"从详,并指向子文档。

### 5.1 对局引擎(详见〔引擎〕全文)
**做什么**:按场景把一局从头跑到尾,产出 `MatchRecord`。
**为什么这样设计**:
- **与裁判解耦、自己不打分**(〔引擎〕§1、§14):保证引擎可独立测试、可被编排器大规模并行调用;评分逻辑变化不影响对局逻辑。
- **「固定剧本 + 活反应」而非全脚本**(〔引擎〕§5):全脚本会让对手发言与上下文脱节(数据失真),且无法合理脚本化投票。解法是只固定"与 AI 措辞无关的外部冲击"(人设、发言时序、探测时点、对手本轮意图、种子),把"对 AI 的反应"放活——既不失真,又保住配对评测要的环境一致。
- **探测 = 触发器 + 意图**(〔引擎〕§9):由活对手当场用自己口吻生成,解决脱节;`pass_if` 可自动判的引擎直接判、细微的交裁判。
- **投票 `vote_policy` 三档**(〔引擎〕§10):`live`(默认,最贴近真人)/`rule`(快测、可复现)/`scripted`(仅压力测试)。投票绝不能写死固定目标,否则"表现好不好"与"是否被票出"脱钩,真值信号就废了。
- **两种场景形态**(〔引擎〕§6):`full_match` 测综合存活;`spotlight` 预置历史、中盘起跑,高效定向测特定局势/轮次。
- **统计降方差而非冻结环境**(〔引擎〕§11):同种子 + 配对做差 + 多 run 取均 + 加样本——用统计对抗方差,比把对手写死温和且不失真。

### 5.2 裁判评分量表(详见〔沙盒〕模块二)
**做什么**:从对局记录产出评分。
**为什么这样设计**:
- **两类信号分开**:客观结果指标(逐轮存活/被票中率,纯计算、是真值)+ 裁判 LLM 的"人味分"。前者告诉你"好不好",后者(分维度)告诉你"为什么、该改哪"——优化才有方向。
- **盲测可疑度两遍走**:第一遍**匿名**让裁判打"是 AI 的概率",偏差最小、且是**后续和真人校准的那个锚**;第二遍才告知身份做诊断与失败定位。
- **多裁判**:降低单一裁判偏差(也防只对一个裁判过拟合)。
- **逐轮密信号而非只看胜负**:4 轮通关太稀疏,逐轮指标给优化器更丰富的"梯度"。
- **量表维度直接派生自三大指纹**(第一章),"出戏/能力暴露"是否决项。

### 5.3 优化器 I/O(详见〔沙盒〕模块三)
**做什么**:吃当前版本 + 诊断,吐多个子版本。
**为什么这样设计**:喂**精选失败案例 + 最差维度 + 已试失败记忆**而非原始转录——收敛快、不爆上下文、防反复提同一种没用的改法;要求每个子版本**带可验证的假设和目标维度**,评测后能回头确认"是否真修好了那个维度"。

### 5.4 场景库(详见〔场景库〕全文)
**做什么**:提供可复现的对局设置集合。
**为什么这样设计**:
- **放弃"覆盖所有场景",改为"覆盖所有重要失败模式"**(〔场景库〕开篇):状态空间无穷,穷举不可行也无意义;场景库是**测试用例集**,不是世界地图。
- **分层抽样 + 两两覆盖**(〔场景库〕第一部分):沿正交维度保证边际覆盖,并单独维护"探测类型 × 社交局势"矩阵——失败集中在组合处。
- **主要靠真人失败回灌生长 + 用真人频率重配比**(〔场景库〕第二部分):覆盖度由真人真实分布背书,而非想象力;真人骚操作的分布,就是场景库该长成的样子。
- **optimize / holdout 分层切分 + 探测轮换**:防止提示词"背答案"而非学会泛化回避。

### 5.5 版本与指标记录(详见〔沙盒〕模块五)
**做什么**:PromptVersion / MatchRecord / ScoreRecord / GenerationEval 四张表,靠 ID 串联。
**为什么这样设计**:**全量留痕是最值钱的资产**——它既是失败挖掘的来源,又让每次改动可追溯、可回退;提示词进 git、每版挂实测指标,使迭代有据可查、能出血脉/排行榜。

## 六、迭代闭环与三道防自欺闸门(系统能否稳定演进的关键)

各模块串成的演进循环见〔沙盒〕"编排循环与验收闸门":当前 champion 在优化集跑分 → 组装诊断喂优化器 → 产出多个子版本 → **同场景同种子配对评测** → 过接受闸门 → 留出集复核 → 晋升/进种群 → 每 M 代真人校准。

整套系统能否"稳定演进",**不取决于能不能跑通,而取决于这三道闸有没有装上**:

1. **配对比较 + 显著性(防噪声)**:对局是高方差离散信号,小样本/单线贪心会被运气骗、卡局部最优。→ 父子同场景同种子、做差、显著性门槛、维持种群。
2. **留出集 + 探测轮换(防过拟合)**:若优化集既优化又验收,几代后就是"背了那几局答案"。→ 留一批永不进优化器的验收场景,定期轮换探测题。
3. **真人对局校准(防 Goodhart)**:沙盒里"人"和裁判都是 LLM,整个环可能优化出"骗过 LLM 却骗不过真人"的版本。→ 真人局是**唯一真值与最终裁决**,沙盒只是快但有偏的预筛器,定期用真人结果校准、监控代理指标是否漂。

> 这三条在〔沙盒〕"避坑清单"里有逐条落地版。**它们是本系统区别于"随便搭个自对弈玩具"的核心。**

## 七、对齐真人:代理与真相的关系

"和真人对齐"不是一次写对,是长期回灌 + 持续验证:

- **机制**:真人失败 → 抽象成 probe 模板 + 场景族 → 回灌场景库(〔场景库〕第二部分);真实失败的分布,就是场景库的目标分布。
- **可量化的对齐体检**:定期算"沙盒分(盲测可疑度) vs 真人实测(存活率)"的**相关性**。高 → 沙盒可信;下滑 → 代理漂了(AI 学会骗沙盒、或真人玩出新套路),要修裁判/侦探池/补场景再校准。
- **承认覆盖不到的部分**:总有第一次见的骚操作。对策不是事前穷举,而是让提示词养成**通用回避反射**——任何"展示能力/暴露身份/反常配合"的请求,默认用人设化的"嫌烦/装没懂/怼回去"来挡。泛化的回避习惯比穷举题库更扛未知;探测轮换正是用来检验它学的是习惯而非答案。

## 八、关键设计取舍记录(供演进时回查,避免误推翻)

| 决策 | 选了什么 | 放弃了什么 | 为什么 |
|---|---|---|---|
| 拟人化的本质 | 行为与博弈参与 | 只做语言修饰 | 暴露 AI 的主要是行为而非单句文本 |
| AI 的目标 | 每轮不当最可疑者 | 赢辩论/找出谁 | 胜利条件宽松,自保即可,平票无害 |
| 最强单招 | 主动凭感觉指认别人 | 被动应答 | 一石三鸟:像真人、引开火、用感性理由避新指纹 |
| 抗测试 | 像嫌烦的人一样回避 | 认真应对/展示能力 | 答对刁钻题=当场暴露 |
| 改提示词的依据 | 客观信号 + 自动评分 | 凭感觉改 | 主观评判无法稳定迭代 |
| 提示词与模型关系 | 绑定评测,或按模型出变体 | 假设可跨模型平移 | 不同模型默认人格不同 |
| 一次改动的判定 | 大量对局 + 配对 + 显著性 | 单局/小样本定胜负 | 信号高方差,会被运气骗 |
| 沙盒对手 | 固定剧本 + 活反应 | 全脚本 / 全自由 | 全脚本失真且无法脚本化投票;全自由方差大、不可配对 |
| 投票 | `live` 默认,三档可选 | 写死固定目标 | 写死会让"表现"与"被票出"脱钩,真值信号失效 |
| 降方差 | 统计手段 | 冻结环境 | 冻结省样本但买来失真 |
| 场景库目标 | 覆盖失败模式 + 真人回灌 | 穷举所有场景 | 状态空间无穷;真实分布才是覆盖依据 |
| 防过拟合 | 留出集 + 探测轮换 + 抽象化回灌 | 原样存真人转录 | 否则只是背答案,不泛化 |
| 应对未知探测 | 通用回避反射 | 逐题防御 | 泛化习惯比穷举题库扛未知 |
| 终极真相 | 真人对局 | 沙盒/裁判分数 | 沙盒是有偏代理,真人是唯一裁决(防 Goodhart) |

> **开放决策**(尚未定,演进时要拍):同局多 AI **是否互相知情**(影响投票是否会误伤队友,见〔提示词〕§五说明);是否为每个大模型维护独立提示词变体;人设是否随产品调性扩充/改语气档位。

## 九、演进路线与扩展指引

**搭建顺序**(详见〔沙盒〕"建议的搭建顺序",别一上来全做):
1. MVP:对局引擎 + 客观结果指标 + 单裁判盲测可疑度 + **手动**改提示词,先把"跑一局、打个分"跑通。
2. 加配对评测 + 显著性,接入优化器 LLM,实现自动一代。
3. 加留出集 + 无回退闸门,防过拟合。
4. 加真人对局校准 + 探测轮换。
5. 加种群 / 多裁判等抗局部最优、降偏差的增强。

**常见扩展点与注意**:
- **加新人设**:在〔提示词〕§三补卡,并在〔场景库〕维度 G 纳入配比;新人设需各自跑回归。
- **加新探测类型**:多来自〔场景库〕回灌流程的真实失败抽象;进 probe bank 要带轮换变体和 `pass_if`,并分 optimize/holdout 实例。
- **换/加大模型**:必须重测(提示词+模型绑定评测),可能要出按模型变体。
- **规模化时盯**:代理-真相相关性是否漂、`status=recurring` 的失败模式(库里有了真人局还在犯=在背答案没泛化)、裁判间一致度。

**演进的红线**:任何改动若触及第一章三大原则或第八章已记录的取舍,先在本文档显式重评再动手——这份文档存在的意义,就是让"为什么这么设计"不随人员更替而丢失。
