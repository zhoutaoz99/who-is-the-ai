# 《谁是AI》AI玩家系统 · 总体设计文档(架构与设计依据)

> **定位**:本文是整套系统的**顶层总览**——回答"这套系统是什么、由哪两大部分组成、以及每个关键决策**为什么**这么定"。
> 它**不**重复模块内部细节:优化沙盒的内部架构见《离线沙盒总纲》,各模块/机制见对应专文(§7 导航)。本文的独特价值在 **§3 核心设计洞察**与 **§5 关键设计决策与权衡**——把贯穿整个设计过程的"设计依据"收口成一份理由档案。

---

## 1. 系统目标与边界

**产品**:多人即时聊天游戏《谁是AI》。房间 3–5 人,混有至少一个 AI 假扮真人;真人靠多轮聊天 + 盲投找出 AI,AI 藏身份。撑过 4 轮还有 AI 存活=AI 赢。

**这套系统要交付两样东西**:
- **A · 一个真人难以分辨的在线 AI 玩家**——它在真实对局里参与聊天与投票。
- **B · 一套能持续把 A 变更强的离线优化机制**——自动迭代 AI 玩家的提示词。

**唯一的成败标准**:真人对局里 AI 的**存活率 / 被识破率**。一切设计都服务于压低"真人觉得它像 AI"。

---

## 2. 系统全景:在线 ⇄ 离线 双回路

```
        ┌───────────────────── 在线层(生产) ─────────────────────┐
        │  AI 玩家 = 当前 champion 提示词 + 人设 + 投票逻辑          │
        │  在真实房间里和真人聊天、盲投,目标:别被识破、撑过4轮     │
        └───────────────┬───────────────────────▲────────────────┘
                上线 champion │                       │ 真人对局结果 + 被抓的失败
                        │                       │(真值 + 回灌素材)
        ┌───────────────▼───────────────────────┴────────────────┐
        │            离线层(优化沙盒)                              │
        │  自对弈引擎 → 裁判评分 → 评分聚合 → 优化器 → 编排器        │
        │  用真人失败回灌场景库、用真人结果校准代理,迭代出更强 champion │
        └─────────────────────────────────────────────────────────┘
```

**关键:这是一个闭环。** 离线沙盒不是一次性调出一版提示词就完事,而是和在线持续互喂——沙盒产出 champion 上线;在线真人对局产出"真值"(校准沙盒的代理指标)和"真实失败"(回灌沙盒的场景库)。**真人对局永远是最终裁决**,沙盒只是放大真人信号、快速预筛的离线加速器。

## 3. 核心设计洞察(设计依据 · 总纲)

整套系统建立在九条贯穿始终的洞察上。每条都是"一个反直觉的判断 + 它如何决定了下游设计"。

1. **AI 输在博弈缺位,不在文笔。** 大模型默认是"有问必答、礼貌周到的好聊天对象";但这局里所有真人都自私、片面、爱算计、想保自己票别人。语言再自然,只要"只回应、不博弈",几轮内必死。→ **AI 玩家的目标从"说得像人"改成"想活下去的骗子:每轮别成为最可疑的人"。** 这是 AI 玩家提示词的立身之本。

2. **对 AI 来说,"展示能力 = 自杀"。** 答对一道刁钻测试题,比答错暴露得更彻底。→ 抗测试一律**人设化回避**(嫌烦/装没懂/怼回去),把能力藏死。

3. **拟人 = 融入这群人这场对话,不是表演"人味"。** 最好的伪装是"略低于平均存在感的普通人",镜像房间的语气与能量。→ 反对任何单维度过度优化(疯狂加错别字、拼命玩梗反而露馅)。

4. **没有度量就没有迭代;而这个游戏自带真值。** 真人投票就是一台免费的、持续运转的标注机。→ 整套优化建立在"真人对局存活/被识破"这个客观真值上,而非主观感觉。

5. **降方差靠统计,不靠冻结环境。** 把对手写死能降方差,但会失真。→ 对手用"固定剧本 + 活反应"(只固定与 AI 措辞无关的外部冲击),配对评测靠**同种子做差**消掉共模噪声。

6. **决策信号 vs 诊断信号必须分离。** 裁判八维量表信息丰富但易被刷。→ **决策只用最接近真值的信号**(客观结果 + 盲测可疑度 + 否决);量表与失败案例只用于诊断"该改哪",绝不当优化目标。

7. **三道防自欺闸门,构成只进不退的棘轮。** 配对+显著性(防噪声)、留出集+探测轮换(防过拟合)、真人校准(防 Goodhart)。→ 噪声驱动的假"进步"过不了重复确认,**只有真实增益代代复利**。

8. **覆盖 = 逼近真人失败分布,不是穷举。** 对局状态空间无穷,穷举不可达且仍会漏真人的骚操作。→ 场景库用分层抽样 + 真人失败回灌生长;未知攻击靠提示词的**通用回避反射**兜底。

9. **依赖只走契约。** 模块强相关但不同质、不同步变。→ 用稳定 schema 解耦,把跨模块的共享约定沉淀进契约,而非把模块合并。

## 4. 架构总览

三层 + 连接它们的契约:

| 层 | 组成 | 职责 | 详见 |
|---|---|---|---|
| **在线层** | AI 玩家提示词 + 人设卡 + 投票逻辑 | 生产环境真正在玩 | 《AI玩家·prompt模板与人设卡》 |
| **离线优化层** | 对局引擎 · 裁判 · 评分聚合 · 优化器 · 编排器 · 场景库 | 自对弈迭代出更强 champion | 《离线沙盒总纲》及各模块专文 |
| **契约层** | 输入契约(场景/探测)· 输出契约(对局记录) | 让上述模块只对接口编程、可独立替换 | 两份 Schema 契约 |
| **参与方** | 侦探 · 填充 · 裁判 提示词(沙盒里扮演对手与评委) | 构成可控、像真人的对局环境 | 三份提示词专文 |

## 5. 关键设计决策与权衡(设计依据 · 明细)

把整个设计过程里反复争论、最终拍板的主要决策,以"选了什么 / 备选 / 为什么 / 代价"记录下来,作为可追溯的理由档案。

| 决策 | 备选方案 | 为什么这么选 | 代价 / 缓解 |
|---|---|---|---|
| 对局用 **scripted_intent(固定剧本+活反应)** | 全脚本 / 全自由 | 固定外部冲击保住配对一致,放活反应保住真实;两者兼得 | 探测投放瞬间略生硬,但真人测 AI 也这么突兀 |
| 投票 **vote_policy 三档,默认 live** | 场景写死投票 | 写死会让"AI 表现↔被票出"脱钩,毁掉真值信号 | live 有成本 → filler 用 rule 缓解 |
| **移除 intent_schedule** | 保留"本轮注入怀疑谁" | "怀疑谁"是对 AI 表现的**内生反应**,注入即抹掉因果、失真 | 起跑后局势控制力变弱 → 由 spotlight 预置历史补 |
| 场景库 **分层抽样 + 真人回灌** | 穷举所有场景 | 状态空间无穷,穷举不可达且仍漏真人骚操作 | 需持续回灌运营 → 用台账/看板管理 |
| 优化器 **单候选 + 外层派靶** | 一次产 K 个候选 | 一次产 K 个会长输出退化、多样性造假、纪律涣散 | K 次调用 → 可并行,总成本相近 |
| 算子选择 **类型约束 + 战绩排序 Top-2** | 穷举每个算子 / 随机 | 多数算子与某破绽不搭;战绩让"哪种改法修哪类破绽"随迭代变准 | 需积累战绩 → 冷启动用先验优先级 |
| 主信号 **盲测可疑度(相对 margin)+ 像玩家不像法医** | 用输赢 / 绝对分 / 法医式分析 | 连续量低方差、相对量贴"相对最可疑才出局"、像玩家才代理真人直觉 | 代理会漂 → 真人校准纠偏 |
| filler **共识跟随投票(rule)** | 随机投 / 认真投 | 跟票放大已有怀疑、不注入噪声,又模拟"不上心真人"并给 AI 掩护 | 需保证侦探占多数,别没人真抓 |
| 契约 **拆输入/输出两份** | 合并 / 塞进引擎文档 | 多方共享的接口不该住进任一方,防字段被单方需求带偏 | 两份文档 → 各自单一事实来源,反更清晰 |
| 关键 schema **设单一事实来源** | 各文档各写一份 | 防 schema 在多处漂移、对不上 | 改字段须走契约升版 |

## 6. 贯穿全局的风险与防护(一张对照表)

| 风险 | 表现 | 防护 |
|---|---|---|
| **Goodhart**(优化代理而非真人) | 沙盒分涨、真人更易识破 | 决策用近真值信号 + 盲测像玩家 + 真人校准相关性体检 + 翻车回滚 |
| **过拟合**(背场景/背题) | optimize 集好、新场景崩 | 留出集 + 探测实例隔离 + 偏好通用回避反射 + 盯 recurring |
| **噪声假信号** | 手气好被当成进步 | 同种子配对做差 + N=场景数 + 显著性+MDE + 代际棘轮 |
| **单维度过度优化** | 疯狂加错别字等反而露馅 | 提示词目标定为"降被识破"非刷单项 + 镜像房间原则 |
| **对手失真** | 侦探太弱/太像机器,沙盒虚高 | 侦探守反指纹纪律 + 强弱档 + 侦探多样性 + 真人校准 |
| **优化器自伤** | 提示词膨胀/自相矛盾/打转 | 长度预算+consolidate + validate_prompt + tried_and_rejected |

## 8. 现状与下一步

- **设计层面已闭环**:在线 AI 玩家、离线优化沙盒、两条契约接缝、四类参与方提示词、核心聚合机制均成文且自洽。
- **下一步:从设计转可运行骨架。** 系统级搭建顺序建议:
  1. 先把**在线 AI 玩家**(提示词+人设+投票)做出可玩版本,人工对局验证基本拟人度;
  2. 搭**离线 MVP**(引擎 scripted_intent+live+full_match → 客观指标 → 单裁判盲测 → 手动改提示词),跑通"能跑一局、能打个分、能手动迭代一轮";
  3. 接入**评分聚合管线**(run→配对→N→CI→verdict)与**配对评测+显著性**,实现第一次自动迭代;
  4. 逐步加留出集、外层派靶+战绩、真人回灌与校准、种群,按《离线沙盒总纲》§8 推进;
  5. 全程让在线真人对局**持续回灌与校准**,形成 §2 的闭环。

---

> 一句话:这套系统的全部设计,可归结为一个目标和一条信念——目标是**让 AI 玩家融入真人、别被识破**;信念是**只让经得起真人检验的真实改进沉淀下来**。前者塑造了 AI 玩家提示词的每条纪律,后者塑造了离线沙盒的每道闸门。其余一切都是为这两点服务的工程。
