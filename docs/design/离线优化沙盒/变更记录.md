## #1 移除 `intent_schedule` 与 `base_intent`(2026-06-25)

### 背景
早期设计(见《对局引擎·方案设计》§5.1「固定剧本 + 活反应」)把对手行为拆成「固定外部冲击」与「放活反应」两类,并用 `intent_schedule`(逐轮给对手注入"本轮意图",如"这轮重点怀疑 B")与 `RosterSlot.base_intent`(静态立场补充)来实现"固定"那一半。MVP 据此实现了从契约、运行时到提示词的完整链路。

### 触发
实现收尾时复盘发现:`intent_schedule` 与沙盒最核心的原则存在真实矛盾——它不是"口味选择",而是把对手**往"写死"那个方向**推。

### 依据原则
> **不冻结环境,用统计降方差**:不靠把对手写死来消方差(那会失真),而靠**同种子 + 配对 + 多 run + 加样本**。(见《离线沙盒 模块设计清单》三大设计原则之二、避坑清单「投票别写死」「探测别用死台词」。)

### 决策
**移除** `intent_schedule` 与 `base_intent`(契约字段、运行时 plumbing、提示词注入位、示例场景、前端展示全部清掉)。侦探/填充的立场与打法**完全由其人设卡承载**(静态、固定),逐轮反应交给真实上下文自然涌现。

### 理由:probe / spotlight / intent 的「可固定性」根本不同

| 机制 | 是否外生于 AI 措辞 | 固定它是否扭曲活反应 | 结论 |
|---|---|---|---|
| **probe**(`probe_schedule`) | 是——真人也会不管你说什么、突然甩怪问题 | 否——固定的是"砸什么探测、何时砸",投手当场看真实上下文生成台词 | **保留** |
| **spotlight**(`seed_history`) | 是——把"为何集火/结盟"作为**既成事实**发生在镜头外 | 否——只设定**初始状态**(所有评测都这么干),起跑之后自然演化 | **保留** |
| **intent_schedule** | 否——在镜头内强行规定活侦探"这轮怀疑谁" | **是**——篡改进行中的判断,把活对手写死 | **移除** |
| **base_intent** | 否——可被用来给特定侦探写死"专盯 B"(预 scripting 的滑坡);且人设卡已承载静态立场 | 是 | **移除** |

关键差别:**probe / spotlight 固定的是「外部冲击 / 初始状态」**(合法,公平且不失真),**intent_schedule 固定的是「对手的活反应」**(失真,违背原则)。

### 替代方案(社交局势怎么实现)
"被集火""结盟"等社交局势不再靠注入意图:
- **`full_match`**:让活侦探基于真实上下文自然涌现——AI 表现好则怀疑自然消退,差则加重,这条因果链被保住,正是要测的东西。
- **`spotlight`**:想测"被集火的 R3",就把预置历史写成"前两轮 B 已被几个人盯上",从有来由的处境起跑。集火是**作为历史背景**给定(发生在镜头外),不是镜头内扭转判断。

### 影响范围
- **输入契约**:Scenario 删 `intent_schedule`;RosterSlot 删 `base_intent`;`IntentDirective` 整型移除;校验(§7)删相关不变量。
- **输出契约**:MatchRecord 的 `Turn` 删 `injected_intent`。
- **运行时**:`GameContext` 删 `myBaseIntent`/`myInjectedIntent`;`Room` 删 `sandboxIntentSchedule`;`Player`/`SandboxPlayerSpec` 删 `baseIntent`;`game.service` 删 `resolveInjectedIntent` 及 `createSandboxRoom` 的 intentSchedule 参;`castAiVote` 的 live/rule/scripted 分支不受影响。
- **提示词**:侦探讨论/投票、填充讨论模板删 `{{base_intent}}` / `{{本轮intent}}` 注入位;**保留** `{{本回合任务}}`(probe)。
- **立场来源**:侦探立场完全由人设卡(含 `howToPlay`)承载,逐轮怀疑谁自然涌现。
- **示例场景 / 前端**:5 个示例与配置页清理。

## #2 删除"AI 自动对抗",沙盒接管其底层机制(2026-06-25)

### 背景
产品层曾有两套自动对局入口:旧的「AI 自动对抗」(产品调试 `createDebugAutoAiRoom` → 2 AI + 3 模拟真人 → `/room`)与新的「离线沙盒对局」(场景驱动 → `/sandbox` → `/game`)。沙盒已在功能上完全取代前者。

### 触发
沙盒上线后,"AI 自动对抗" 成为冗余入口;且沙盒复用了其底层 `debugAutoAi` 机制,留下两套并存的概念。

### 决策
- **删除"AI 自动对抗"产品层**:大厅卡片、`createDebugAutoAiRoom`、`addDebugAi`/`removeDebugAi`、`createDebugAutoAiPlayers`、`updateDebugAutoAiSequentialSpeech`、对应 gateway 事件与前端控件。
- **把沙盒复用的底层 `debugAutoAi` 机制彻底重命名为沙盒自有**(用户确认"彻底重命名"):
  - `room.debugAutoAi` 标志 → 删除,改以 `isSandboxRoom(room)`(= `sandboxScenarioId != null`)判定。
  - `debugAutoAiSequentialSpeech` → 删除(沙盒恒为顺序发言)。
  - `debugAutoAiSpeech`(pass 状态)→ `sandboxSpeech`;speech loop 系列方法 `*DebugAutoAiSpeech*` → `*SandboxSpeech*`。
  - `updateDebugModel`/`deleteDebugAutoAiRoom` → `updateSandboxPlayerModel`/`deleteSandboxRoom`;事件 `debug.ai.updateModel`/`debug.ai-room.delete` → `sandbox.player.updateModel`/`sandbox.room.delete`。
  - `canStartDebugAutoAiRoom` → `canStartSandboxRoom`;相关 payload 类型改名/删除。

### 理由
"AI 自动对抗 / debugAutoAi" 作为概念已无独立价值(沙盒是唯一自动对局路径);保留旧名会让代码里并存两套术语,且 `debugAutoAi` 字面上就是被取代的功能。重命名后,自动对局机制由沙盒单一所有,产品对局与沙盒各走各的、概念清晰。

### 影响范围
- 后端:`game.types`/`game.rules`/`game.service`/`game.snapshot`/`game.gateway`(删 `debugAutoAi` 字段与产品调试方法、改名复用机制);清理无用常量(`DEBUG_AUTO_*`/`ACTIVE_ICEBREAKER_PERSONA_ID`)与 `createAiPlayer`/`hasActiveIcebreaker`。
- 前端:`game-types`/`game-client`(删产品调试方法、改名复用方法);大厅删「AI 自动对抗」卡、沙盒卡 class 改 `sandbox-entry`;`/game` 观战模式改按 `sandboxScenarioId` 判定、身份标签用沙盒角色;`/room` 删全部调试控件、回归纯普通房型。
- 快照:`RoomSnapshot` 不再导出 `debugAutoAi`/`debugAutoAiSequentialSpeech`。

### 涉及文档
《对局引擎·方案设计》《场景与探测·Schema契约》《对局记录·Schema契约》(均无 inline 变更说明需收口;本次为代码层删除,模块文档直接呈现当前状态)。

## #3 优化器候选生成:从「一次调用产 K 个」改为「K 次独立调用各产 1 个」(2026-06-26)

### 背景
《优化器模块·方案设计》§9 的提示词模板要求优化器在**单次调用**里一次性产出 `{{K}}` 个候选子版本(每个候选都含完整 `prompt_text`),`k_children` 即"一次调用产出的候选数"(§13 起步建议 3–5)。§1 的循环图也把"产出 k 个候选"画成优化器单次调用的产出。

### 触发
复核时提出疑虑:**K 越大,单次产出的质量越差——不是"可能",而是几乎必然。** 当 K 取到 10、每个候选又带上千字完整提示词时,这是上万 token 的单次超长输出,会触发几个相互叠加的退化机制,让候选质量与可用率同步塌方。根子不在"K 个候选"本身,而在于把"需要 K 个候选"和"一次调用产出 K 个候选"绑死了。

### 依据原则
> **优化器原则 3:k 个候选要多样,不是单点最优**(《优化器模块·方案设计》§4);**原则 4:小步定向改,不整段重写**(§4)。"一次调用产 K 个"恰恰同时削弱这两条——多样性靠模型在单次输出里硬凑(易退化为同一改法的几种抠字微调),超长输出又让"只改片段、别整段重写"的纪律松动。把多样性收回到外层分配、让每次调用聚焦单候选,才与这两条原则一致。

### 决策
**把"一次调用产 K 个"拆成"K 次独立调用、每次产 1 个(或 ≤2 个)候选",K 个候选靠多次独立采样聚合得到,不再由单次输出硬凑。**

- **聚焦**:每次调用只专注一个候选 → 单候选质量拉满、格式稳、纪律到位、输出 token 短。
- **靶子外层分配**:每次调用在输入里指定**本次只攻的靶子**——从 `weak_dimensions` 取第 i 个破绽(`{{TARGET}}`),并可指定算子(`{{EDIT_TYPE}}`)。不同调用打不同靶、用不同算子,多样性由外层分配,可控且不易作假。
- **去重**:按 `(target, edit_type)` 与文本相似度去重,撞了的补采一次,凑齐 k 个。
- **成本**:K 次短调用 vs 1 次超长调用,总 token 量级相近甚至更省(省掉"维持 K 个候选一致性"的开销),且 K 次调用**可并行**,墙钟时间反而更短。

### 理由:K 增大时三个机制同时恶化
1. **格式先崩,不是创意先崩。** 超长生成里模型越写越飘:排在后面的候选开始偷懒(只给 diff 不给完整 `prompt_text`)、字段缺失、JSON 末尾断裂或括号不配对。第 8、9、10 个候选的 JSON 结构往往先烂掉 → 整批解析失败,坏样本既拖累好样本,又白费算力。
2. **多样性靠单次硬凑,既假又不稳。** 一次产 K 个时,多样性依赖模型在同一次输出里制造差异;模型天然倾向产出"一个改法的几种微调",而非"不同破绽 × 不同算子"的真·多样走法——这恰好违背原则 3。
3. **纪律随输出变长而松动。** 单候选要守的规矩(只攻一个破绽、给可证伪假设、小步定向、别整段重写、优先通用反射)越多越密,输出越长,模型对这些约束的遵循率越低。

拆成独立多次调用后,每次调用的上下文短、靶子单一,三条退化机制各自失去触发条件。

### 替代方案(为何不直接把 K 压小)
直接锁死 K≤3 能缓解退化,但**牺牲了候选数量与多样性,治标不治本**。"想要更多样候选"与"想要单候选质量"看似两难,其实只有"拆成独立多次调用"能两者兼得。因此解法不是减小 K,而是改变产出的组织方式——把"一次产 K 个"换成"K 次各产 1 个、靶子在外层分配"。

### 影响范围
- **§9 提示词模板**:改为**单候选(或 ≤3)版本**。把 `{{K}}`(产出 K 个)降为"针对本次指定的破绽 `{{TARGET}}` / 算子 `{{EDIT_TYPE}}` 产出 1 个候选";运行时变量新增 `{{TARGET}}`、`{{EDIT_TYPE}}`;输出端补一句"只改动片段、给锚点,不要重抄未改部分"(倾向 diff 式输出,既降 token,又防重抄未改部分而引入回退)。
- **§1 循环图**:把"一次调用产出 k 个候选"改为"k 次独立调用聚合出 k 个候选"。
- **§13 可调参数**:`k_children` 的含义从"一次调用产出数"改为"由 k 次独立调用聚合的候选数";在 `parent_sampling` 旁新增**靶子分配策略**(把"破绽 × 算子"组合在 k 次调用间分配、撞样补采)。

### 涉及文档
- 《优化器模块·方案设计》§1 / §9 / §13:本条变更的落点(提示词模板、`k_children` / `parent_sampling` 与新增的靶子分配策略)。模块文档据此更新为当前状态后,以 `〔变更 #3〕` 指回本条。
