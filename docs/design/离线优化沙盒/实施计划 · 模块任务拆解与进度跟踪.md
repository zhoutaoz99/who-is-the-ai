## 2. 全局增量路线（Phase 0 → 5）

> 每个 Phase = 一个可验收里程碑。**未跑通当前 Phase 的验收,不进下一个。** 阶段映射自《总纲 §8》《编排器 §16》《优化器 §15》的搭建顺序。

### Phase 0 ✅ 引擎单局可跑
- **做**:引擎按场景跑完整局,产出可复现 `MatchRecord`(JSON 落盘)。
- **验收**:`POST /sandbox/prepare` + `POST /sandbox/start` 跑通 5 个示例场景,`sandbox-out/` 产出具名 JSON,转录/投票/探测/auto_eval 字段齐全。
- **状态**:✅ 已完成(见 §3.1)。唯一遗留:`judge_eval_needed` 的探测无消费者(M2.9 接)。

### Phase 1 ✅ MVP 闭环（已完成 · 手动/单线）
> 目标:**能跑一轮"评测 → 判优劣 → 晋升",champion 真的能迭代一次。** 这一步不追求防自欺,只要闭环自洽。
> **状态:已完成**。全自动路径 `POST /sandbox/orchestrator/run-generation-auto`;手动路径 `run-generation`(传 prompt_text)。纯逻辑全实测通过,端到端待真实对局验证。
- **M2 裁判(最小)**:客观结果指标(纯算)+ **单裁判、局末一次**盲测可疑度(`suspicion_margin`)+ 否决项。不做逐轮轨迹/八维/多裁判。
- **M3 聚合(最小)**:`run 取均 → (scenario,seed) 配对做差 → scenario 收敛(N=场景数)→ bootstrap 估 CI → MDE+CI 卡 verdict`。先只产**验证信号**。
- **M4 优化器(最小)**:**手动优化器**(人读验证信号+瞄准信号,手改提示词)先验证"信号→改动→评测"通;再接 **LLM 单候选、单靶子**。
- **M5 编排器(最小)**:**单线贪心**(champion → 1 子) + `scripted_intent` 配对评测 + 接受闸门(显著性+MDE+否决不升)。无留出、无种群、无校准。
- **验收**:从一个种子 champion 出发,跑一代 → 编排器判定 promote/reject → 产出新 champion 或写 `tried_and_rejected`,全程落盘可复现。**用一个小场景集(如 5–10 个示例场景 ×2 seed ×3 run)就能验**。

### Phase 2 ⬜ 防过拟合（留出 + 探测轮换 + 无回退闸）
- **M5.7 留出集复核** + **M6.3 探测实例轮换(holdout 隔离)** + **M2.7 逐轮可疑度轨迹**(失败定位)+ 优化集/留出集分层切分。
- **验收**:子版本过优化集闸后,在**未见过探测**的 holdout 上复核;背答案的改动过不了 holdout。

### Phase 3 ⬜ 优化器"会学习"
- **外层派靶(K 次单候选)** + **算子表(类型约束)** + **operator_scoreboard 战绩排序** + **tried_and_rejected 全局记忆** + **假设验证回环** + **validate_prompt 预校验**。
- **验收**:优化器不再盲搜——被拒假设进 `tried_and_rejected`,算子选择随战绩收敛;`recurring` 失败被定向攻击。

### Phase 4 ⬜ 防自欺 + 抗局部最优
- **种群 top-k + 父代采样** + **多裁判集成** + **真人校准(相关性体检 + 回滚)** + 真人为最终裁决。
- **验收**:代理指标与真人结果相关性达标;沙盒说变差/真人说变差时能回滚。

### Phase 5 ⬜ 对齐真人 + 场景库生长
- **失败回灌流水线** + **失败模式台账** + **评测集版本化重基线** + **free 模式旁路探索** + **用真人频率重配比** + **v1 场景库(120)搭建** + (可选)DB 化持久化。
- **验收**:场景库主要由真人失败生长;holdout 验证修复泛化;真人校准确认失败模式真人率下降。
