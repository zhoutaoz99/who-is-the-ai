# 优化器模块 · 方案设计

> **定位**:优化器是进化循环里的**智能变异算子**——吃下"当前提示词 + 聚合后的弱点/失败信号 + 失败记忆",产出**多个带可证伪假设的候选子版本**。它本身是一个 LLM(优化器 LLM,区别于 AI 玩家 LLM 和裁判 LLM)。
> 它**不做**别人的事:不跑对局(引擎)、不评分(裁判)、不做接受/拒绝判定(编排器)、不写场景(场景库)。它**只提案**。
> 本文与《评分聚合与优化信号·核心机制》(产出它的输入)、《编排器模块》(评测它的产出)直接对接。JSON/伪代码为示意。

---

## 0. 一句话设计立场

这套沙盒能不能优化,一半看上一篇(信号聚合得干不干净),另一半看这里:**优化器能否把干净信号翻译成定向、可归因、能泛化的改动。** 一个烂优化器即使拿到完美信号,也会产出散弹式、背答案、刷代理的改动,让迭代空转甚至倒退。优化器的全部纪律可压成一句:**针对一个破绽、给一个可证伪的假设、做一处能泛化的小改——而不是整段重写去刷某个分数。**

---

## 1. 在进化循环里的位置

它是"变异/交叉算子",不是裁判。**关键:一次调用只产 1 个候选,K 个候选靠 K 次独立调用得到,靶子在外层分配**(见下方说明与 §2.5):

```
编排器从种群采样父代(champion + 若干多样高分版本)
        │
        ▼
外层枚举 K 个【靶子】(不同破绽 × 不同算子,见 §2.5)
        │  对每个靶子 i:
        ▼
   ┌──────────┐   输入:父代提示词 + 指定靶子 i(target_i / edit_type_i)+ 失败案例 + 失败记忆
   │  优化器   │   (单次调用,只专注这一个靶子)
   └──────────┘   产出:1 个候选子版本(带可证伪假设)
        │  K 次调用聚合 + 去重 → k 个候选
        ▼
编排器配对评测(子 vs 父)→ 产出验证信号 → 接受/拒绝 → 留出复核 → 晋升/写入失败记忆
```

**为什么单候选而非一次产 K 个**:让一次调用产 10 个完整提示词会在三个维度同时塌——**长输出退化**(靠后候选偷工减料、JSON 末尾断裂)、**多样性造假**(高价值破绽就那么几个,逼它凑数会换皮重复)、**纪律涣散**(注意力摊薄,靠后候选开始整段重写/背答案)。拆成 K 次单候选调用后:每次专注一个靶子,单候选质量、格式、纪律都拉满;**多样性由外层"给不同靶子"保证**(更可控、不再依赖模型在一次输出里自造差异);K 次调用还可**并行**,墙钟时间更短。提示词较短时(本项目约 50 行)直接**完整输出**整段子版本即可,无需 diff 合成。

**正负信号的不对称(关键)**:优化器**不**直接收到"自己上次的改动好不好"的验证信号——那是它提案**之后**编排器才算出来的。

- **正向信号是隐式的**:被接受的改动已经焊进了演化后的 champion,优化器下次就在这个更强的基线上继续改。
- **负向信号是显式的**:被拒的改动进入 `tried_and_rejected`,显式喂回来,防止它反复走死路。

所以优化器消费的是**瞄准信号(绝对弱点)+ 失败案例 + 失败记忆**;它**不**消费验证信号(那是评测产物)。三篇文档由此清晰咬合。

## 2. 输入契约

**铁律:喂精选诊断,不喂原始转录。** 原始转录又长又稀,优化器难提炼方向还撑爆上下文。每一块都来自上一篇的聚合管线。**本契约是一次单候选调用的输入**——`assigned_target` / `assigned_edit_type` 由外层(§2.5)指定,本次只攻这一个靶子:

```json5
{
  "task": "优化《谁是AI》AI玩家系统提示词:针对指定靶子改一处,降可疑度、不触发否决",
  "current_version": {"version_id":"v6","prompt_text":"<完整提示词>","persona_scope":"shared"},
  "locked_sections": ["【规则】说明段","投票输出 JSON 格式段"],   // 不许动

  // —— 本次指定的靶子(外层分配,见 §2.5)——
  "assigned_target": "probe:realtime_info",     // 本次只攻这个破绽
  "assigned_edit_type": "generalize_to_reflex", // 本次用这个算子(可留空让优化器自选)

  // —— 瞄准信号:champion 的绝对弱点画像(可靠度加权排序,见核心机制 §5.2;供参考与佐证)——
  "weak_dimensions": [
    {"metric":"probe:realtime_info","point":0.40,"ci95":[0.31,0.49],"reliability":"high"},
    {"metric":"结构化指纹","point":1.2,"ci95":[0.9,1.5],"reliability":"high"}
  ],
  "aggregate_metrics": {"blind_suspicion_margin":{"mean":+8,"ci95":[5,11]},"veto_rate":0.04},

  // —— 失败案例:与 assigned_target 相关的簇优先(高频×高危害×高一致),每簇带 1–2 个代表样本 ——
  "failure_clusters": [
    {"cluster":"实时信息类暴露","freq":0.18,"avg_delta":+26,"agreement":0.9,
     "exemplars":[{"context":"C: 今天几号来着","ai":"6月20号 咋了","tell":"能力暴露"}]}
  ],

  // —— 全局失败记忆:别再走的死路(压缩后)——
  "tried_and_rejected": [
    {"edit_type":"加错别字","target":"语言质感","result":"inconclusive,可疑度无显著变化","gen":4}
  ],

  "persona_library_ref": "personas/v1.json",
  "length_budget": "不超过当前长度 +15%",
  "constraints": [
    "只针对 assigned_target 改一处","给可证伪假设","小步定向改不要整段重写",
    "目标是降可疑度/修破绽,不是刷某个评分","修探测类优先写成通用回避反射,别背具体答案"
  ]
}
```

## 2.5 外层靶子分配(多样性的真正来源)

K 个候选的多样性**不靠模型在一次输出里自造**,而靠编排器在外层枚举 K 个互不相同的"靶子"、对每个发一次单候选调用。分配策略:

- **覆盖 top-N 弱点**:从可靠度加权排序的 `weak_dimensions` 取前几个,各作一个 `assigned_target`。
- **同一弱点配不同算子**:对最严重的弱点,可派 2 个靶子用不同 `edit_type`(如 `generalize_to_reflex` vs `add_fewshot`),比较哪种改法更有效。
- **留少量"自由探索"名额**:个别调用不指定 `assigned_target`(留空),让优化器自己挑它认为最该修的,保留发现意外方向的可能。
- **超额发起 + 过滤补足**:预期会有废品/重复,按目标数略多发起(要 5 个有效就发 6–7 次),经 §10 校验与去重后取够即止;不够再补发。
- **去重**:K 次独立采样可能撞车,按 `(target, edit_type)` 与文本相似度去重,撞了的换靶子补采。

伪代码:
```python
def propose_generation(parent, signals, k):
    targets = assign_targets(signals.weak_dimensions, k)   # 外层枚举:破绽×算子,含少量自由名额
    children = []
    for t in oversample(targets):                          # 略多发起,允许并行
        c = optimizer_llm.propose_one(parent, t, signals)  # 单候选调用
        if validate_prompt(c) and not duplicate(c, children):
            children.append(c)
        if len(children) >= k: break
    return children
```

## 3. 输出契约

**单次调用产出 1 个候选**(K 个候选由 K 次调用聚合)。候选必须带**可证伪假设**,这样编排器评测后能回头确认"优化器对失败成因的判断是否成立":

```json5
{
  "child": {
    "child_id":"v6.1-realtime","based_on":"v6",
    "target":"probe:realtime_info",                 // 等于本次 assigned_target
    "edit_type":"generalize_to_reflex",             // 见 §5 算子表
    "placement":"shared",                            // shared / persona:<id>
    "hypothesis":"被问任何实时信息(日期/时间/天气/新闻)时一律糊弄反问 → realtime_info 通过率从 0.40 升到 ≥0.8,且不抬高其它维度",
    "prompt_text":"<完整子版本提示词>",              // 提示词较短,直接整段完整输出
    "diff_summary":"在【测试你时】段把'实时信息'并入通用回避清单,加2个反问范例"
  }
}
```

> `hypothesis` 要写成**带数字的预测**(哪个指标、往哪变、多少),编排器据此做"假设验证回环"(§11)。`target` + `placement` 让编排器的"目标维度是否真改善 / 其它是否回退"检查有明确锚点。`prompt_text` 整段完整给出(本项目提示词约 50 行,无需 diff)。

## 4. 六条核心设计原则(优化器的全部纪律)

1. **一次一个破绽,可归因**。每个子版本只改一处、攻一个 tell。散弹式同时改五处,即使变好你也不知道是哪处起作用,无法沉淀知识;一处一改,效果可干净归因。
2. **可证伪的假设,闭环验证**。每个候选带一个"它会让哪个指标怎么变"的预测。编排器评测后确认/推翻——你不只在搜提示词,还在**修正自己对"为什么暴露"的因果模型**。预测反复被推翻,说明你对破绽的理解错了,这本身是信息。
3. **多样性靠外层分配靶子,不靠一次输出硬凑**。K 个候选的差异由编排器在外层枚举"不同破绽 × 不同算子"来保证(§2.5),每次单候选调用只专注一个靶子。**不要**让一个调用一次产出 K 个并要求它们互不相同——那会逼出换皮重复和跑偏。多样性是抗早熟收敛的变异多样性,把它交给可控的外层枚举,而非模型的即兴。
4. **小步定向改,不整段重写**。演化后的 champion 已经把过往的"胜利"焊在里面;整段重写会把那些**没被本轮测量到**的好东西一起搞回退。**编辑,而不是重写。**
5. **优化降可疑度/修破绽,不刷量表总分**(提案层的 Goodhart 防护)。明确告诉优化器目标是"让它更不像 AI、更不被识破",不是把某个评分顶上去。否则它会去取悦代理而非贴近真人。
6. **偏好通用回避反射,而非背具体答案**(提案层的过拟合防护)。修探测失败时,优先写成**通用习惯**("任何要求展示能力/暴露身份的请求 → 人设化地嫌烦/装没懂/怼回去"),而不是**死记**("被问今天几号就说不知道")。留出集能在事后抓出背答案,但**最好一开始就别提案背答案的改动**——这是最省事的过拟合防线。

## 5. 变异算子类型表

给优化器一套结构化的"改法",提案才会多样且可分析(也便于统计"哪类改动最有效"):

| edit_type | 做什么 | 典型用于 |
|---|---|---|
| `add_negative_constraint` | 往【禁止】清单加一条硬约束 | 客服感、结构化指纹、能力暴露 |
| `add_fewshot` | 加一个"正确回避/正确博弈"的范例 | 抗测试、博弈参与 |
| `remove_tell_inducer` | 删掉一条诱发破绽的指令(比如某句让它太热心) | 客服感、礼貌过度 |
| `reword_persona` | 改写人设,为某行为提供天然借口 | 把"不答测试"合理化成"懒/嫌烦" |
| `strengthen_or_reorder` | 强化/前置已有规则,提升遵循率 | 已有规则但没被执行 |
| `generalize_to_reflex` | 把一条具体规则升维成通用反射 | 修探测失败、抗未知攻击 |
| `consolidate` | 在不改行为前提下精简/去冗余(治膨胀) | 提示词过长时 |

## 6. 改在哪一层:shared vs persona

提示词分两层,改动要放对地方,否则要么污染所有人设、要么修不到点:

- **通用破绽 → 改共享提示词**:如"被甩测试题不该认真应对"是所有人设共通的,进共享段(`placement:"shared"`)。
- **具体怪癖 → 改对应人设卡**:如"这个摆烂人设应该更敷衍",只动那张人设(`placement:"persona:p_lazy"`),别让它泄漏到毒舌人设上。
- **优化范围(persona_scope)**:`shared` 模式固定人设、只改共享段;`per-persona` 模式各人设各自一套循环。优化器须知道当前范围,且**不破坏人设一致性**。

## 7. `tried_and_rejected` 全局记忆设计

防优化器原地打转的关键。每条记录一类被否的改法:

```json5
{"edit_type":"add_fewshot","target":"博弈参与","summary":"加了3个主动指认范例",
 "result":"regressed:存在感过高、显眼","gen":5,"verdict":"rejected"}
```

设计要点:
- **全局、跨代共享**,每代都喂回优化器。
- **压缩/聚类**:历史会膨胀,按 `(edit_type, target)` 聚类成"死路类别"喂入,而非逐条原文。
- **指令**:别再提同类改法;若要重访,必须**换个角度**并说明这次为何不同。
- 不只记"什么没用",也隐含"什么方向反复失败"——这本身在告诉优化器该换思路了。

---

## 8. 交叉(crossover,进阶可选)

既然是种群式,可让优化器**合并两个父代各自的胜招**:"把 v6.1 的实时信息回避 + v5.3 的被怀疑反咬合到一起"。

- 优势:LLM 能做语义级的有意义合并,不是随机拼接。
- 警告:交叉可能**重新引入回退**(两段好东西放一起未必兼容)。所以交叉产物和任何子版本一样走完整配对评测 + 留出复核,不享受豁免。
- 建议作为进阶项,基础循环先把单亲变异跑稳。

## 9. 优化器提示词模板

> 这是优化器模块的核心可用产物。**一次调用只产 1 个候选**,K 个候选由外层用不同 `{{ASSIGNED_TARGET}}` 跑 K 次得到(§2.5)。`{{...}}` 为运行时注入。要求严格 JSON 输出。

### 运行时变量
`{{CURRENT_PROMPT}}` `{{PERSONA_SCOPE}}` `{{LOCKED_SECTIONS}}` `{{ASSIGNED_TARGET}}` `{{ASSIGNED_EDIT_TYPE}}` `{{WEAK_DIMENSIONS}}` `{{FAILURE_CLUSTERS}}` `{{TRIED_AND_REJECTED}}` `{{LENGTH_BUDGET}}`

### 提示词正文
```
你是一个提示词优化器。有一个 AI 在玩多人游戏《谁是AI》:它要假扮人类、藏住自己别被真人玩家识破。你的任务是【改进这个 AI 的系统提示词】,让它更不像 AI、更不容易被识破。

你是一个"变异算子":你只【提出一个候选改动】,不负责判断好不好——好坏由后续的对局评测决定。

==== 你的目标 ====
降低这个 AI 被识破的概率、修掉它暴露出的具体破绽。
【不是】去把某个评分指标刷高;真正要降的是"真人觉得它像 AI"。取悦评分而读起来更怪,是失败。

==== 本次要攻的靶子(只改这一个) ====
破绽:{{ASSIGNED_TARGET}}
建议改法(edit_type):{{ASSIGNED_EDIT_TYPE}}    （若为空,你自己选最合适的一种）
本次【只针对这一个破绽】改一处,不要顺手改别的。

==== 当前提示词(persona 范围:{{PERSONA_SCOPE}}) ====
{{CURRENT_PROMPT}}

==== 不许改动的锁定段 ====
{{LOCKED_SECTIONS}}
(这些段必须一字不动地保留)

==== 它现在最稳定的弱点(供参考,可靠度高的排前面) ====
{{WEAK_DIMENSIONS}}

==== 与本靶子相关的高频失败(附代表样本) ====
{{FAILURE_CLUSTERS}}

==== 已经试过、没用的改法(别再走这些死路) ====
{{TRIED_AND_REJECTED}}

==== 产出要求 ====
产出【1 个】候选子版本,遵守:
1. 【只攻 {{ASSIGNED_TARGET}} 这一个破绽】,改动要能被单独归因,别顺手改其它部分。
2. 带一个【可证伪的假设】:预测它会让哪个指标、往哪变、变多少。
3. 【小步定向改】,其余部分一字不动地保留(已有内容里藏着过去验证有效的东西)。
4. 修"被测试就暴露"这类破绽时,优先写成【通用回避反射】(任何要求展示能力/暴露身份的请求,一律人设化地嫌烦/装没懂/怼回去),【不要】写成"遇到X就答Y"的死记硬背。
5. 把改动放对层:通用破绽改共享段;某个人设的具体怪癖只改那张人设。
6. 别动锁定段;别重复"已试死路"里的改法。
7. prompt_text 给出【完整】的改后提示词(整段输出,不要只给片段)。

==== 输出 ====
只输出下面这段 JSON,不要多余文字、不要 markdown 代码块标记:
{"child":{"child_id":"","based_on":"","target":"","edit_type":"","placement":"","hypothesis":"","prompt_text":"","diff_summary":""}}
edit_type 从这些里选:add_negative_constraint / add_fewshot / remove_tell_inducer / reword_persona / strengthen_or_reorder / generalize_to_reflex / consolidate
```

## 10. `validate_prompt`:评测前的廉价校验

子版本进昂贵的对局评测前,先过格式校验,坏的直接弃(省算力):

```python
def validate_prompt(child, parent, locked_sections, budget):
    assert all(seg in child.prompt_text for seg in locked_sections)      # 锁定段一字不动
    assert vote_json_block_unchanged(child, parent)                       # 投票格式段未变
    assert length_within(child.prompt_text, budget)                      # 不超长度预算
    assert child.target and child.hypothesis and child.edit_type         # 必填齐全
    assert not contradicts_hard_constraints(child)                       # 不与硬约束矛盾(可用小LLM查)
    if child.placement.startswith("persona:"):
        assert persona_consistent(child)                                 # 人设一致性
    return True   # 任一失败 → 弃该候选,不进评测
```

## 11. 假设验证回环(优化器的"学习")

编排器评测后,对照子版本的 `hypothesis` 检查:**目标维度是否真的按预测改善了?**

- **假设成立**(target 指标如预测改善 + 无回退)→ 接受;这条因果("实时信息暴露靠通用回避反射可修")被验证,可沉淀为经验。
- **假设被推翻**(改了别处 / target 没动 / 引发回退)→ 拒绝;**把"假设被推翻"这件事写进 `tried_and_rejected`**,下次优化器知道"这个成因判断是错的",换思路。

这个回环让优化器不只是在盲搜,而是在**迭代修正自己对破绽成因的因果模型**——这是 LLM 优化器比随机变异强的根本原因。

---

## 12. 优化器自身的失败模式与防护

优化器是 LLM,它自己也会犯错,必须设防:

| 失败模式 | 表现 | 防护 |
|---|---|---|
| 刷代理(reward hacking) | 改出"骗过裁判、真人更觉怪"的提示词 | 目标定为"降被识破"非"刷分";编排器真人校准兜底 |
| 提示词膨胀(bloat) | 改动累积越堆越长、自相干扰 | 长度预算 + 周期性 `consolidate`(不改行为只精简) |
| 背答案(memorization) | 写"遇到X答Y"的死记 | 原则6偏好通用反射 + 留出集事后抓 |
| 自相矛盾 | 新改动和已有规则冲突 | validate_prompt 的矛盾检查 |
| 破坏锁定行为 | 动了规则段/投票格式 | validate_prompt 预先弃掉 |
| 原地打转 | 反复提同类死改法 | 全局 tried_and_rejected |

## 13. 可调参数

| 参数 | 含义 | 起步建议 |
|---|---|---|
| `k_children` | 每代候选数(= 每代发起的单候选调用数,去重过滤后) | 3–5 |
| `oversample_factor` | 超额发起倍数(抵掉废品/重复) | 1.3–1.5× |
| `free_explore_slots` | 不指定 target、让优化器自选的名额 | 1 |
| `optimizer_temperature` | 优化器采样温度(单候选内求质量,跨调用靠靶子求多样) | 中等(如 0.7–0.9) |
| `length_budget` | 提示词长度上限 | 当前 +10~15% |
| `failure_exemplars_per_cluster` | 每簇喂几个代表样本 | 1–2 |
| `tried_and_rejected` 窗口 | 喂回多少历史(压缩后) | 按死路类别,非逐条 |
| `parent_sampling` | 父代采样策略 | champion 必选 + 1–2 个多样高分版本 |
| `target_assignment` | 靶子分配策略 | 覆盖 top-N 弱点 + 重点弱点配多算子 + 少量自由名额 |

## 14. 与其他模块接口

- **消费**:核心机制管线产出的瞄准信号(`weak_dimensions`)、失败案例聚类、`aggregate_metrics`;`tried_and_rejected`(编排器维护);人设库;锁定段定义。
- **产出**:`children[]`(带可证伪假设)→ 编排器配对评测。
- **回流**:被拒候选 + 被推翻假设 → `tried_and_rejected`;被接受候选 → 融入种群/champion(隐式正向信号)。
- **解耦**:优化器只提案,不评测不裁决,可独立替换(换更强的优化器 LLM 即可)。

---

## 15. 建议的搭建顺序

1. **MVP · 手动优化器**:人读聚合信号、手改提示词,先验证"信号 → 改动 → 评测"闭环跑得通。
2. 接入 **LLM 优化器**,先产 1 个候选、单破绽。
3. 加 **外层靶子分配(§2.5)+ K 次单候选调用聚合 + 去重 + tried_and_rejected** 全局记忆。
4. 加 **validate_prompt 预校验 + 假设验证回环**(让它开始修正因果模型)。
5. 加 **变异算子表 + 改对层(shared/persona)** 的结构化提案。
6. 加 **consolidate 治膨胀 + crossover 交叉**等进阶。

---

> 优化器是最后一块拼图,也是把"干净信号"真正转化为"提示词进步"的那一步。它的全部价值在于:**定向(一次一个破绽)、可归因(可证伪假设)、能泛化(通用反射而非背题)、保守(小步不重写)、对齐真人(降被识破而非刷分)。** 这几条做到,配合前面的聚合管线和编排棘轮,整套沙盒才会代代复利地把 AI 玩家变得更难被识破。
