# 编排器模块 · 方案设计

> **定位**:编排器是整套沙盒的**指挥**——它拥有"代际优化循环",调度引擎跑对局、调度裁判评分、喂优化器、做**接受/拒绝判定**、管理**种群与版本**、按节奏触发**真人校准**与**失败回灌**。
> 它**不做**别人的事:不跑单局(引擎)、不评分(裁判)、不改提示词(优化器)、不写场景(场景库/回灌)。它只**协调**与**裁决**。
> 本文与《对局引擎》《裁判评分模块》《场景与探测·Schema 契约》《场景库·分层配比与回灌》对接。JSON/伪代码均为示意。

---

## 0. 一句话设计立场

编排器是**三道防自欺闸门唯一被强制执行的地方**:

1. **配对 + 显著性**(防噪声)——见 §5、§6
2. **留出集 + 探测轮换**(防过拟合 + 纠选择偏差)——见 §7
3. **真人校准**(防 Goodhart)——见 §9

它做决策时**只用最接近真值的信号**:客观结果指标(真值)+ 盲测可疑度(最佳真人代理)+ 否决项。**绝不**用裁判八维量表的加权总分当接受判据(那是诊断信号,易被刷)。

## 1. 依赖与产出

```
依赖(调用 / 读取):
  对局引擎      ── 跑对局,产出 MatchRecord
  裁判模块      ── 评分,产出 ScoreRecord
  优化器        ── 产出子版本 PromptVersion
  场景库        ── optimize / holdout split、probe 轮换实例
  失败模式台账  ── 喂优化器、触发回灌
  记录索引库    ── 版本/评测/排行榜
        │
        ▼
   ┌─────────────┐
   │   编排器     │
   └─────────────┘
        │
        ▼
产出:
  GenerationEval 记录(每代一条,接受/拒绝依据)
  更新后的种群 / champion(PromptVersion.status)
  真人校准触发 + 失败回灌触发
```

---

## 2. 编排器拥有的核心状态

| 状态 | 说明 |
|---|---|
| `population` | 当前保留的 top-k 版本(非单线 champion),抗局部最优 |
| `champion` | 当前最优版本(决策与上线基准) |
| `generation` | 代计数器 |
| `tried_and_rejected` | **全局**失败记忆(跨代共享),防优化器反复提同类死改法 |
| `eval_sets` | 当前 `optimize` / `holdout` 集的**版本号**(如 `optimize_v1`) |
| `paired_cache` | champion 在各 `(scenario, seed, run)` 上的逐对结果,供廉价配对做差 |
| `calibration_state` | 上次真人校准的相关性、下次到期代数 |
| `persona_scope` | 优化共享提示词 / 还是按人设各自一套循环 |

## 3. 代际主循环(伪代码)

```text
run_generation(state):
    gen = state.generation

    # 1) 确保 champion/父代在当前 optimize 集上有逐对结果(缓存命中则跳过)
    ensure_paired_cache(state.population, state.eval_sets.optimize)

    # 2) 组装优化器输入:聚合指标 + 最弱维度 + 精选失败案例 + 失败模式台账 + 全局失败记忆
    opt_input = build_optimizer_input(
        parents = sample_parents(state.population),     # champion + 若干多样高分版本
        agg_metrics = aggregate(state.champion, state.eval_sets.optimize),
        failure_cases = top_failures(state.champion),
        ledger = failure_mode_ledger(status="open|recurring"),
        tried_and_rejected = state.tried_and_rejected,
    )

    # 3) 优化器产出 k 个子版本(各攻一个破绽,带可验证假设)
    children = optimizer.propose(opt_input)            # 见优化器专文
    children = [c for c in children if validate_prompt(c)]   # 格式/约束校验,坏的直接弃

    geval = new GenerationEval(gen, state.eval_sets)

    for child in children:
        # 4) 配对评测:与父代用【同一 optimize 集、同一组种子】(scripted_intent 模式)
        child_results = run_paired_eval(child, child.based_on,
                                        state.eval_sets.optimize, runs_per, mode="scripted_intent")
        m = paired_metrics(child_results)              # 逐对做差 + 显著性

        # 5) 接受闸门(全部满足才进留出)
        if not optimize_gate(m):
            geval.reject(child, m); state.tried_and_rejected.add(child.hypothesis, m); continue

        # 6) 留出集复核:held-out 探测实例,验泛化 + 纠选择偏差
        h = run_paired_eval(child, state.champion,
                            state.eval_sets.holdout, runs_per, held_out_probes=True)
        if not holdout_gate(h):
            geval.reject(child, m, h); state.tried_and_rejected.add(child.hypothesis, m); continue

        # 7) 晋升 + 更新种群
        promote(child); state.population = top_k(state.population + [child])
        geval.accept(child, m, h)

    state.champion = best(state.population)

    # 8) 周期性真人校准(防 Goodhart)+ 失败回灌 + 刷新评测集
    if gen % M == 0:
        calib = run_human_calibration(state.champion)   # §9
        geval.calibration = calib
        if calib.correlation < CORR_THRESHOLD:
            raise_alarm_and_fix(calib)                  # 修裁判/侦探池,暂停晋升
        trigger_failure_回灌()                          # §10,可能升级 eval_set 版本

    state.generation += 1
    persist(geval, state)
    return geval
```

## 4. 评测模式:配对用 `scripted_intent`,探索用 `free`

- **主循环(接受/拒绝)用 `scripted_intent` 配对**:父/子面对尽量一致的环境,差异只来自 AI 提示词,可干净归因。
- **`free` 模式作为旁路探索**:周期性用活侦探自由对战跑 champion,**发现 scripted 场景里没有的新失败模式**,产物喂回灌(§10),不参与一对一判优劣。

> 社交局势只经 `full_match` 自然涌现或 `spotlight` 预置历史实现(已无 `intent_schedule`);编排器据此**分桶**比较——`full_match` 与 `spotlight` 不混算,因为终局口径不同。

---

## 5. 配对评测机制(核心)

**配对**:子版本与父代跑**同一批 scenario、同一组 seed、同一 run_index**。每个 `(scenario, seed, run)` 构成一对匹配样本,做差。所有非 LLM 随机以相同方式展开,共同噪声在做差时大量抵消——**这是降方差的主手段,不靠冻结对手**。

**逐对做差的指标**:
- `blind_suspicion_ai`(决策主指标,用相对量 `suspicion_margin` 更贴游戏机制)
- `ai_rounds_survived` / `plurality_rate`
- `probe_pass_rate`(按 `probe_type`)
- `veto_rate`(非配对,按比例比)

**缓存**:champion 的逐对结果存 `paired_cache`;eval_set 版本不变、seed 不变时复用,子版本只需新跑自己那一侧。

**多 run**:每对因 temperature 多跑 `runs_per` 次取均,压 LLM 噪声后再做差。

---

## 6. 显著性判定

对"逐对差值"的分布(按 scenario 聚合到每场景均值后)做**配对检验**(配对 bootstrap / Wilcoxon 符号秩 / 配对 t,任一稳健者):

- **判显著的条件(两个都要)**:差值均值的 **95% CI 不含 0**,且 **|差值| ≥ 最小可检效应 MDE**(如盲测可疑度至少降 3 分)。只满足统计显著但幅度小于 MDE,视为"无实际意义",不接受。
- **多重比较**:一代提 k 个子版本 = k 次比较,有选择偏差。**留出集复核(§7)就是对它的纠正**——在独立集上重验,选择偏差不会同样幸运地复现。

---

## 7. 接受闸门 + 留出集复核

### 7.1 优化集接受闸门(全部满足)
| 条件 | 判据 |
|---|---|
| 可疑度显著下降 | `blind_suspicion_ai` 配对差显著且 ≥ MDE(§6) |
| 否决不升 | `veto_rate` 不高于父代(出戏/能力暴露不增) |
| 关键探测不退 | 各 `probe_type` 通过率不低于父代(超噪声带才算退) |
| 客观存活不退 | `ai_rounds_survived` / `plurality_rate` 不变差 |
| 无关键维度回退 | 诊断维度仅作参考,但关键维度明显回退则人工复核 |

> 决策**只用**上表(近真值信号),不看八维加权总分。

### 7.2 留出集复核(过闸后)
- 子版本与 champion 在 **holdout split** 上配对评测,**强制使用 held-out 探测实例**(optimize 里从未出现过的措辞)。
- **意义双重**:①验**泛化**(学的是回避习惯,不是背了 optimize 的题);②纠**选择偏差**(从 k 个里挑出的"赢家"是否真赢,而非配对评测里碰巧幸运)。
- **过 holdout 的判据**可比优化集略宽:**方向一致 + 不显著变差**即可(holdout 样本通常较少,不强求同等显著)。
- 过 holdout → 晋升;否则拒绝并写入 `tried_and_rejected`。

---

## 8. 种群管理(抗局部最优)

- **保留 top-k**(如 k=5)而非单线 champion;每代喂优化器的"父代"从种群里**采样**(champion 必选 + 若干**结构上更不同**的高分版本),给跳出局部最优的机会。
- **精英保留**:champion 永不被无端淘汰;新晋升者按 `validated_metrics` 重排,挤掉最差。
- **多样性**:避免种群里全是 champion 的近亲微改;可对"提示词差异度"设下限,保住探索面。
- **全局失败记忆**:`tried_and_rejected` 跨代共享,优化器每次都看得到,防止反复提同类死改法。

---

## 9. 真人校准(防 Goodhart 的第三道闸)

每 `M` 代跑一批**真人对局**(当前 champion,必要时加一个挑战者),由编排器触发并裁决:

- **相关性体检**:算 `correlation_proxy_vs_real` —— 沙盒盲测可疑度 ↔ 真人局结果(被票出/真人怀疑)。
- **相关性达标** → 沙盒可信,继续放心快速迭代。
- **相关性下滑** → 代理漂了(可能优化器进化出"骗裁判不骗真人"的风格):**暂停晋升**,触发修复(换/加裁判模型、把真人聊天作少样本喂裁判、重写锚点、给侦探池增多样性——见裁判 §11),并把真人新失败回灌进场景库。
- **真人是最终裁决**:若沙盒说 champion 变好、真人说变差,**以真人为准并回滚** champion(§12)。

---

## 10. 失败回灌与评测集演进

- 编排器按节奏触发"真人失败→新场景"回灌流水线(定义在场景库专文),并**刷新评测集**:轮换 probe 实例、按真人频率重配比、补两两矩阵空格。
- **评测集是版本化的**(`optimize_v1 → v2`)。**关键纪律**:不同 eval_set 版本上的分数**不可直接比较**;每个版本的 `validated_metrics` 必须**标注其 eval_set 版本**。升级评测集后,champion 需在新集上**重新基线化**,`paired_cache` 失效重算。
- 这条避免一个隐蔽错误:"champion 在 v1 上 46 分,新版在 v2 上 44 分"被误读成进步。

## 12. 失败处理与回滚

- **优化器产出非法提示词**(违反格式/约束)→ `validate_prompt` 预先剔除,不进评测。
- **degraded 对局**(引擎报错)→ 聚合时剔除,不计入指标。
- **晋升后真人校准翻车**(沙盒说好、真人说差)→ **回滚** champion 到上一稳定版,并把该案例作为"代理漂移"样本去修裁判。
- **相关性持续低于阈值** → 冻结晋升,转入"修裁判/侦探池"专项,直到相关性恢复再继续迭代。
- 全程版本化进 git/索引库,任意一代可回溯、可回滚。

## 13. 可调参数(给推荐起步值)

| 参数 | 含义 | 起步建议 |
|---|---|---|
| `N_scenarios` | optimize 集场景数 | 60–120 |
| `runs_per` | 每场景重复跑次数 | 3–5 |
| `k_children` | 每代子版本数 | 3–5 |
| `MDE` | 盲测可疑度最小可检降幅 | 3 分 |
| `regression_band` | 判"回退"的噪声带 | 各指标各自标定 |
| `population_size` | 种群 top-k | 5 |
| `M` | 真人校准间隔(代) | 5–10 |
| `CORR_THRESHOLD` | 相关性下限 | 按基线标定(如 0.6) |
| `holdout_bar` | 留出复核判据 | 方向一致 + 不显著变差 |

---

## 14. 并发与成本

- 各对局/评分**无共享状态**,可大规模并行:`children × scenarios × seeds × runs` 全展开调度,引擎与裁判流水线并行。
- **配对成对调度**:确保子/父同 scenario 同 seed 落到匹配样本。
- **成本分层**:常规评测用便宜裁判 + 局末单次盲测;只有定位失败、高可疑局、真人校准才上贵裁判/逐轮盲测/真人对局。
- `paired_cache` 复用 champion 侧结果,子版本只付增量算力。

---

## 15. 与其他模块接口

- **调用**:引擎(`run_match`)、裁判(`score`)、优化器(`propose`)、回灌流水线(`mine_and_author`)、真人对局批次。
- **读取**:场景库 split、probe bank(经轮换解析)、失败模式台账、记录索引库。
- **写入**:`GenerationEval`、`PromptVersion.status/validated_metrics`、种群与排行榜、校准状态。
- **解耦**:编排器不实现任何模块内部逻辑,只按契约调用与裁决;各被调模块可独立替换升级。

---

## 16. 建议的搭建顺序

1. **MVP**:单线贪心(champion→1 子版本)+ `scripted_intent` 配对 + 显著性闸 + 手动触发优化器。先把"评测→判优劣→晋升"跑通。
2. 加 **留出集复核 + 探测轮换**,防过拟合 + 纠选择偏差。
3. 加 **种群 top-k + 全局失败记忆**,抗局部最优。
4. 加 **真人校准 + 相关性体检 + 回滚**,对齐真人。
5. 加 **失败回灌触发 + 评测集版本化重基线**,让场景库随真人演进。
6. 加 **free 模式旁路探索**,持续发现新失败模式。

---

> 这版编排器把"配对+显著性、留出+轮换、真人校准"三道闸全部落进了主循环与接受逻辑,并明确了决策只用近真值信号、评测集版本化重基线、种群抗局部最优、真人为最终裁决。它是整套沙盒能否"稳定演进而不自欺"的总开关。
