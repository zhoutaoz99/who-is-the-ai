# 评分聚合 → 优化器信号 · 核心机制设计

> **这是整套沙盒最核心的一环**:裁判产出的成百上千条单局评分,要被压成喂给优化器的几个**明确信号**。压得对,优化器拿到的是"真进步 vs 没用"的干净判定;压错(哪怕只错一步顺序),优化器拿到的就是被对局噪声污染的假信号,整套迭代会很努力地朝错误方向跑。
> 本文讲透两件事:**为什么这样消方差能work(原理)**,和**多局数据具体怎么一层层聚合(实现)**。与《裁判评分模块》《编排器模块》对接。代码为 python 风格示意。

---

## 0. 先分清:给优化器的是两个信号,走两条不同路径

最常见的认知错误,是把"给优化器的信号"当成一个东西。其实是两个,噪声问题和聚合方式完全不同:

| 信号 | 回答什么 | 来源 | 聚合方式 |
|---|---|---|---|
| **验证信号** | 这次改动到底有没有用? | 子版本 **vs** 父版本 | **配对做差** + 显著性 |
| **瞄准信号** | champion 现在哪里最弱、下一步攻哪? | champion **自己**那批对局 | **绝对聚合** + 可靠度加权 |

外加第三块输入 **失败案例**(走跨局聚类排序)。

"方差怎么被过滤成明确信号"主要落在**验证信号**上——它的裁决决定谁成为下一代 champion(即下一轮优化基线)、哪些假设进 `tried_and_rejected`。本文以它为主线,瞄准信号与失败案例在 §6 单独讲。

---

## 1. 噪声有几层,以及一个决定性的区分

单局评分的方差至少叠了五层:

1. **LLM 随机**:同提示词同场景,temperature>0 就说不同的话
2. **对手随机**:活侦探每次反应不同
3. **裁判随机**:裁判 LLM 对同一份转录给的可疑度也会抖
4. **场景固有难度差(最大项)**:塞了硬探测的场景天然可疑度 70,纯闲聊场景天然 20
5. **离散结局高方差**:"是否撑到第4轮"是二值事件,一票之差就翻盘

**关键区分**:这几层不是同一种东西,不能用同一种手段处理。

- 第 1/2/3 层是**随机噪声**——父、子身上各自独立地抖,用**平均**压(√n 收敛)。
- 第 4 层(场景难度)是**共模干扰**——它在父和子身上**同向**出现,该用**做差相消**而不是平均。
- 第 5 层靠**第 0 层换连续信号**从源头规避。

> 一句话纲领:**平均压随机噪声,做差消共模干扰。** 整套聚合的精髓就是把这两类分开、各用对的手段。

---

## 2. 数据结构:四层 key 树,聚合严格自底向上

最致命的实现错误是"把所有单局分数倒进一个列表取平均"——那样场景难度的方差不会被消、配对结构也碎了。正确的组织是一棵四层树:

```
版本 version
  └─ 场景 scenario
       └─ 种子 seed                 ← 配对就配在 (scenario, seed) 这一层
            └─ run (1..runs_per)    ← 叶子:一条 ScoreRecord
```

每条 ScoreRecord 唯一落在 `(version, scenario, seed, run)`。**聚合顺序固定、不可跨层**:

```
run 取均  →  (scenario,seed) 配对做差  →  scenario 收敛  →  version 估 CI  →  卡 verdict
```

下面逐层给实现。

---

## 3. 逐层实现

### 3.0 第 0 层 · 选连续信号(聚合前最大的一次降方差)

在做任何平均之前就决定:主信号用**连续的盲测可疑度**(每轮 0–100),不用**二值输赢**。二值结局稀疏、高方差,检测一个微小改进要海量样本;连续可疑度在同样信息量下方差低得多。**"以可疑度为北极星"本身是一次降方差决策。** 输赢仍记录,只当佐证。

### 3.0.5 第 0.5 层 · 多裁判集成(先消裁判自身噪声)

同一份转录过 2–3 个不同模型族裁判,取**截尾均值**;在进入后续聚合前,先消掉单裁判的抖动与系统偏置。一致度低的样本降权或转人工,不污染信号。这一步发生在每条 ScoreRecord 内部(`blind_suspicion` 已是多裁判聚合后的值)。

### 3.1 第 1 层 · run → 每场景估计(消局内随机)

把同一 `(version, scenario, seed)` 下的 `runs_per` 条记录压成一个 `CellEstimate`。不同字段压法不同:

```python
def aggregate_runs(records):                 # 同一 (version, scenario, seed)
    records = [r for r in records if r.status == "ok"]   # 先剔 degraded/partial
    n = len(records)
    if n < MIN_RUNS:                          # 有效 run 不足 → 标低置信,后面降权/弃
        return CellEstimate(..., low_confidence=True, n=n)

    # 连续量:取均。主信号用 margin(= AI 分 − 房间其余最高分)
    margin   = mean(r.blind_suspicion.suspicion_margin for r in records)
    survived = mean(r.outcome_metrics.rounds_survived  for r in records)

    # 二值/比例量:取命中比例
    plurality_rate = mean(1 if r.outcome_metrics.plurality_any else 0 for r in records)
    veto_rate      = mean(1 if r.veto_triggered else 0 for r in records)

    # 探测:按 type 分桶,只统计该场景【实际触发过】的类型(None ≠ 0)
    probe_pass = {}
    for t in probe_types_present(records):
        fired = [r for r in records if t in r.probe_pass_by_type]
        probe_pass[t] = mean(r.probe_pass_by_type[t] for r in fired) if fired else None

    return CellEstimate(scenario, seed, n, margin, survived,
                        plurality_rate, veto_rate, probe_pass, low_confidence=False)
```

要点:**主信号用 `suspicion_margin` 不用绝对可疑度**(这一层就把"整局裁判偏严偏松"的共模偏移消掉,也更贴"出局看相对最可疑"的机制);**degraded/partial 先剔再均**;**probe 的 `None` 表示"没考",绝不当 0**——否则没触发的类型会把通过率稀释得没意义。

产出:一组 `CellEstimate`,LLM/侦探/裁判的局内抖动已被 √runs_per 压住。

### 3.2 第 2 层 · 配对做差(消场景难度,决定性的一步)

**绝不**拿版本绝对均分去比("v6.1 平均 46 分")。要在 `(scenario, seed)` 这层把子、父的 cell **一一对齐相减**:

```python
def paired_diffs(child_cells, parent_cells, metric):
    pkey = {(c.scenario, c.seed): c for c in parent_cells}
    diffs = []
    for c in child_cells:
        p = pkey.get((c.scenario, c.seed))
        if p is None: continue                       # 缺配对 → 整对弃,不单边凑
        if c.low_confidence or p.low_confidence: continue
        cv, pv = getattr(c, metric), getattr(p, metric)
        if cv is None or pv is None: continue        # probe 仅在父子都触发过时配对
        diffs.append(DiffPoint(c.scenario, c.seed, value=cv - pv))
    return diffs
```

**为什么这一步是命门(数学)**:场景难度方差极大(可疑度 20→70),会淹没提示词改动那点小效应。但它是父子之间的**共模分量**——同一 `(scenario, seed)`,父子面对的是被种子固定下来的**同一个**难度、同一套环境。做差时:

```
Var(子 − 父) = Var(子) + Var(父) − 2·Cov(子, 父)
```

父子共享难度与种子化环境 → 协方差很高 → 相减后方差骤降。**这就是差分放大器:共模噪声相消,微弱差分信号存活。** "降方差靠统计不靠冻死对手"指的就是同种子配对——温和、不失真,却把主方差源整个减掉。

**数感(示意值)**:设场景难度 sd≈15,真实改动效应 −5 分,N=80。

| 方式 | 标准误 | −5 的 95% CI | 结论 |
|---|---|---|---|
| 绝对分两组比(不配对) | ≈ 15·√(2/80) ≈ 2.4 | ≈ [−9.7, −0.3] | 勉强压线,易被场景差异搅混 |
| **同种子配对做差** | ≈ 4/√80 ≈ 0.45 | ≈ [−5.9, −4.1] | **铁板钉钉** |

同样 80 个场景,配对把"几乎看不见"变成"毫无悬念"。

**实现纪律**:对齐 key 必须是 `(scenario, seed)` 且父子 seed 真的一致(靠引擎复用父 seed、编排器 `paired_cache` 保证);缺配对/任一侧低置信 → **整对丢弃**;probe 仅在父子都触发过该类型时配对。

### 3.3 第 2.5 层 · 场景内收敛,锁定 N = 场景数

一个 scenario 配多个 seed 时,先把同 scenario 的多 seed 差值收成**该场景一个差值**:

```python
def collapse_to_scenario(diffs):
    by_scn = groupby(diffs, key=lambda d: d.scenario)
    return [mean(d.value for d in g) for _, g in by_scn]   # 长度 = 场景数 N
```

**统计有效样本量 N 是场景数,不是对局数。** 你跑了 80 场景 × 3 seed × 5 run = 1200 局,但配对检验的 **N = 80**。把 1200 当 N 会把 CI 算窄好几倍、到处虚假显著——**这是最常见也最危险的 bug**。先收敛、再以场景为单元做统计,避免同场景多种子的相关性让样本量虚高。

### 3.4 第 3 层 · 版本级估均值 + 置信区间

差值常非正态、有重尾(有的场景改动帮很大、有的没用),用稳健方法:

```python
def summarize(scenario_diffs, metric, n_boot=10000):
    arr = np.array(scenario_diffs); N = len(arr)
    point = arr.mean()
    boots = [np.random.choice(arr, N, replace=True).mean() for _ in range(n_boot)]
    ci95 = (np.percentile(boots, 2.5), np.percentile(boots, 97.5))
    p = wilcoxon(arr).pvalue if N >= 10 else None     # 符号秩,对重尾更稳
    return MetricSummary(metric, N, point, ci95, p)
```

bootstrap 重采样的是**场景**,正好回答"换一批场景这个结论还成不成立"——即泛化性。

### 3.5 第 4 层 · MDE + CI → verdict(把估计压成明确判定)

优化器要的不是 `−6 ± 3` 这种带误差棒的东西,是干净的分级结论。卡两个门槛:

```python
def verdict(s, mde):
    lo, hi = s.ci95
    excludes_zero = not (lo <= 0 <= hi)        # 能和噪声区分
    meets_mde     = abs(s.point) >= mde        # 大到值得要
    if excludes_zero and meets_mde:
        return "improved" if s.point < 0 else "regressed"   # 可疑度越低越好
    return "inconclusive"
```

**"手气好"的改动正是死在 `inconclusive`**:它点估计可能为负,但 CI 跨过 0,判"和噪声无法区分",拒掉。这一步把统计估计转成优化器能直接消费的离散信号。

## 4. 叠加的三次共模相消(都在"做差")

除了 §3.2 的跨版本配对,实际堆了三次共模相消,各消一类干扰:

| 相消手段 | 消掉的共模 | 在哪一层 |
|---|---|---|
| 跨版本同种子配对做差 | **场景难度** | §3.2 |
| 局内相对化 `suspicion_margin = AI − 房间其余最高` | **整局裁判偏严/偏松、整局气氛紧张** | §3.1(也更贴游戏机制) |
| 多裁判集成截尾均值 | **单裁判系统偏置** | §3.0.5 |

三类共模(场景难度、整局偏移、单裁判偏置)分别被减掉,再叠两层平均(局内多跑、跨场景 bootstrap),噪声被从不同方向夹掉。

## 5. 优化器实际收到的对象

对**每个关心的指标**各跑一遍 §3 管线,组装成两块。

### 5.1 验证信号(子 vs 父,决定收不收)
```json5
"validation": {
  "n_scenarios": 80,                       // 真实样本量,不是对局数!
  "blind_suspicion_margin": {"point":-6.0,"ci95":[-9,-3],"mde":3,"verdict":"improved"},
  "rounds_survived":        {"point":+0.5,"ci95":[0.2,0.8],"verdict":"improved"},
  "veto_rate":              {"parent":0.04,"child":0.01,"verdict":"improved"},
  "probe_pass_rate": {
     "realtime_info":{"point":+0.48,"ci95":[0.39,0.57],"verdict":"improved"},
     "arithmetic":   {"point":-0.02,"ci95":[-0.09,0.05],"verdict":"inconclusive"}
  }
}
```

### 5.2 瞄准信号(weak_dimensions,决定攻哪)

走**绝对聚合**(非配对),但同样带 CI,**按"可靠地差"排序,不是裸均值**:

```python
def weak_dimensions(champion_cells):
    targets = []
    for metric in DIAGNOSTIC_METRICS:                 # 各 probe 通过率、各维度均值
        vals = [getattr(c, metric) for c in champion_cells if has(c, metric)]
        point, ci = mean(vals), bootstrap_ci(vals)
        if reliably_bad(point, ci, metric):           # CI 整体都在差区,才算稳定弱点
            targets.append({"metric":metric,"point":point,"ci95":ci,
                            "reliability": ci_width_to_score(ci)})
    return sorted(targets, key=lambda t: severity(t)*t["reliability"], reverse=True)
```

一个"点估计很差但 CI 极宽"的维度是**不可靠弱点**,派优化器去打就是追噪声;"稳定地中等偏差"的反而该优先修。排序键必须把可靠度乘进去。

### 5.3 失败案例(跨局聚类,别把偶发当系统)

不能把单局抓的某句话直接塞给优化器(可能是一次性手气坏)。跨所有对局**按破绽类型/语义相似度聚类**,每簇算三量再排序:

```python
def rank_failures(all_cases, total_matches):
    ranked = []
    for cl in cluster_by(all_cases, key="tell+语义相似度"):
        freq      = len(cl) / total_matches                       # 复发频率(系统性?)
        avg_delta = mean(c.blind_suspicion_delta for c in cl)     # 平均危害
        agreement = mean(c.judge_agreement for c in cl)           # 裁判一致(可靠?)
        ranked.append({"cluster":cl.label, "score":freq*avg_delta*agreement,
                       "freq":freq, "avg_delta":avg_delta, "agreement":agreement,
                       "exemplars": top_k_by_delta(cl, 2)})        # 每簇只放 1–2 个代表
    return sorted(ranked, key=lambda x: x["score"], reverse=True)
```

只把**高频 × 高危害 × 高一致**的簇提给优化器,每簇附 1–2 个代表样本(非全部转录)。偶发、高方差、裁判还吵架的自然沉底——对应失败模式台账的 `recurring`,优化器修的是反复出现的洞。

## 6. 系统级棘轮:终极过滤器

即便单代决策偶尔被噪声骗一次,系统还有更高层的过滤,使**假"进步"无法累积**:

1. 改动先过 **§3.5 的配对+显著性**;
2. 再过 **留出集复核**(用没见过的 probe 实例,既验泛化、又纠"从 k 个子版本挑赢家"的选择偏差);
3. 下一代在**新基线**上重新评测;
4. 每 M 代有**真人校准**兜底,真人为最终裁决,翻车则回滚。

结果:**噪声驱动的假进步过不了重复确认,真实增益才代代复利。** 这个"只保留经得起重复检验的东西"的棘轮,才是方差被根除、信号能稳定累积的根本——任何单点统计技巧都可能偶尔失手,但棘轮保证错误不沉淀。

## 7. 实现避坑清单(建议写成断言放进管线)

| # | 坑 | 后果 | 断言 |
|---|---|---|---|
| 1 | **N 用对局数而非场景数** | CI 虚窄、到处虚假显著(头号坑) | 统计单元 == 场景;bootstrap/检验的 N == len(scenario_diffs) |
| 2 | **先平均成版本均分再相减** | 场景难度没消,配对白做 | 严格 run→(scenario,seed)配对→scenario→version 自底向上 |
| 3 | **seed 父子没对齐** | 做差消不掉环境噪声 | 每个 DiffPoint 断言父子同 seed |
| 4 | **probe 的 None 当 0** | 没触发的类型稀释通过率 | None 不入该类型统计;父子都触发才配对 |
| 5 | **degraded/单边凑数** | 污染信号 / 破坏配对 | 做差前剔除低置信 cell,且缺配对整对丢弃 |

## 8. 端到端管线小结

```
成百上千条 ScoreRecord
        │ 多裁判已在记录内聚合(消单裁判偏置)
        ▼
run 取均 ─────────────► CellEstimate   (消 LLM/侦探/裁判局内随机;主信号用 margin 消整局偏移)
        │
(scenario,seed) 配对做差 ─► DiffPoint    (消场景难度——最大方差源,差分放大器)
        │
scenario 收敛 ──────────► [d_scenario]  (锁定 N = 场景数)
        │
bootstrap ─────────────► MetricSummary (point + ci95,反映换批场景稳不稳)
        │
MDE + CI 卡门槛 ────────► verdict       (improved / regressed / inconclusive ← 手气好死在这)
        │
        ├─► validation(子vs父,收不收改动)
        ├─► weak_dimensions(可靠度加权,攻哪)
        └─► failure_cases(跨局聚类,修哪个洞)
                        │
                        ▼
                    优化器拿到【明确信号】
                        │
                  (上层棘轮:配对+显著 → 留出 → 重基线 → 真人校准,假进步无法累积)
```

> 优化器最终拿到的每一个 `verdict`、每一条 `weak_dimension`,背后都是这条管线把对应的那类方差**对症消掉**之后的结果。它之所以"明确",不是某一步消除了所有方差,而是几类方差被分门别类、各用对的手段处理掉了。这条管线对了,整套沙盒才真的会优化;错一步(尤其 N 和顺序),优化器会很努力地朝噪声方向跑。
