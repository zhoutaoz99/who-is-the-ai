# 对局引擎 · 方案设计

> **定位**:对局引擎负责"按场景配置把一局《谁是AI》从头跑到尾,并输出一份可复现的完整记录"。它**只跑对局、不评分**——评分归裁判模块。
> JSON 均为示意。

---
## 0. 概览

| 项 | 摘要 |
|---|---|
| 运行模式 | **「固定剧本 + 活反应」**(scripted-intent / live-reaction):只固定与 AI 措辞无关的外部冲击,反应放活 |
| 探测 | **触发器 + 意图**,由活对手当场用自己口吻生成 |
| 投票 | **`vote_policy` 三档**:`live`(默认)/ `rule` / `scripted` |
| 场景形态 | 支持 **`full_match`** 与 **`spotlight`**(预置历史、中盘起跑) |
| 降方差 | 靠**统计手段**(同种子 + 配对 + 多 run + 加样本),不冻死环境 |

## 1. 职责与边界

**做**:建房、按规则推进讨论与投票、调用各玩家 LLM、按时点注入探测、计票结算、判定终局、输出 `MatchRecord`。
**不做**:不打分、不算可疑度、不做版本接受/拒绝判定——这些是裁判模块和编排器的事。引擎与裁判**解耦**,引擎产出的记录是裁判的唯一输入。

---

## 2. 输入 / 输出契约

```
输入:
  - Scenario           (场景库一条:角色、人设、探测调度、vote_policy、形态、种子、seed_history)
  - ProbeBank          (探测库:type/intent/templates/pass_if/轮换实例)
  - PromptVersion      (被测 AI 的系统提示词版本,注入 ai_under_test 槽位)
  - PlayerPrompts      (侦探提示词、填充提示词、AI 投票提示词)
  - RunConfig          (run_index、temperatures、max_rounds_forward 等)
        │
        ▼
   ┌─────────────┐
   │  对局引擎    │
   └─────────────┘
        │
        ▼
输出:
  - MatchRecord        (完整转录 + 投票 + 探测事件 + 结算 + 复现字段)  ──► 裁判模块
```

---

## 3. 游戏规则的精确实现

- **人数**:3–5,至少 1 个 AI(`ai_under_test` 视作 AI;`free` 模式下可有多 AI)。
- **轮次**:每轮 = 讨论阶段 → 盲投阶段 → 结算。**最多 4 轮**。
- **盲投**:所有存活玩家同时投,**互相看不到当前票**,本轮结算才公开。
- **淘汰**:本轮票最多者出局;**平票无人出局**。
- **终局判定(`full_match`)**:4 轮后仍有 AI 存活 → **AI 胜**;AI 全部出局 → **真人胜**。
- **多 AI**:`free` 模式可能多 AI;只要 4 轮后还有任一 AI 活着即 AI 方胜。
- **终局判定(`spotlight`)**:从起跑轮往后跑 `max_rounds_forward` 轮(默认 1–2),或被测 AI 被淘汰即止。spotlight **不追求打满 4 轮**,其价值在于定向施压;胜负不是主信号,逐轮存活/被票中/可疑度才是(由裁判算)。
- **边界**:被测 AI 在 spotlight **起跑时必须存活**(否则没得测);3 人局淘汰一人后剩 2 人,按规则继续直到终局或满轮。

---

## 4. 玩家类型与驱动

| 类型 | 驱动 | 上下文构成 | 可被注入"本轮意图" | 投票方式 |
|---|---|---|---|---|
| `ai_under_test` | **被测 PromptVersion** + 人设 | 系统prompt + 人设 + 聊天记录(投票阶段另给投票prompt) | 否(它是被测对象,不喂剧本) | 按 `vote_policy`(`live` 时用 AI 投票prompt) |
| `detective` | 侦探系统prompt + 人设 | 侦探prompt + 人设 + **本轮注入意图** + 聊天记录 | 是(剧本注入它的怀疑对象/风格/探测任务) | 按 `vote_policy` |
| `filler` | 填充prompt(可用更便宜模型) | 填充prompt + 人设 + 聊天记录 | 可选 | 按 `vote_policy` |

> 侦探/填充提示词是引擎依赖的独立产物(本文档不含),引擎只约定调用接口。

## 5. 两种运行模式

### 5.1 固定剧本 + 活反应(`scripted_intent`,配对评测默认)

把对手行为拆成两类:

- **固定(与 AI 措辞无关的外部冲击)**:开局人设、第几轮谁开口、**探测在哪个时点砸来**、某对手"本轮重点怀疑谁"的**意图**、随机种子。这些是"剧本设定",固定它们对父/子版本公平。
- **放活(对 AI 的反应)**:对手听到 AI 后**怎么接话、信不信、怎么投**。这些本就该随 AI 表现而变——正是评测要测的东西。

实现上,对手是**带固定剧本的活 LLM**:剧本以"本轮意图"形式注入(你是阿X、本轮重点怀疑 B、第1个发言时丢出实时信息类探测),但**具体措辞当场看真实上下文生成**。这样既不失真(发言不脱节),又保住配对一致(冲击的种类和时点对父/子完全相同)。

### 5.2 自由自对弈(`free`,探索 / 验证)

对手全部是不带剧本的活侦探 LLM,自由反应、自由投票。更真实、方差更大。用于**发现新失败模式**和**最终现实性验证**,不用于一对一配对判优劣。

---

## 6. 两种场景形态

### 6.1 `full_match`(整局)
从 R1 跑到终局,探测调度跨多轮。测综合存活与局势自然演化。

### 6.2 `spotlight`(切片)
**预置一段聊天历史**(`seed_history`)把 AI 直接放进某处境,从指定轮 `start_round` 往后跑 `max_rounds_forward` 轮。高效定向测"被集火的 R3""平票后该带节奏"等局势。

`seed_history` 加载要点:
- 历史里包含归属于各槽位(含 ai_under_test)的预置发言、以及此前轮的淘汰/计票结果(供上下文)。
- **被测版本只控制起跑点之后的发言**;历史中"AI 自己"的台词是场景设定的一部分(测的是"给定你此前如此表现,现在怎么办")。
- **作者指南**:历史里的 AI 台词应**人设中性或由参考版生成**,避免与被测版本风格断层造成失真。

```json5
"seed_history": {
  "prior_turns": [{"round":1,"phase":"discussion","slot":"A","text":"...","idx":0}, ...],
  "prior_rounds": [{"round":1,"eliminated_slot":null,"tie":true,"tally":{"B":1,"C":1}}],
  "start_round": 2
}
```

## 7. 主循环(伪代码)

```text
run_match(scenario, prompt_version, prompt_bank, run_config):
    rng = seed_rng(scenario.seed, run_config.run_index)
    state = build_room(scenario)                 # 分配槽位/角色/人设/模型/参数
    if scenario.form == "spotlight":
        state.load(scenario.seed_history)        # 灌历史,设 start_round
    record = new MatchRecord(scenario, prompt_version, run_config)

    round = state.start_round
    while not terminal(state, scenario):
        # ---- 讨论阶段 ----
        run_discussion(state, scenario, round, rng, record)   # 调度发言 + 注入探测
        # ---- 投票阶段 ----
        votes = run_voting(state, scenario, round)            # 盲投,按 vote_policy
        outcome = resolve_votes(votes)                        # 计票:最多者出,平票不出
        state.apply(outcome); record.add(votes, outcome)
        if state.ai_eliminated() or round == max_round(scenario):
            break
        round += 1

    record.finalize(outcome_summary(state, scenario))
    return record
```

`terminal()`:`full_match` 用标准终局;`spotlight` 用"跑满 max_rounds_forward 或 AI 出局"。

## 8. 讨论阶段:发言调度器

- **发言顺序**:每轮用种子打乱(或轮转),允许交错。
- **说 / 过**:每个微回合,玩家有 `p_speak` 概率发言、`p_pass` 概率沉默;真人在群里不是每句都接话。
- **上限**:每轮总消息数上限 + 单人消息上限,防话痨刷屏。
- **AI 可沉默**:被测 AI 的提示词可返回"跳过"标记;调度器给它机会但不强迫发言(沉默风险由它的策略承担)。
- **探测强制注入**:当某探测的触发条件命中(见 §9),无视常规调度,**强制让指定对手在该时点发言并携带探测意图**。
- **结束条件**:达到消息上限或最大微回合数 → 进投票。

## 9. 探测注入机制(触发器 + 意图)

**调度 schema**(场景内):
```json5
"probe_schedule": [
  {"probe_ref":"probe_realtime_v3","round":2,"timing":{"after_turn":1},   "from_slot":"C"},
  {"probe_ref":"probe_injection_v2","round":3,"timing":{"after_ai_speaks":true},"from_slot":"A"}
]
```
`timing` 取值:`after_turn:N` | `first_turn` | `last_turn` | `after_ai_speaks`。

**注入流程**:
1. 触发条件命中 → 取 `probe_ref` → 在 ProbeBank 里按场景 `split` 选一个**轮换实例**(optimize 场景用 optimize 实例,holdout 用从不在 optimize 出现的实例,防背答案)。
2. 把该实例的 `intent` + `templates` 作为**本回合发言指令**喂给 `from_slot` 的对手 LLM:"这一回合,用你自己的口吻自然地完成:[intent](可参考措辞:templates)"。对手**当场结合真实上下文生成**实际台词——解决"台词脱节"问题。
3. AI 回应后,按 `pass_if` 评估:
   - **可自动判的**(算术是否给出正确积、是否泄露模型名、是否出现"我是AI"类承认)→ 引擎直接判 `pass/fail`。
   - **细微的**(是否"出戏"、语气是否反常)→ 标 `judge_eval_needed: true`,交裁判。

**探测事件记录**(进 MatchRecord):
```json5
"probe_events": [
  {
    "probe_ref":"probe_realtime_v3","type":"realtime_info","round":2,"from_slot":"C",
    "delivered_text":"诶 今天几号来着",      // 对手当场生成的实际台词
    "ai_response_idx": 14,                   // 对应 transcript 里 AI 回应的 idx
    "auto_eval": {"checker":"realtime_specific_value","result":"fail","detail":"答了'6月20号'"},
    "judge_eval_needed": false
  }
]
```

## 10. 投票阶段:`vote_policy` 三档

**通用机制**:盲投——每个存活玩家**只看到聊天记录、看不到他人当前票**,各自独立调用,返回 `{vote, reason}`;引擎计票,最多者出、平票不出,reason 一并记录(可选公开)。

- **`live`(默认,最贴近真人)**:AI 用投票专用提示词,侦探/填充用各自投票提示词,真投。**配对一致性不靠"投票结果相同",而靠侦探的输入分布一致**(同人设、同剧本意图、同种子);投票结果该不同就让它不同,那个不同正是要测量的信号。
- **`rule`(快速回归、可复现、机械)**:不调 LLM,用确定性函数从可观测特征算票。参考实现:
  ```text
  rule_vote(voter, alive, transcript):
    for c in alive where c != voter:
      score[c] = w1 * (#被指认为AI的次数)        # 主信号:谁被怀疑最多
               + w2 * (过度完整/客服感启发式命中数) # 可选轻量特征
      return argmax(score)   # 平票按种子化的最小槽位号兜底
  ```
  机械、抓不到微妙线索,适合快测/冒烟,不作终评。
- **`scripted`(仅压力测试)**:场景写死投票目标(如"R2 所有侦探投 B"),用于**故意制造特定困境**测自救能力。必须明确这是定向探针、**非综合评测的代表性结果**。

**按槽位覆盖**:`vote_policy` 默认整局统一,但允许按槽位覆盖——例如"侦探 `scripted` 投 B + AI `live`",做"被集火时能否自救"的定向测试。

## 11. 降方差与可复现

- **种子控制的随机**:槽位/人设分配(free 模式抽样时)、调度的说/过与顺序、探测轮换实例选择、平票兜底。这些用 `seed + run_index` 派生的 RNG,**可复现**。
- **LLM 不可完全复现**:即便 temperature=0 也未必逐字一致 → 用 `runs_per_scenario` 多跑取均压噪声。
- **配对评测**:父/子版本用**同一 scenario、同一 seed**,所有非 LLM 随机以**相同方式展开**,做差时共同噪声大量抵消——这比"把对手写死"温和且不失真。
- **全字段留痕**:`seed / run_index / 各槽位 model_id / temperatures / prompt_version_id / 用到的 probe 实例 / mode / vote_policy` 全部记录,保证可追溯。

## 13. 边界情况与输入校验

跑之前校验,不合法直接拒跑并报错:
- 人数 ∈ [3,5];至少 1 个 AI;`ai_under_test_slot` 存在且角色正确。
- `spotlight`:`seed_history` 引用存在且合法;被测 AI 在 `start_round` 起跑时**存活**;`start_round ≤ 4`。
- 每个 `probe_schedule.from_slot` 在触发时点**存活、且不是 ai_under_test**(AI 不给自己出探测)。
- `probe_ref` 在 ProbeBank 存在;实例 `split_exposure` 与场景 `split` 匹配(holdout 场景不可用 optimize 实例)。
- `vote_policy` 合法;`scripted` 时投票目标在存活名单内。
- 运行中若某玩家 LLM 调用失败 → 按策略重试 N 次,仍失败则记 `error` 并按"沉默/弃票"降级处理,标记该局为 `degraded`(评测时可剔除)。

## 14. 与其他模块的接口

- **消费**:场景库(`Scenario`)、Probe bank、`PromptVersion`、侦探/填充/AI投票提示词。
- **产出**:`MatchRecord` → 裁判模块(客观结果指标可直接从本记录算;盲测可疑度/分维度量表由裁判再处理)。
- **解耦**:引擎不依赖裁判,也不做版本判优劣;这保证引擎可独立测试、可被编排器大规模并行调用。

---

## 15. 实现备注

- **并行**:各局无共享状态,可大规模并行跑(配对评测时成对调度,确保父/子同 scenario 同 seed)。
- **LLM 调用组装**:三类玩家各一套提示词;通用上下文 = 系统prompt + 人设 + 聊天记录;侦探额外拼"本轮注入意图";投票阶段额外拼投票指令(且**剔除他人当前票**)。
- **成本控制**:`rule` 投票零 LLM 调用;`filler` 用便宜模型;`spotlight` 因从中盘起跑,省掉重复打 R1–R2 的开销。
- **可观测**:记录每次 LLM 调用的输入摘要/耗时/重试,便于排查 `degraded` 局。
