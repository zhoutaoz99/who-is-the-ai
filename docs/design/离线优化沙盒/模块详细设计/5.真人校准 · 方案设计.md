# 真人校准 · 方案设计(防 Goodhart 的第三道闸)

> **定位**:本文是真人对局喂回沙盒的**第二条支路**——校准"**代理可信度**":验证沙盒里的代理指标(盲测可疑度、存活率)能不能预测真人对局的真实结果,漂了就修。
> 它**不**做的事:不优化提示词(那是优化器)、不回灌场景(那是《场景库·回灌流程》的另一条支路)、不跑对局/不评分。
> 与《裁判评分模块》§11、《编排器模块》§9、两份 Schema 契约对接。本文以 **A(已上线、有真实对局日志)为主线**,**B(组织测试局/小样本)、C(暂无真人数据)作为适配**;并默认**后台已知哪个席位是 AI**(校准盲测可疑度需要真值标签)。

---

## 1. 两条支路:校准 vs 回灌(先分清)

真人对局产出两种不同的回流,**喂给不同地方、解决不同问题**:

```
                       ┌── 失败回灌(《场景库·回灌流程》)──► 场景库
真人对局日志 ──────────┤   喂:真实失败片段 → 长出新场景;问题:沙盒"考什么"
                       └── 真人校准(本文) ─────────────────► 裁判 / 侦探池(代理)
                           喂:真实结果 → 验证&修正代理可信度;问题:沙盒"评得准不准"
```

- **回灌**改变沙盒"测什么"(覆盖度);**校准**改变沙盒"评得准不准"(代理保真度)。
- 二者用同一批真人日志,但抽取的信息和下游完全不同。本文只管校准。

**为什么需要它**:沙盒里"AI 玩家、侦探、裁判"全是 LLM,优化器很可能进化出"骗得过 LLM 裁判、骗不过真人"的版本(系统级 Goodhart)。**盲测可疑度是代理,不是真相**;真人对局是唯一真值。校准就是定期体检"代理 ↔ 真值"还对不对得上,对不上就把代理拉回来。

---

## 2. 采集:真人对局要存什么(主线 A)

每局真人对战完整落盘,字段尽量复用《对局记录·Schema契约》的 MatchRecord(它本就允许承载真人局):

- `prompt_version_id`:这局 AI 用的哪版 champion(**校准按版本聚合,这是 join key**)。
- `ai_under_test_slot` + 后台真值标签:**哪个席位是 AI**(后台已知,最干净;不依赖玩家事后标注)。
- `transcript` / `votes` / `eliminations` / `outcome`:真人怎么聊、怎么投、AI 有没有被票出、第几轮出。
- 真人侧元信息:玩家数、轮次、(可选)玩家匿名 ID 用于去重刷量。
- **隐私去标识**:真人 `text` 须按回灌流程同款脱敏(抹 PII/用户名);校准本身大多只用结果字段(投票/淘汰),但若要把真人聊天喂裁判做少样本(§5),更要先脱敏。

**采样策略(关键,别采歪)**:
- **要代表性,不要只采被抓的局**。回灌偏爱"AI 被抓"的局(那才有失败素材),但校准要的是**整体分布**——既包含 AI 被识破的,也包含 AI 蒙混过关的,否则相关性会被截断、估偏。
- **覆盖多版本**:校准要把"沙盒分高低"和"真人结果好坏"在**一批不同 champion 版本**上比,所以日志要带版本、且最好横跨几代,差异才拉得开。
- **足量**:真人局高方差(见 §4),单版本要攒到足够样本相关性才稳。

---

## 3. 对齐:把沙盒分与真人结果配成可比数据对(核心难点)

校准的核心是构造一组 **(代理值, 真人值)** 配对,**配对单元是 `prompt_version_id`**(同一版 champion,既有沙盒预测、又有真人实测)。

**代理值(沙盒侧,每个版本一个)**:该版本在沙盒评测集上的聚合**盲测可疑度**(相对量 `suspicion_margin`),以及备用的沙盒存活率——直接取自该版本的 `validated_metrics`(注意带 `eval_set_version`,跨评测集版本不可混)。

**真人值(真人侧,每个版本一个)**,从真人日志按版本聚合:
- 主:**真人识破率 / 被票出率**——该版本 AI 在真人局里被票出局的比例(或逐轮被票中率)。
- 辅:**真人怀疑强度**——若后台已知 AI 席位,可算"真人投票里指向 AI 的票占比""AI 收到的票数 / 应得均摊票",作为连续的真人侧可疑度。

**配出来的样子**(每行一个版本):

| version | 沙盒 suspicion_margin(代理) | 真人被票出率(真值) | 真人指向AI票占比(真值) |
|---|---|---|---|
| v5 | +9 | 0.62 | 0.41 |
| v6 | +6 | 0.48 | 0.33 |
| v6.1 | +2 | 0.31 | 0.22 |

> **对齐纪律**:代理与真人必须是**同一 `prompt_version_id`**;真人侧若版本上线有先后,注意控制"对手真人也在变强"的混杂(见 §4)。

## 4. 相关性体检:用什么统计、阈值、样本量

**用 Spearman 秩相关(`correlation_proxy_vs_real`),不是 Pearson。** 我们要的是"**沙盒说 A 比 B 强,真人是不是也觉得 A 比 B 强**"——即**排序一致**,而非绝对值线性吻合。代理和真值的量纲、刻度都不同(可疑度分 vs 被票率),秩相关只看序、更稳健、对离群不敏感。

- **判定**:相关系数 ≥ 阈值(起步建议 **ρ ≥ 0.6**,按你的基线标定)→ 代理可信,放心用沙盒快速迭代;**显著低于阈值或持续下滑** → 代理漂了,触发 §5 修复。
- **样本量**:相关性的统计单元是**版本数**,不是真人局数。要相关性可信,至少需要**一批有梯度的版本**(比如 8–15 个 champion,沙盒分有高有低);每个版本下的真人局要足够多,把单版本的真人值估稳(真人局二值结局高方差,单版本几十局起)。
- **控混杂**:真人对手会随时间变强/变熟练(meta 漂移),使"晚上线的版本被票率更高"并非版本更差。缓解:**同期对照**(同一时间窗同时上线两版做 A/B,真人随机分配),或在分析里把"上线时间/对手段位"作为协变量回归掉。
- **别只看一个数**:除了总相关性,看**残差**——哪些版本"沙盒说强、真人却识破"(Goodhart 的具体案例),这些版本是诊断漂移成因的最佳样本(§5)。

### 4.1 `correlation_proxy_vs_real` 的具体计算细则

把上面的口径落到可实现级。设有 N 个版本,每个版本一对 `(代理值, 真人值)`。

**主指标取哪两列**:代理值用沙盒 `suspicion_margin`;真人值**以"被票出率"为主指标**(直接对应游戏目标——AI 有没有被淘汰)。"指向 AI 票占比"作辅助交叉验证(连续、更早期、更灵敏),各算一个 ρ 比对,但 `correlation_proxy_vs_real` **主报被票出率那个**。

**方向约定(关键,先想清正负)**:两列都是"越大越像 AI"——margin 高=沙盒觉得像 AI,被票出率高=真人识破了。所以**健康状态是强正相关 ρ → +1**;出现负相关或接近 0,即代理与真人脱钩=漂了。文档里"ρ≥0.6"指的就是这个正方向。

**算法(标准 Spearman)**:
1. 把两列各自转成**秩**(rank);
2. 对两组秩算 **Pearson 相关**——这就是 Spearman 定义。
3. **并列处理**:多个版本沙盒分相同(并列)时,**不能用**简化式 `ρ=1−6Σd²/(N(N²−1))`(它假设无并列),必须用"对秩做 Pearson"的**通用式**(并列取平均秩),否则结果偏。实现直接用 `scipy.stats.spearmanr(proxy, human)`,它自带并列处理与 p 值:
```python
from scipy.stats import spearmanr
rho, p = spearmanr(proxy_margins, human_eliminated_rates)   # rho 即 correlation_proxy_vs_real
```

**小样本必须给置信区间,不能只看点估计(头号坑)**:版本数 N 常只有 8–15,此时 ρ 点估计方差很大,单看一个数判"漂没漂"会被少数版本的抖动骗。做法:**对版本做 bootstrap**——有放回重采 N 个版本、各重算 ρ、取 2.5/97.5 分位得 ρ 的 95% CI:
```python
import numpy as np
def rho_ci(proxy, human, n_boot=10000):
    idx = np.arange(len(proxy))
    boots = [spearmanr(proxy[s], human[s]).correlation
             for s in (np.random.choice(idx, len(idx), replace=True) for _ in range(n_boot))]
    return float(np.nanmean(boots)), tuple(np.nanpercentile(boots, [2.5, 97.5]))
```
**判定改用 CI**:CI 下界 ≥ 阈值才算"代理可信";CI 跨过阈值或含 0 → 当"低置信",不据此冻结/解冻,先补真人样本。

**可选 · 传导真人值噪声(更严谨)**:每个版本的真人被票出率本身是几十局估出的、带噪声。严格做法是**双层 bootstrap**:外层重采版本、内层在版本内重采真人局重估其被票出率,再算 ρ。起步可先忽略内层、只 bootstrap 版本;真人样本特别少时再加内层。

**退化情形(标低置信,不下结论)**:版本数 < 最小值(如 <6)、沙盒分几乎无梯度(所有版本 margin 挤在一起,秩无意义)、或某些版本真人局太少——任一出现就把本次校准标 `verdict: low_confidence`,**只攒数据、不触发冻结或解冻**,等数据够了再判。

## 5. 代理漂移的诊断与修复

相关性掉了,先**定位是哪一环漂了**,再对症修。两个主要嫌疑:

**嫌疑一 · 裁判漂了**(盲测可疑度和真人怀疑不一致)。
- 症状:沙盒盲测给某版本打低可疑度,真人却频繁识破;或裁判在残差大的版本上,其盲测排序和真人怀疑排序对不上。
- 修(对应裁判 §11):
  - 把**真人聊天 / 真人怀疑标注**作为**少样本**喂给盲测裁判,把它的判断往真人那边拉;
  - **换 / 加裁判模型**(提多样性,别让代理依赖单一模型的偏好);
  - **重写量表锚点**,纠正系统性高估/低估;
  - 重申"像玩家不像法医"——裁判过度取证会在真人不在意处乱扣分,导致代理偏离。

**嫌疑二 · 侦探池太弱**(沙盒里 AI 轻松过关,真人却能识破)。
- 症状:沙盒存活率普遍偏高、可疑度普遍偏低,但真人识破率不低——说明沙盒的"对手"没真人厉害,AI 是被放水了。
- 修:
  - **侦探池增强 / 增多样性**:加更强模型的 hard 侦探、混入更多风格,别让 AI 只学会"专破某一种侦探";
  - 把**真人识破 AI 的真实手法**(来自残差版本的日志)灌进侦探,让沙盒对手更接近真人;
  - 注意这与回灌有交集:新手法既补侦探(校准),也可固化成新探测/场景(回灌)。

**修完要确认修好了**:用一批**没参与本次诊断**的真人样本(留出的真人对局)重算相关性,确认回升到阈值以上,才解除冻结(§6)。否则是"对着这批残差过拟合裁判",换批真人又漂。

## 6. 与编排器的联动(节奏 / 冻结 / 回滚)

校准是周期性触发的闸,由编排器调度(对应编排器 §9):

- **节奏**:每 `M` 代跑一次校准批次(M 起步 5–10)。
- **冻结晋升**:`correlation_proxy_vs_real` 低于阈值 → **暂停晋升新 champion**,转入 §5 修复,直到留出真人样本确认相关性回升。冻结期间沙盒可继续跑、攒数据,但不让任何"可能是 Goodhart 产物"的版本上线。
- **回滚**:若某个**已晋升 champion** 在真人实测里翻车(真人识破率显著高于它在沙盒的预测,或高于上一稳定版)→ **以真人为准,回滚** champion 到上一稳定版,并把这次"沙盒说好、真人说差"的案例作为最有价值的漂移样本去修裁判/侦探。
- **真人为最终裁决**:这条贯穿——沙盒只是快预筛器,任何沙盒结论与真人结论冲突,真人胜。

## 8. 适配:B(测试局/小样本)与 C(暂无真人数据)

**B · 还没上线,只能组织真人测试局。**
- 样本少、贵 → **降频、攒批**:不按固定 M 代,而是攒够一批版本再做一次校准;每版真人局数难上去时,相关性阈值放宽、置信区间放大,结论当"弱信号"用。
- **设计上提效**:测试局优先安排在**沙盒分差异大的版本**之间(梯度拉满,相关性最容易看出来),别把宝贵真人局浪费在两个几乎一样的版本上。
- 后台已知 AI 席位这点不变,真值标签照样干净。
- 这阶段校准频率低,所以**更要克制晋升**:没有足够真人验证时,别让沙盒分独自决定上线,多保留人工复核。

**C · 暂时没有任何真人数据(冷启动)。**
- 真人校准只能**先搁置**;此时整套沙盒处于"未校准"状态,**默认它可能在 Goodhart,不能把沙盒分当真**。
- **过渡替代**:可用一个**强外部模型(与裁判/侦探不同族、更强)当"伪真人"**,跑一批对局当临时真值代理——但必须明确这只是占位:它仍是 LLM,**不能真正防 Goodhart**,一旦有真人数据就立刻切换。
- 冷启动期的纪律:**多迭代、慎上线**;把"拿到第一批真人数据并完成首次校准"当成里程碑,在此之前所有沙盒结论都标注"未经真人校准"。

## 9. 与其他模块接口

- **消费**:真人对局日志(MatchRecord 形态)、各版本 `validated_metrics`(沙盒侧代理值)。
- **产出**:`CalibrationRun`(相关性、判定、诊断、修复动作)→ 编排器(冻结/回滚决策);诊断出的漂移成因 → 裁判修复(§5 嫌疑一)/ 侦探池修复(§5 嫌疑二);残差版本的真人日志 → 也可分流给场景库回灌。
- **解耦**:校准只做"代理 ↔ 真值"的体检与触发,不实现裁判/侦探/优化器的内部逻辑;它是编排器调度下的一个周期性闸。

---

> 一句话:**失败回灌让沙盒"考得更全",真人校准让沙盒"评得更准"。** 没有校准,所有沙盒分都可能是自欺;有了校准,沙盒才有资格当真人对局的快速预筛器。它是把"真人是最终裁决"这条信念,真正落到流程里的那道闸。
