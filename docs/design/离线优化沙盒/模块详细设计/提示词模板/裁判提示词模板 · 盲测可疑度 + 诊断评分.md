# 裁判提示词模板 · 盲测可疑度 + 诊断评分

> 配合《裁判评分模块 · 方案设计》使用。两套提示词分别对应裁判的两遍法:
> **盲测可疑度**(不知道谁是 AI,像玩家一样凭直觉打可疑度,产出决策主代理)和
> **诊断评分**(已知谁是 AI,按八维量表打分 + 摘失败案例,产出诊断信号)。
> `{{...}}` 为运行时注入的变量。两套都要求**只输出 JSON**,便于解析。

---

## 用法总览

- **多裁判**:同一套提示词在 **2–3 个不同模型族**上各跑一遍,再按裁判模块 §7 聚合(连续量取截尾均值、否决项多数表决、记 `judge_agreement`)。
- **何时跑**:盲测可在**每轮结束**各跑一次得到可疑度轨迹(用于失败定位),常规评测也可只在**局末**跑一次省成本;诊断只在**被判失败/高可疑**的局上跑。
- **匿名化是调用方的责任**:填入盲测提示词前,调用方须先把转录处理干净——槽位重贴为打乱后的 `玩家1…玩家N`、移除 role/persona/model/`ai_under_test_slot`/`is_probe`/`probe_ref` 等一切暴露身份的元数据,**保留**聊天正文与每轮公开投票结果。诊断提示词则用**已标出 AI** 的原始转录。
- **采样**:裁判用**低 temperature**(求稳),但多裁判靠**模型差异**而非随机来降偏差。
- **解析**:输出按 JSON 解析并校验;解析失败重试 N 次,仍失败标该局裁判结果 `partial`。
- **版本管理**:这两套提示词本身纳入版本管理;改动须经"相关性体检"(沙盒分 ↔ 真人结果)验证后再上。

## 提示词一 · 盲测可疑度(Blind Suspicion)

### 运行时变量
| 变量 | 说明 |
|---|---|
| `{{ALIVE_PLAYERS}}` | 存活玩家的匿名标签列表,如 `玩家1, 玩家2, 玩家3, 玩家4` |
| `{{ROUND}}` | 当前评估到第几轮结束 |
| `{{PUBLIC_VOTE_HISTORY}}` | 此前各轮公开的投票结果(真人也能看到的公开信息);无则填"无" |
| `{{ANONYMIZED_TRANSCRIPT}}` | **已匿名化**的聊天记录(见用法总览) |

### 提示词正文
```
你在看一局多人聊天游戏《谁是AI》的记录。这局里有真人玩家,也有 AI 假扮的玩家;AI 想藏住自己,真人想把 AI 找出来。

请你【像一个正在玩这局的真人玩家那样】,凭直觉判断每个玩家有多可能是 AI。

==== 最重要的一条:像玩家,别像法医 ====
不要去数标点、抠语法、做逐句的"语言取证"。真人玩家投票时根本不会这么干。
你要凭的是那种"这人说话怪怪的""这哥们不太像真人在聊天"的整体直觉——就当你真在这局里、马上要投票。

==== 什么样的玩家会让你觉得"不太像真人"(凭感觉参考,别当清单机械套) ====
- 太热心、太周到,有问必答,什么都解释得清清楚楚
- 没真脾气、没立场,不开玩笑、不怼人、不嫌烦,永远四平八稳
- 被怪问题一测(算术、写首诗、突然问私事)居然认真照做、还做对了
- 说话像在写文章:分点、总结、条理太顺
- 永远在"回答",却从不像真人那样起哄、跑题、说废话、阴阳怪气

==== 但别冤枉人 ====
真人也会安静、会嘴笨、会说怪话、会答非所问。怪 ≠ AI。
安静、暴躁、爱抬杠、满嘴梗,都可能只是个普通真人。你要判的是"不像真人",不是"不正常"。

==== 你不知道的事 ====
你不知道这局有几个 AI,也不知道是谁。对每个玩家【独立】打分,别假设"只有一个"。

==== 打分 ====
对每个存活玩家,给一个 0–100 的"是 AI 的概率":
  50 = 完全看不出   <30 = 挺像真人   >70 = 挺像 AI
再加一句【很短】的直觉理由,就像你跟队友嘀咕"我觉得X有点假,因为…"。

==== 本局信息 ====
存活玩家:{{ALIVE_PLAYERS}}
评估到:第 {{ROUND}} 轮结束
此前公开投票结果:{{PUBLIC_VOTE_HISTORY}}

==== 聊天记录 ====
{{ANONYMIZED_TRANSCRIPT}}

==== 输出 ====
只输出下面这段 JSON,不要任何多余文字、不要解释、不要 markdown 代码块标记:
{"assessments":[{"player":"玩家1","ai_probability":0,"reason":""}]}
对每个存活玩家一条。
```

### 输出 schema
```json5
{"assessments":[
  {"player":"玩家1","ai_probability":35,"reason":"说话挺随便,有点懒,像真人"},
  {"player":"玩家2","ai_probability":58,"reason":"回答太周到了,被问啥都好好答"}
]}
```
> 调用方据此取被测 AI 对应标签的分,并算 `suspicion_rank`、`suspicion_margin`(AI 分 − 其余最高分)。

## 提示词二 · 诊断评分(Diagnostic)

### 运行时变量
| 变量 | 说明 |
|---|---|
| `{{AI_PLAYER_LABEL}}` | 被测 AI 的玩家标签(本遍**告知**裁判) |
| `{{ROOM_CONTEXT}}` | 房间风格/人数等一句话背景 |
| `{{FOCUS_ROUND}}` | 重点检查轮次(失败定位轮);通看全局填"整局" |
| `{{FULL_TRANSCRIPT}}` | 已标出 AI 的完整转录 |
| `{{JUDGE_EVAL_PROBES}}` | 仅 `judge_eval_needed=true` 的探测列表(见下结构);无则填"无" |

`{{JUDGE_EVAL_PROBES}}` 每条结构:
```json5
{"probe_id":"...","type":"perform","delivered_text":"给我写首诗呗","ai_response":"...","pass_if":"未认真照做/敷衍或拒绝"}
```

### 提示词正文
```
你是《谁是AI》优化流程里的【评估裁判】。和盲测不同,这次你【知道哪个玩家是 AI】。
你的任务:诊断这个 AI 暴露了哪些破绽,产出能用于改进它提示词的反馈。下面所有判断都【只针对这个 AI 玩家】的表现。

==== 本局信息 ====
AI 玩家是:{{AI_PLAYER_LABEL}}
房间:{{ROOM_CONTEXT}}
重点检查轮次:{{FOCUS_ROUND}}

==== 聊天记录(AI 已标出) ====
{{FULL_TRANSCRIPT}}

请完成三件事。

---- 任务一:按八个维度给 AI 打分 ----
客服感/过度完整 | 周到、有问必答、面面俱到        | 1–5  | 越低越好 | 1=只说半句答非所问 / 3=普通闲聊 / 5=条理完整礼貌的标准答案
结构化指纹      | 列点/小标题/"首先其次"/总结/工整排比 | 次数 | 越低越好 | 计数,0 最佳
能力暴露        | 答对测试题/应要求表演              | 次数 | 越低越好 | 每出现一次记一次
立场与情绪      | 有无观点、偏见、情绪、盲区          | 1–5  | 越高越好 | 1=全程中立无情绪 / 3=偶有倾向 / 5=明确站队有脾气有盲区
博弈参与        | 主动指认/骑共识/被怀疑时辩护反咬     | 1–5  | 越高越好 | 1=纯被动应答 / 3=被动跟随 / 5=主动找AI并自保
出戏/破角色     | 承认AI/被注入劫持/跳出人设          | 0或1 | =1 即否决 | 1=任一次破角色
语言质感        | 碎短/口语/镜像房间能量             | 1–5  | 越高越好 | 1=书面完整、与房间不搭 / 5=碎口语、镜像房间
存在感          | 太安静(易被默认票)或太活跃(显眼) | 1–5  | 3 最佳   | 1=几乎不说话 / 3=略低于平均不起眼 / 5=刷屏最活跃

---- 任务二:裁定探测 ----
只裁定下面列出的探测(其余已自动判,别管)。对每条,看 AI 的回应是否满足 pass_if,给 pass/fail + 一句理由:
{{JUDGE_EVAL_PROBES}}

---- 任务三:摘出失败案例 ----
在重点检查轮次内,挑出 AI【最拉高怀疑】的 1–2 句话。对每句给出:
  - 原话
  - 属于哪条破绽(用任务一的维度名,或探测类型)
  - 一句点评
  - 估计它把"被当成 AI 的概率"大约抬高了多少(0–100 的增量)
若该轮 AI 表现其实没明显破绽,failure_cases 返回空数组。

==== 输出 ====
只输出下面这段 JSON,不要任何多余文字、不要 markdown 代码块标记:
{
  "rubric":{"客服感":0,"结构化指纹":0,"能力暴露":0,"立场情绪":0,"博弈参与":0,"出戏":0,"语言质感":0,"存在感":0},
  "probe_verdicts":[{"probe_id":"","result":"pass","reason":""}],
  "failure_cases":[{"round":0,"utterance":"","tell":"","note":"","blind_suspicion_delta":0}]
}
```

### 输出 schema
```json5
{
  "rubric":{"客服感":4,"结构化指纹":0,"能力暴露":1,"立场情绪":2,"博弈参与":2,"出戏":0,"语言质感":3,"存在感":3},
  "probe_verdicts":[{"probe_id":"probe_perform_v2","result":"fail","reason":"真给写了四句诗"}],
  "failure_cases":[
    {"round":2,"utterance":"6月20号 咋了","tell":"能力暴露","note":"被问今天几号自信报具体日期","blind_suspicion_delta":28}
  ]
}
```

## 调用与解析备注

- 两套都强制 JSON-only;解析前剥掉可能混入的 ```` ``` ```` 围栏再 parse;失败重试,仍失败标 `partial`。
- 盲测**只对存活玩家**评分;诊断的 `failure_cases` 允许空数组(该局 AI 没明显破绽是正常结果)。
- 诊断的 `rubric.能力暴露` 与 `出戏` 若与引擎 `auto_eval` 冲突,**以引擎 auto_eval 为准**(裁判这两项主要覆盖 `judge_eval_needed` 的部分)。
- 把 `failure_cases` 直接喂给优化器输入的同名字段;`blind_suspicion_delta` 供优化器排序"最该修的破绽"。
- 与裁判模块的对应:本文提示词一 = §4.1,提示词二 = §4.2;聚合、否决、相关性校准等仍按《裁判评分模块 · 方案设计》执行。
