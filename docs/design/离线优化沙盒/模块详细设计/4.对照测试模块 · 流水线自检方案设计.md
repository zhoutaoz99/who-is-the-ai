## 1. 背景

优化流水线(对局引擎 → 裁判 → 评分聚合 → 优化器 → 编排器闸门)是一套**测量 + 决策**系统。它会输出"子版本相对父版本是否显著改善"的裁决,并据此晋升/拒绝。问题是:**怎么知道这套机器本身是对的?** 如果聚合把噪声当信号、闸门抓不住烂提示词、或对真实改进方向判反,整条棘轮就会朝错误方向复利。

验证一套测量系统的唯一可靠办法是**喂给它已知答案的输入,看它是否报出该结论**(known-answer / 对照实验)。本模块把这一方法工程化:在一个**冻结评测集**(见 [基线场景库 baseline_smoke_v1](./N.2.基线场景库%20·%20拟人化能力覆盖与种子清单.md))上,用三条结果可预期的"对照提示词"当 child,与当前 champion(parent)做 child-vs-parent 配对评测,核对流水线是否如预期反应。

> 关键边界:本模块验证的是**优化流水线机器是否正常**(主张 A),**不是 AI 是否更像人**(主张 B)。后者需要 holdout + 扩量 + 真人校准(见 §11)。

本模块含**两类互补的自检**:

- **评估链对照(§5、§6.1–6.6)**:用负/正/空三条手写对照,验"对局引擎 → 裁判 → 评分聚合 → 闸门"这条**测量+决策链**(即"裁判")判得对不对。
- **优化器自检(§6.7,零对局)**:挖一个已知可修的坑喂给真优化器,验"优化器产出的子代是否覆盖了坑"——即"诊断进 → 编辑出"对不对。

两者解耦了"优化器有效性"的两半:**优化器自检**验"优化器 ⇒ 好编辑";**正对照**验"好编辑 ⇒ 被流水线 credit"。组合即"优化器 ⇒ 被 credit 的改进",**无需端到端跑优化器**(那会重测一遍评估链、又叠加优化器自身随机性,既贵又吵)。

## 2. 目标

- 一键在指定 `set_id` 上跑**负/正/空**三对照(child-vs-parent),核对流水线行为是否符合预期。
- **过程可视化**:逐局(父 + 各对照)实时状态 + 每条对照的判定结果 + 闸门 + 指标,经 socket 流式呈现。
- **无副作用**:不改 champion、不增代数、不写历史代/版本库血脉。
- 最大化复用编排器现成原语(配对评测 + paired_cache、评分聚合、接受闸门),与正式闸门**同一套口径**。

## 3. 非目标

- **不**对 AI 拟人度下结论(N 小、无 holdout 时尤其不可)。
- **不**实现持久化与重启续接(测试工具,run 态仅存内存)。
- **不**新增评分/聚合/闸门逻辑——只调用既有实现。
- **不**改场景/探测 schema。

## 4. 约束与假设

| 约束 | 现状(以代码为准) | 影响 |
|---|---|---|
| 运行时取提示词 | `ai.service.buildDiscussionSystemPrompt` 按 `version_id` 从 DB(`sandbox_prompt_versions`)读正文,**读不到则静默回退默认** `ai-player/system-discussion.txt` | control 正文**必须先落库再评测**,否则对照失效(见 §6.3) |
| 占位符 | 讨论提示词用 `{{persona}}` 注入人设 | control 提示词须保留 `{{persona}}` |
| 投票提示词 | `buildVoteSystemPrompt` 恒用默认 `system-vote.txt`,**不受 version 影响** | 对照只作用于讨论阶段;父子投票口径一致,不偏配对差 |
| 统计样本量 | 聚合 `N = 场景数`;`minWilcoxonN=10`、`mdeMargin=3`(`aggregate/types.ts`) | `baseline_smoke_v1`(6 场景)下 `p=null`、CI 偏宽,多判 `inconclusive`(正对照判定须宽松) |
| 评测集 | `baseline_smoke_v1` 仅 optimize、无 holdout(见 N.2) | 只能验第一道闸 |

## 5. 方案概览

```
前端 /control-test 页 ──POST run(set_id)──► ControlTestController.startRun()
                                                  │ 立即返回 run_id(非阻塞)
                                                  ▼
                                          ControlTestService.execute()(后台)
   parent 评测一次 ──┐                              │
                     ├─ PairedEvalService.runVersionEval(+paired_cache,父代复用)
   对照×N 评测 ──────┘   → buildValidation(子vs父) → optimizeGate → assess(核对预期)
                                                  │ EventEmitter: status/game/control/done
                                                  ▼
                                          ControlTestGateway ──emit controltest.*──► socket
                                                  ▼
                                          前端 game-client 合并 → 页面过程可视化
```

三对照(child 提示词由 parent 派生或替换,见 §6.2):

| 对照 | child 怎么造 | 期望流水线反应(= pass 判据) | 它证明 |
|---|---|---|---|
| **空 null** | 与 parent **逐字相同**(仅版本号不同) | `blind_suspicion_margin` 判 `inconclusive` **且**闸门 `reject` | 噪声不被当信号(配对做差无偏、MDE 标定对) |
| **负 sabotage** | 换成"标准助手"(算对题/照写/报具体值/承认是AI/分点客服腔) | 闸门 `reject` **且**(`margin` 判 `regressed` 或 `probe_pass:*`/`veto_rate` 退步) | 决策信号方向对、能抓烂(auto_eval/veto 生效) |
| **正 known-good** | parent + 小幅强化算术/实时/表演回避 | **不**被判 `regressed`、`probe_pass`/`veto`/存活不退步(N 小不强求 promote) | 对真实改进方向敏感、不判反 |

## 6. 详细设计

### 6.1 模块组成(`apps/api/src/sandbox/control-test/`)

| 文件 | 职责 |
|---|---|
| `control-prompts.ts` | 三对照提示词构造器 + 内容哈希版本号 `ctl-<kind>-<hash8>` |
| `control-test.types.ts` | 运行态 `ControlTestRun` + 结果 `ControlResult` + 逐局 `ControlGameItem` 等类型 |
| `control-test.service.ts` | 后台流式 run:kickoff、execute、逐局记录、判定(assess)、事件 emit |
| `control-test.gateway.ts` | 两服务的 `EventEmitter` → socket `controltest.*` / `optcheck.*` 桥接 |
| `control-test.controller.ts` | REST:对照测试 `run`/`stop`/`state`/`controls` + 优化器自检 `optimizer/*` |
| `optimizer-holes.ts` | 可挖的坑注册表 `DIGGABLE_HOLES` + `digHole`(删纪律行)+ 关键词兜底 |
| `optimizer-check.types.ts` | 优化器自检运行态 `OptimizerCheckRun` + 单坑结果 `OptHoleResult` |
| `optimizer-check.service.ts` | 零对局自检:挖坑 → 合成诊断 → 真优化器 → L0/L1/L2′ 判定、流式 emit |

### 6.2 三对照提示词(`control-prompts.ts`)

- **null**:`build = (parent) => parent`,正文与父逐字相同。其版本号按内容哈希得 `ctl-null-<hash>`,与 champion 版本号不同 → 走**独立第二臂评测**,两臂只差 LLM 采样噪声(真正的 A-A)。
- **negative**:`build = () => SABOTAGE`,一段独立的"标准助手"提示词(末尾保留 `{{persona}}`)。会让 `bsl_001/002/003`(算术/表演/实时)的 `auto_eval` 判 fail,并让 `bsl_007/008/023`(纯社交)的盲测可疑度上升——**同时压 auto_eval 与裁判两条信号通路**。
- **positive**:`build = (parent) => parent + POSITIVE_HARDENING`,在父提示词后追加针对算术/实时/表演的小幅回避强化。

### 6.3 无副作用与"先落库再评测"(关键设计)

`DEC-CTL-NO-SIDE-EFFECT`:对照评测**不走** `orchestrator.runGeneration`(那会改 champion/代数/历史代),而是直接拼 `PromptVersion` + 调用底层 `PairedEvalService`。control 版本是临时的:

1. 版本号 = `ctl-<kind>-<内容哈希>`:内容变则 id 变,既避免复用陈旧 `paired_cache`,又让 null 对照与父代分属两臂。
2. **先 `await repo.upsertPromptVersion(child)` 落 DB,再评测**。因为对局运行时 `ai.service` 按 `version_id` 从 DB 读正文,读不到会**静默回退默认提示词**使对照失效;`PromptVersionStore.save` 是 write-through(异步),不保证落库早于对局读取,故必须显式 `await` 直写 repo。
3. 评测结束 `finally` 中 `repo.deletePromptVersion(id)` 删除。control 版本**从不进** `PromptVersionStore`(内存版本库),因此不出现在版本列表/血脉中;残留的 paired_cache / ai.service 内存缓存按 id 隔离、无害。

### 6.4 运行生命周期与流式事件

`ControlTestRun.phase`:`evaluating_parent` → `running_controls` → `settled`。

- `startRun()`:校验 set/champion → 建内存 run(emit `status`)→ 后台 `void execute()` → 立即返回 `run_id`。守卫:已有非 settled 的 run 时拒绝并发。
- `execute()`:
  1. 父代评测一次(`onGameStatus` 记 `side="parent"`,emit `game`);三对照复用其评分(paired_cache 按版本号命中)。
  2. 逐对照:落库 → 评测(`onGameStatus` 记 `side=kind`)→ `buildValidation` → `optimizeGate` → `assess` → push 结果(emit `control` + `status`)。
  3. 落定:phase=`settled`、算 `overall_pass`、emit `status` + `done`。**settled 的 run 保留**,供刷新/导航后回看,直到下次 `startRun`。
- `stop()`:置 `stopRequested`,`PairedEvalService` 在局间 `shouldStop` 退出,settle 为 `stopped`(`finally` 仍清理 control 版本)。

### 6.5 判定逻辑(`assess`)

对每条对照,从 `ValidationReport` 抽每个 form 桶的 `margin(blind_suspicion_margin)/ rounds_survived / plurality_rate / veto_rate / probe_pass:*` 的 `verdict`,折成几个跨桶布尔量,再按 `kind` 给出 `pass` 与 `notes`。无配对数据(`buckets` 为空)→ `pass=false` 并提示"检查对局是否大量失败"。

**先约定中间量**(均来自 `assess`,跨所有 form 桶聚合;`*Regressed` 取"任一桶判 regressed"):

| 量 | 定义 |
|---|---|
| `marginImproved` | 任一桶 `margin.verdict == improved` |
| `marginRegressed` | 任一桶 `margin.verdict == regressed` |
| `allMarginInconclusive` | 所有桶 `margin.verdict == inconclusive` |
| `anyProbeRegressed` | 任一桶任一 `probe_pass:*` 判 regressed |
| `vetoRegressed` / `survivalRegressed` | `veto_rate` / `rounds_survived` 任一桶判 regressed |
| `rejected` | `gate.decision == "reject"`(**复用正式 `optimizeGate`,与生产同口径**) |

**三条对照的 `pass` 判据(与 `control-test.service.ts` 逐字对齐):**

| 对照 | `pass = ` | 闸门项? | 读法 |
|---|---|---|---|
| **空 null** | `allMarginInconclusive && rejected` | ✅ 必须 `reject` | 同提示词只剩噪声:主指标须停在 inconclusive,且**不许过闸**。任一桶判 improved/regressed,或闸门 promote → 判失败(把噪声当信号)。 |
| **负 negative** | `rejected && (marginRegressed \|\| anyProbeRegressed \|\| vetoRegressed)` | ✅ 必须 `reject` | 烂提示词:既**不许过闸**,又必须至少有一路近真值信号(margin/probe/veto)判退步。烂提示词竟过闸 = 最严重失灵。 |
| **正 positive** | `!marginRegressed && !anyProbeRegressed && !vetoRegressed && !survivalRegressed` | ❌ **不含闸门项** | 真改:只要求四路近真值信号**都不判 regressed**(方向不判反)。闸门 promote 还是 reject **完全不参与** pass。 |

> **关键:闸门决策(promote/reject)只对空/负对照是 pass 的必要条件;对正对照则被刻意排除。**
> 原因见 §4 约束与 §8 `正对照判据宽松`:正对照只做"小幅"真改,在 `baseline_smoke_v1`(N=6 场景、`minWilcoxonN=10`、`mdeMargin=3`)下 CI 偏宽、Wilcoxon `p=null`,主指标 `margin` 几乎必然停在 `inconclusive` 而非 `improved`。而闸门要 promote 必须 `margin` 显著 `improved`(见 [编排器闸门](./0.编排器模块%20·%20方案设计.md),`optimizeGate` 主信号"仅不退不够")。所以**正对照下闸门几乎一定 `reject`,这是预期行为,不构成失败**——若把"必须 promote"写进正对照判据,真改也会被频繁假阴。"未达 improved"只写进 `notes` 提示点估计方向,不影响 `pass`。

**`notes`(诊断性,不影响 `pass`,仅解释异常/提示):**

| 对照 | 触发条件 | note |
|---|---|---|
| null | `marginImproved \|\| marginRegressed` | "A-A 却出现显著判定(把噪声当信号),流水线异常" |
| null | `!rejected` | "A-A 不应过闸(同提示词不该被判进步)" |
| negative | `!rejected` | "烂提示词竟过闸 → 决策信号失灵(最严重)" |
| negative | `!(marginRegressed \|\| anyProbeRegressed \|\| vetoRegressed)` | "无任何近真值信号判退步 → 可能 N 太小、裁判未生效或 auto_eval 未点火" |
| positive | `marginRegressed` | "真实改进被判 regressed → 方向判反" |
| positive | `anyProbeRegressed` | "probe_pass 退步 → 对探测改进不敏感或判反" |
| positive | `!marginImproved` | "未达 improved(N=6 下属预期;看点估计方向即可)" |

> 据此可读懂"闸门 reject 但正对照显示通过":正对照 pass 不看闸门,只要 margin 是 inconclusive(非 regressed)即过;`notes` 里的"未达 improved"是**预期提示**,非失败标记。`overall_pass = 三条 control 均 pass`(`settle`)。

### 6.6 前端页面(`/control-test`)

复用编排器页的过程可视化范式:
- **控制面板**:评测集单选、对照勾选(预览 `controls`)、种子/每种子局数/讨论秒/裁判模型、一键跑 / 停止。
- **过程可视化**:阶段叙述 + 三段 stepper;**父 + 各对照各一条进度条**(从 `games` 折算);**逐局列表**(`pending→running→scoring→finished/failed`,带轮次/AI存活/margin/否决 + 「对局记录」直达房间);**对照结果卡片**(如预期✓/异常✗ 徽章、闸门建议、各指标点估计+CI+verdict、notes);底部 caveats。
- 数据流:`game-client` 订阅 `controltest.status`(整体快照替换)/`game`(逐局 upsert)/`control`(单条结果合并);`GET state` 做首屏与断线兜底。

### 6.7 优化器自检(零对局)

验"优化器本身"是否有效,**不跑任何对局**。关键洞察:优化器的产物就是提示词,**直接看它有没有覆盖挖的坑即可**,不必跑完整流水线测其行为(那归正对照)。

流程(逐坑;`optimizer-check.service.ts` / `optimizer-holes.ts`):

1. **挖坑**:从基线删掉一条【有人测试你时】的具体抗测试纪律(`digHole` 按 marker 删整行),造出该探测类型处理被弱化的**种子(仅内存,不落库)**。可挖的坑见 `DIGGABLE_HOLES`(arithmetic / perform / injection / are_you_ai)。
2. **合成诊断**:零对局 → 无 ScoreRecord → 不能用 `championProfile` 算 profile,故**手搓** `ChampionProfile`(把挖的坑种成 `weakDimensions[0]`、对应 `probePassByType` 给低值)。优化器契约是"诊断进→编辑出",喂**合成但忠实**的诊断正是隔离优化器的正确做法。
3. **真优化器提案**:`buildOptimizerInput(种子, 合成profile, 坑target, …)` → `OptimizerService.propose`(纯 LLM 调用,不读 DB、不跑对局)。
4. **判定**:**L0** `validatePrompt`(保留 `{{persona}}`、长度预算、确改动、必填齐)· **L1** `child.target_dimension == 坑 target` · **L2′ 覆盖判定**——LLM judge「子代是否含与被删纪律【同概念、具体可操作】的指令」(judge 失败回退关键词)。
5. **坑深自检**:基线对每个探测类型还有通用兜底(【绝对禁止】#3 / 【守住身份】),单行删除多为**浅坑**,故对**种子也跑一次覆盖判定**:种子已覆盖 = 浅坑(标注,仍按子代覆盖记分);种子未覆盖而子代覆盖 = 干净恢复。
6. `pass = L0 && L1 && 子代覆盖`;`overall_pass = 全坑 pass`。

成本:每坑 **1 次优化器调用 + 2 次覆盖 judge 调用,0 对局、0 DB 写**(种子/子代都只在内存,`propose` 不读 DB)。后台流式经 `optcheck.*`;前端在 `/control-test` 页"优化器自检"区呈现逐坑卡片(L0/L1/L2′ 徽章 + 假设 + 命中原句 + notes)。

> 为什么不用端到端"修复"测试:见 §1——"优化器⇒好编辑"(本节)与"好编辑⇒被 credit"(正对照)两半解耦后组合即可,端到端会重测评估链并叠加优化器随机性。

## 7. 数据模型 / 接口 / 配置

### 7.2 Socket 事件(默认 namespace)

| 频道 | 载荷 | 含义 |
|---|---|---|
| `controltest.status` | `ControlTestRun \| null` | 运行态全量快照(phase 变更/对照完成/落定时发) |
| `controltest.game` | `ControlGameItem` | 单局状态增量(按 side×scenario×seed×run upsert) |
| `controltest.control` | `ControlResult` | 单条对照判定完成 |
| `controltest.done` | `{run_id, decision, overall_pass, error?}` | 落定 |
| `optcheck.status` | `OptimizerCheckRun \| null` | 优化器自检运行态全量快照 |
| `optcheck.hole` | `OptHoleResult` | 单个坑结果增量(按 hole_id 合并) |
| `optcheck.done` | `{run_id, decision, overall_pass, error?}` | 优化器自检落定 |

### 7.3 关键类型(`control-test.types.ts`,摘要)

- `ControlTestRun`:`run_id / phase / set_id / eval_set_version / parent_version_id / plan / kinds / current_kind? / games[] / controls[] / caveats[] / overall_pass? / decision? / error?`。
- `ControlResult`:`kind / label / child_version_id / expectation / gate / buckets[] / pass / notes[]`。
- `ControlGameItem`:`side("parent"|kind) / scenario_id / seed / run / status / room_id? / match_id? / margin? / veto? / phase? / current_round? / ai_alive? / ai_total? / error?`。
- `OptimizerCheckRun`:`run_id / phase / base_version_id / optimizer_model_id? / judge_model_id? / current_hole? / holes[] / overall_pass? / decision? / error?`。
- `OptHoleResult`:`hole_id / target / probe_type / status / child_version_id? / validate? / target_hit? / seed_covered? / coverage? / hypothesis? / pass / notes[] / error?`。

## 8. 备选方案与取舍

| 决策 | 备选 | 选择与理由 |
|---|---|---|
| `DEC-CTL-BACKGROUND` 后台流式 run | 阻塞 HTTP 返回完整报告 | 选**后台流式**:满足过程可视化(逐局/逐对照实时),且单次跑多局耗时长,阻塞接口体验差、易超时 |
| `DEC-CTL-NO-SIDE-EFFECT` 独立无副作用评测 | 复用 `runGeneration` | 选**独立**:对照不应改 champion/代数/历史;复用 `runGeneration` 会污染迭代状态 |
| `DEC-CTL-EPHEMERAL-VERSION` content-hash 临时落库再删 | 进 `PromptVersionStore` | 选**临时直写 repo + 删除**:不污染版本库/血脉;哈希 id 兼顾缓存正确性与 A-A 第二臂 |
| `DEC-CTL-IN-MEMORY` run 态仅内存 | 持久化 + 重启续接 | 选**内存**:测试工具,重启丢失可接受;省去持久化复杂度 |
| 正对照判据宽松(只断言 not regressed) | 断言必须 `improved`/`promote` | 选**宽松**:N=6 下真改也常 `inconclusive`,强断言会假阴 |
| `DEC-OPT-CONTENT-CHECK` 优化器自检零对局看覆盖 | 端到端"修复"测(跑完整流水线) | 选**零对局覆盖检查**:优化器产物是文本,直接看是否覆盖坑;"好编辑⇒被 credit"由正对照负责,端到端会重测评估链+叠加优化器随机性 |
| `DEC-OPT-SYNTH-PROFILE` 合成 champion 画像 | 跑坏种子算真 profile | 选**合成**:优化器契约是"诊断进→编辑出",忠实合成诊断即可隔离优化器、省一轮对局;瞄准信号本身的验证留作可选 |

## 9. 风险与失败模式

| 风险 | 触发条件 | 影响 | 缓解 |
|---|---|---|---|
| 写穿竞态 | control 正文未落库即开跑 | 回退默认提示词→对照失效(假"无差异") | 评测前 `await repo.upsertPromptVersion`(§6.3) |
| N 太小多 inconclusive | 6 场景 | 正/空对照难显著 | 正对照判据宽松;caveat 显式提示;扩量后更准 |
| 正对照不可靠强断言 | 真改未达显著 | 误报"流水线异常" | 只断言 not regressed + 报点估计方向 |
| 与编排器并发 | 同时跑 control-test 与一代 | 共用 GameService | GameService 多房间安全;control-test 自身单 run 守卫 |
| 重启丢 run 态 | 进程重启 | 进行中 run 丢失 | 测试工具,接受;settled 结果本就临时 |
| 仅验第一道闸 | set 无 holdout | 不覆盖泛化/过拟合 | caveat 提示;补 holdout 成员后升级 set |

## 10. 验证方式

- **静态**:`apps/api` `nest build`、`apps/web` `next build`(TypeScript + lint)均通过;`/control-test` 路由生成。
- **逻辑**:`assess` 判定用 12 个代表性用例(含 null 误判、负对照漏抓、正对照判反等异常分支)独立校验通过。
- **优化器自检挖坑机制**:`digHole` 在真实基线上对 4 个坑(arithmetic/perform/injection/are_you_ai)各命中**唯一行、恰删 1 行**,已校验通过。
- **优化器自检实跑(需 LLM key)**:`/control-test` 页"优化器自检"→「跑优化器自检」→ 预期各坑 L0✓/L1✓/L2′✓ 且 `overall_pass`;若某坑 L2′✗,即优化器没把被删的具体处理补回。
- **实跑(需 API+DB+LLM key+web)**:首页(debug)或编排器页 →「对照测试」→ 选 `baseline_smoke_v1` →「一键跑三对照」→ 预期 **空=inconclusive+reject、负=reject+退步、正=不退步**;`overall_pass=true` 即该集合范围内机器可信。

## 11. 已知限制

- 只验**第一道闸**(评测集 holdout 为空时);第二道闸(留出复核/泛化)、第三道闸(真人校准/防 Goodhart)不在本模块范围。
- 对照只作用于**讨论阶段**;投票恒用默认提示词(父子一致,不偏配对差)。
- run 态**不持久化**;重启后进行中 run 丢失。
- 阻塞性 LLM 成本:默认 `(1+对照数)×场景数×seeds×runs` 局,耗时与费用随之增长(可减 `kinds` / 调 `runs_per_seed`)。
- 本模块结论仅及**流水线机器**,不及 AI 拟人度。
