# 编排器前台 · 界面设计(过程可视化 + 人机确认)

> **定位**:为当前全自动一代闭环(`POST /sandbox/orchestrator/run-generation-auto`)设计前台界面,核心三件事:**过程可视化**、**人工确认**、**接受/拒绝**。
> **参考**:旧 `/iteration`(自动对局评估自迭代)的显示逻辑——socket 流式实时进度 + `awaiting_confirmation` 人机暂停。旧链路已废弃(D1),本设计对接当前 `SandboxModule` 编排器,**复用其 UX 模式而非代码**。
> **关键约束**:当前编排器是**阻塞式 REST**(一次请求跑完一整代才返回、无进度、无暂停点)。要实现"过程可视化 + 人机确认",编排器后端需演进为**后台运行 + 进度事件 + 确认暂停**(本文 §3/§4/F0)。本文是前端 + 必要后端演进的整体设计。

---

## 1. 用户体验目标(对齐旧 iteration 的体感)

1. **过程可视化**:一代闭环每一步实时可见——`评测 champion → 优化器提案 → validate → 评测 child → 闸门 → (确认) → 落定`。每个 `(scenario,seed,run)` 的 `margin / veto` 落地即显示,不等整代跑完。
2. **人工确认**:`confirm` 模式下,优化器产出 + 闸门给出建议后**暂停待人审**;人能看提示词 diff、看验证信号、改提示词、再决定。
3. **接受/拒绝**:一键**接受**(晋升 child 为 champion)或**拒绝**(进 `tried_and_rejected`);可选**编辑后接受**(改 `prompt_text` 再晋升)。
4. 全程**可观测、可复现、可回看**(历史代)、**可回滚**(任意一代的 champion 可重激活)。

---

## 2. 一代闭环的生命周期状态机(扩展编排器)

当前(同步、黑盒):
```
champion → eval → propose → validate → gate → promote/reject
```

演进后(后台、可暂停,一次只有一个活跃 run):
```
idle
 → evaluating_champion     (跑 champion 在 eval 集上的全部 (scenario,seed,run),逐局 emit)
 → optimizing              (优化器单候选提案)
 → validating              (validate_prompt)
 → evaluating_child        (跑 child 配对评测,逐局 emit)
 → gating                  (buildValidation + optimizeGate)
 → awaiting_confirmation   (仅 confirm 模式:暂停,等人 accept/reject)
 → settled                 (promoted | rejected)
 → idle
```

- **mode = `auto`**:过 `gating` 后按闸门结果直接 `settled`(全自动,等同当前)。
- **mode = `confirm`**:过 `gating` 后停在 `awaiting_confirmation`,等 `POST /confirm`。
- 活跃 run **持久化**(写入 `orchestrator-state.json` 的 `active_run`),断线/重启可续(对齐旧 iteration `reconcileStaleRuns`)。

## 3. 传输:socket 事件(复用 `game-client` 基建)

> 复用旧 iteration 的 socket 流式模式;备选传输见 §8。前端 `lib/game-client.tsx` 已有 socket 单例 + `iteration.*` 监听先例,加 `orchestrator.*` 同构。

**后端**:`OrchestratorService` 增 `EventEmitter`;新增 `OrchestratorGateway` 把内部事件桥接到 socket。

| socket 事件 | 时机 | 载荷 |
|---|---|---|
| `orchestrator.status` | 每次 phase 变化 / 重连 | 全量快照:`state` + `active_run{phase, mode, generation, champion, child, progress{champion_done/total, child_done/total}}` |
| `orchestrator.match` | 每局评分完成 | `{version, scenario_id, seed, run, margin, veto, status}` |
| `orchestrator.proposal` | 优化器产出 + validate 后 | 候选 `child` meta + `diff_summary` + `validate{ok,reasons}` |
| `orchestrator.gate` | 闸门判定后 | `validation`(各桶各 metric verdict)+ `gate{decision,reasons,marginVerdict}` |
| `orchestrator.done` | 一代 `settled` | `{generation_id, decision, champion_after}` |

前端 `game-client`:`orchestrator.status` 做合并式快照(同旧 `iteration.status` 的 merge 而非 replace);其余增量更新对应区块。重连时 `GET /sandbox/orchestrator/state` 拉一次全量。

## 4. REST 端点(编排器演进)

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/sandbox/orchestrator/run-generation-auto` | **改为非阻塞 kickoff**:校验入参 → 起 `active_run`(后台)→ 立即返回 `{run_id}`;进度走 socket |
| POST | `/sandbox/orchestrator/stop` | 中止活跃 run → `settled(stopped)` |
| POST | `/sandbox/orchestrator/confirm` | body `{accept: bool, edited_prompt_text?: string}`;仅在 `awaiting_confirmation` 可用 → `promote`/`reject` → `done` |
| GET | `/sandbox/orchestrator/state` | 全量快照(重连/首屏) |
| GET | `/sandbox/orchestrator/generations` | 历史代列表(读 `sandbox-out/generations/`) |
| GET | `/sandbox/orchestrator/generations/:id` | 单代详情 |
| GET | `/sandbox/orchestrator/versions` | PromptVersion meta 列表 |
| GET | `/sandbox/orchestrator/versions/:id` | 含 `prompt_text`(diff 用) |

> 现有 `seed-baseline` / `state` / `run-generation`(手动 child)保留。

### 5.1 控制面板
- **eval 配置**:场景多选(来自 `GET /sandbox/examples`)、`seeds_per_scenario`、`runs_per_seed`、裁判模型、优化器模型、`discussion_seconds`、`eval_set_version`。
- **模式**:`auto`(闸门自动晋升)/ `confirm`(暂停待人确认)。
- **指定靶子**:`assigned_target`(可选,空=自动取 champion 最弱探测类型)。
- **按钮**(随 phase 切换):`开始一代` / `停止`;`awaiting_confirmation` 时显 `接受候选` / `拒绝候选` / `编辑后接受`。
- **状态行**:`phase` / 第 N 代 / champion version_id / tried_and_rejected 数。

### 5.2 过程可视化面板
- **step badge + narration**:phase 解释文(对齐旧 `currentStepText`,如 `awaiting_confirmation → "闸门建议晋升,确认后进入下一代"`)。
- **stepper**:`eval_champion → optimize → validate → eval_child → gate → confirm → settle`,每步 `done / active / pending`。
- **双进度条**:champion 评测 `done/total`、child 评测 `done/total`(复用 `.timer-track/.timer-fill`)。
- **单局流**(当前评测侧):每条 `scenario | seed | run → margin / veto / 状态`,`orchestrator.match` 到一条追加一条。
- **champion 画像卡**:各 probe 通过率条 + mean margin + veto rate(来自 `championProfile`)。
- **优化器提案卡**:`target / edit_type / hypothesis / diff_summary` + 折叠的 prompt diff( vs champion)+ `validate{ok,reasons}`。
- **闸门卡**:各 metric verdict(`blind_suspicion_margin / rounds_survived / veto_rate / plurality_rate / probe_pass:*`)+ decision + reasons。

### 5.3 历史代(全宽)
- 列表(来自 `generations/`):每条 `gen N | champion before→after | child | gate decision | margin 点估`。
- 点 `详情` → 单代详情 modal(各桶各 metric verdict + CI + gate reasons + child diff)。

### 5.4 版本面板(可选,Phase 1.5)
- PromptVersion 列表,champion 高亮;点看 `prompt_text`、与父代 diff、`validated_metrics`。
- **只读 + diff**,不做大规模在线编辑(编辑走"编辑后接受"轻量入口);完整版本管理延后。

## 6. 人机确认流(核心 · confirm 模式)

```
跑到 gating → 产出 gate{decision, reasons} + 完整 validation
   → 暂停 awaiting_confirmation
   → 前台展示:提案 prompt diff + 各 metric verdict(含 CI)+ 闸门建议 + champion 画像
   → 人审,三选一:
       〔接受候选〕   → POST /confirm {accept:true}        → promote child → done
       〔拒绝候选〕   → POST /confirm {accept:false}       → tried_and_rejected → done
       〔编辑后接受〕 → 改 prompt_text → POST /confirm {accept:true, edited_prompt_text} → 用改后文本 promote → done
```
等价于旧 iteration 的 `postRoundMode: auto_optimize_wait_confirm`。
`auto` 模式则跳过暂停,闸门结果直接落定(等同当前全自动行为)。

## 8. 备选与权衡

| 决策 | 选 | 备选 | 理由 |
|---|---|---|---|
| 进度传输 | **socket** | SSE / 轮询 | 复用 `game-client` 基建 + 旧 iteration UX;SSE 更简但单向且要新端点;轮询卡顿 |
| 阻塞 vs 后台 | **后台 kickoff** | 阻塞 | 一代分钟级,阻塞会让 HTTP/前台干等、超时 |
| 在线编辑 | **confirm 内轻量** | 完整版本编辑器 | 完整编辑延后;confirm 给"编辑后接受"即可覆盖人工改提示词 |
| 单 run | **一次一个活跃 run** | 多 run 并发 | MVP 单线贪心;多 run/多种群在 Phase 4+ |

## 10. 验收

1. **过程可视化**:`auto` 模式跑一代,stepper/进度条/单局流实时刷新;断线重连后 `status` 快照同步。
2. **人工确认**:`confirm` 模式跑到 gate 后停在 `awaiting_confirmation`;人能看 prompt diff + 各 metric verdict(含 CI)+ 闸门建议,并 `接受/拒绝/编辑后接受`。
3. **接受/拒绝**:`accept` → champion 更新 + 历史代记录 `promoted`;`reject` → `tried_and_rejected` + 历史代 `rejected`。
4. **可复现/回看**:历史代列表可点开看完整 validation + gate;版本面板可看任意版本提示词与父代 diff。

---

> 一句话:这套前台把"一代闭环"从黑盒阻塞调用,变成**可旁观过程、可在闸门前叫停确认、可一键接受/拒绝/微改**的可控迭代台——复用旧 iteration 的 socket 流式 + `awaiting_confirmation` 体感,对接当前 `SandboxModule` 编排器。实现前需先把编排器演进为后台 + 事件 + 确认暂停(F0),再做前台(F1/F2)。
