# 端到端走查 · 一条数据从场景到晋升

> **用途**:各模块的 schema 都齐了,但散在十几份文档里。本文用**同一个贯穿例子**(场景 `sc_0123` / 探测 `probe_realtime_v3` / 候选版本 `v6.1-realtime`),把数据从一条场景一路走到一次晋升,**每一棒交接处贴真实数据**,让人一眼拼出全景。
> 这是"把已有 schema 串成一条线"的导览,**不引入新设计**;字段权威定义仍以各专文/契约为准(每步标注出处)。

---

## 0. 全景:七棒接力 + 一次放大

```
① Scenario(输入契约)
     │  引擎读它,跑一局
     ▼
② MatchRecord(输出契约)        ← 引擎产:这局客观发生了什么
     │  裁判读它,评分
     ▼
③ ScoreRecord(裁判)            ← 一局的分:客观指标 + 盲测可疑度 + 失败案例
     │  ★这里要放大:很多局 → 聚合
     ▼
④ 聚合信号(核心机制)          ← 多局压成:验证信号 / 瞄准信号 / 失败案例
     │  喂优化器(单候选×外层派靶)
     ▼
⑤ child 候选(优化器)          ← 针对一个破绽的新提示词 + 可证伪假设
     │  编排器配对评测
     ▼
⑥ GenerationEval(编排器)       ← 子vs父配对 + 显著性 + 留出复核 → 接受/拒绝
     │  晋升
     ▼
⑦ PromptVersion=champion        ← 上线为新 AI 玩家
     │  (在线真人对局)
     └──► 回灌场景库 + 真人校准 ── 闭环回到 ①/③ 的可信度
```

> **③ 和 ④ 之间是关键放大**:走查前六棒讲"一条数据的旅程",但 ③→④ 实际是"**几百上千条 ScoreRecord** 被聚合成几个信号"。我们先单局走完一条,再在 §4 展开"多局如何收敛"。

---

## 1. ① 起点:一条 Scenario(出处:《场景与探测·Schema契约》)

编排器从场景库 `optimize` split 取出这条 spotlight 场景。它设定了:房间 4 人、被测 AI 在槽位 B(用懒散人设 `p_lazy`)、R2 起跑、第 2 轮由侦探 C 在第 1 条发言后丢出"实时信息"探测。

```json5
{
  "scenario_id":"sc_0123","form":"spotlight","split":"optimize",
  "mode":"scripted_intent","seed":12345,"ai_under_test_slot":"B","max_rounds_forward":2,
  "coverage_tags":{"probe_type":"realtime_info","social_situation":"even","room_style":"casual",
                   "round_position":"R2","difficulty":"normal","room_size":4,"ai_persona":"p_lazy"},
  "roster":[
    {"slot":"A","role":"detective","persona_id":"p_det_aggr","model_id":"m_det_x"},
    {"slot":"B","role":"ai_under_test","persona_id":"p_lazy"},   // 提示词版本运行时注入
    {"slot":"C","role":"detective","persona_id":"p_det_quiet","model_id":"m_det_x"},
    {"slot":"D","role":"filler","persona_id":"p_meme","model_id":"m_filler_y"}
  ],
  "seed_history":{"prior_turns":[/* R1 历史 */],"prior_rounds":[{"round":1,"tie":true}],"start_round":2},
  "probe_schedule":[{"probe_ref":"probe_realtime_v3","round":2,"timing":{"after_turn":1},"from_slot":"C"}],
  "vote_policy":"live","source":{"type":"human_failure","match_id":"hm_8842"}
}
```

注意 `source.match_id=hm_8842`——**这条场景本身就是从真人失败回灌来的**(第⑦棒闭环的产物),它考的正是"被问实时信息会不会露馅"。`probe_ref` 指向的探测实例:

```json5
{"probe_id":"probe_realtime_v3","type":"realtime_info","intent":"突然问只有真人/联网才知道的实时信息,看AI会不会自信答错",
 "templates":["今天几号啊","现在几点了"],"pass_if":"未自信给出具体实时答案;以人设化方式糊弄/反问",
 "auto_check":{"checker":"realtime_specific_value","params":{"value_types":["date","time"]}},
 "split_exposure":"optimize","rotation_group":"rg_realtime"}
```

**这一棒在测的就是被测版本 `v6.1-realtime`**:它的提示词被注入 B 槽位,引擎据此跑一局。(为对照,父版本 `v6` 会用**同一条场景、同一 seed** 也跑一局——配对评测的根基,见 §5。)

## 2. ② 引擎产出:MatchRecord(出处:《对局引擎》《对局记录·Schema契约》)

引擎按主循环跑:加载 R1 历史 → R2 讨论(侦探 A 先怀疑 B,侦探 C 按 `probe_schedule` 在第 1 条后丢出实时信息探测)→ B 回应 → 盲投 → 结算。产出(节选关键部分):

```json5
{
  "match_id":"m_sc0123_v6.1-realtime_run3","scenario_id":"sc_0123","scenario_form":"spotlight",
  "prompt_version_id":"v6.1-realtime","run_index":3,"seed":12345,"mode":"scripted_intent","vote_policy":"live",
  "ai_under_test_slot":"B","start_round":2,
  "transcript":[
    {"idx":13,"round":2,"slot":"C","role":"detective","text":"诶 今天几号来着","is_probe":true,"probe_ref":"probe_realtime_v3"},
    {"idx":14,"round":2,"slot":"B","role":"ai_under_test","text":"6月20号 咋了"}   // ← AI 自信报了具体日期
  ],
  "votes":[
    {"round":2,"voter_slot":"A","target_slot":"B","reason":"你太干净了","policy_applied":"live"},
    {"round":2,"voter_slot":"C","target_slot":"B","reason":"刚那答得太顺","policy_applied":"live"},
    {"round":2,"voter_slot":"D","target_slot":"A","reason":"随便","policy_applied":"rule"}   // filler 走共识跟随
  ],
  "eliminations":[{"round":2,"eliminated_slot":"B","tie":false,"tally":{"B":2,"A":1}}],
  "probe_events":[
    {"probe_ref":"probe_realtime_v3","type":"realtime_info","round":2,"from_slot":"C",
     "delivered_text":"诶 今天几号来着","ai_response_idx":14,
     "auto_eval":{"checker":"realtime_specific_value","result":"fail","detail":"答了'6月20号'"},
     "judge_eval_needed":false,"status":"delivered"}
  ],
  "outcome":{"ai_won":false,"ai_rounds_survived_from_start":0,"ai_eliminated_round":2,"reached_terminal":"ai_eliminated"},
  "status":"ok"
}
```

**这一棒发生了什么**:探测是"触发器+意图"由侦探 C 当场用自己口吻说出(`诶 今天几号来着`,不是死台词);AI 在 idx14 **自信报了"6月20号"**——`auto_eval` 当场判 `fail`(引擎用 `realtime_specific_value` checker 自动判,无需裁判);随后被票出(B 得 2 票)。**注意:这一 run 里 v6.1 表现失败了。** 这不矛盾——见 §4。

## 3. ③ 裁判产出:ScoreRecord(出处:《裁判评分模块》《裁判提示词模板》)

裁判读 MatchRecord,两类信号:

```json5
{
  "match_id":"m_sc0123_v6.1-realtime_run3","prompt_version_id":"v6.1-realtime","scenario_form":"spotlight",
  "judges":["judge_a","judge_b"],"judge_agreement":0.82,
  "outcome_metrics":{"rounds_survived":0,"eliminated_round":2,"plurality_by_round":{"2":true},
                     "probe_pass_by_type":{"realtime_info":0}},   // 直接采用引擎 auto_eval
  "blind_suspicion":{"per_round":[{"round":2,"scores":{"玩家1":35,"玩家2":58,"玩家3":30,"玩家4":40},"ai_score":58}],
                     "ai_final":58,"suspicion_rank":1,"suspicion_margin":+18},   // ★决策主代理
  "rubric":{"客服感":3,"结构化指纹":0,"能力暴露":1,"立场情绪":2,"博弈参与":2,"出戏":0,"语言质感":3,"存在感":3},
  "veto_triggered":false,
  "failure_cases":[{"round":2,"utterance":"6月20号 咋了","tell":"能力暴露","note":"被问今天几号自信报具体日期",
                    "blind_suspicion_delta":+28,"probe_ref":"probe_realtime_v3"}]
}
```

**两类信号各走各路**:`blind_suspicion`(裁判匿名盲测、像玩家不像法医)给出**相对量 `suspicion_margin=+18`**(AI 比房间其余最高分还高 18,最可疑)——这是决策主代理;`rubric` + `failure_cases` 是诊断,定位到 idx14 那句"能力暴露",`blind_suspicion_delta=+28`,供优化器知道"该修哪"。

---

## 4. ★ ③→④ 放大:很多局 → 聚合信号(出处:《评分聚合与优化信号·核心机制》)

**这是最关键、也最容易误解的一棒。** 上面那条是 `v6.1` 在 `sc_0123` 的**一个 run** 的 ScoreRecord。实际上:`v6.1` 在 **80 个 optimize 场景 × 5 runs = 400 局**都有 ScoreRecord,父版本 `v6` 在**同样 80 场景同样 seed**也有 400 局。聚合管线把这 800 条压成给优化器/编排器的信号:

```
800 条 ScoreRecord
  → 第1层 run 取均:每个 (version, scenario, seed) 压成一个 CellEstimate(消局内随机)
  → 第2层 (scenario,seed) 配对做差:v6.1 − v6,逐场景相减(消场景难度,差分放大器)
  → 第2.5层 scenario 收敛:N = 80 个场景(不是 800 局!统计单元是场景)
  → 第3层 bootstrap:估 paired_diff 的均值 + 95% CI
  → 第4层 MDE+CI:卡成 verdict
```

**于是单局的"失败"被正确地放进了全局视角**:`v6.1` 在 `sc_0123` 这一 run 被票出(margin +18,很糟),但聚合到**整个 optimize 集**后,它的盲测可疑度**比父版本 v6 平均低了 6 分**(见 §5)——因为 v6.1 针对"实时信息"做了通用回避反射改动,在**大多数**实时信息场景上不再露馅,即使在 `sc_0123` 这个 run 上仍翻了车。**单局是噪声,版本级配对差才是信号。**

聚合产出三块喂优化器(瞄准信号示例,来自 v6 的弱点画像):

```json5
"weak_dimensions":[{"metric":"probe:realtime_info","point":0.40,"ci95":[0.31,0.49],"reliability":"high"}],
"failure_clusters":[{"cluster":"实时信息类暴露","freq":0.18,"avg_delta":+26,"agreement":0.9,
                     "exemplars":[{"context":"C: 今天几号来着","ai":"6月20号 咋了","tell":"能力暴露"}]}]
```

> 注:这里展示的是**对父版本 v6 评测**得到的瞄准信号——它指出"realtime_info 通过率仅 0.40,是稳定弱点",**正是这个信号驱动了下一棒优化器去攻它**。

## 5. ⑤ 优化器产出:child 候选(出处:《优化器模块》)

编排器外层把 `probe:realtime_info` 作为一个**靶子**派给优化器(算子选 `generalize_to_reflex`,因为它在该破绽战绩最高),**单候选调用**产出 `v6.1-realtime`:

```json5
{"child":{
  "child_id":"v6.1-realtime","based_on":"v6","target":"probe:realtime_info",
  "edit_type":"generalize_to_reflex","placement":"shared",
  "hypothesis":"被问任何实时信息(日期/时间/天气/新闻)一律糊弄反问 → realtime_info 通过率 0.40 升到 ≥0.8,且不抬高其它维度",
  "prompt_text":"<完整子版本提示词:在【测试你时】段把'实时信息'并入通用回避清单>",
  "diff_summary":"加2条实时信息类反问范例,升维成通用回避反射"
}}
```

**这一棒的纪律**:只攻一个破绽(realtime_info);带**可证伪假设**(预测通过率 0.40→≥0.8);用通用回避反射而非"被问几号就说不知道"的背答案。这个 child 经 `validate_prompt` 预筛(锁定段未动、长度合规)后进入评测。

## 6. ⑥ 编排器裁决:GenerationEval(出处:《编排器模块》)

`v6.1-realtime` 与父版本 `v6` 在同一 optimize 集配对评测,过接受闸门后再上留出集复核:

```json5
{
  "generation":7,"parent_id":"v6","child_id":"v6.1-realtime",
  "eval_set":{"optimize":"optimize_v1","holdout":"holdout_v1"},"n_scenarios":80,"runs_per":5,
  "paired_metrics":{
    "blind_suspicion_ai":{"parent_mean":52,"child_mean":46,"paired_diff":-6,"ci95":[-9,-3],"mde":3,"significant":true},
    "veto_rate":{"parent":0.04,"child":0.01,"regressed":false},
    "probe_pass_rate":{"realtime_info":{"parent":0.40,"child":0.88,"regressed":false}}   // ← 假设被验证!0.40→0.88
  },
  "optimize_gate":"pass",
  "holdout":{"eval_set":"holdout_v1","held_out_probes":true,"blind_suspicion_ai_paired_diff":-5,"ci95":[-8,-2],"holds":true},
  "decision":"promote"
}
```

**三道闸在这一棒落地**:① 配对+显著性——可疑度配对差 −6、CI `[-9,-3]` 不含 0 且超 MDE,**显著**;② 无否决(veto 降)、无回退(realtime 通过率 0.40→**0.88**,优化器的**假设被验证**);③ 留出集用 **held-out 探测实例**复核仍 holds(−5),证明是**泛化**而非背了 optimize 的题。→ `decision: promote`。

---

## 7. ⑦ 晋升 + 闭环(出处:《编排器》《真人校准》《场景库·回灌》)

`v6.1-realtime` 晋升为新 champion,`PromptVersion.status=champion`,上线为在线 AI 玩家。然后闭环启动:

- **真人对局**用新 champion 跑,产真实结果;
- **真人校准**(每 M 代):把 v6.1 的沙盒 `suspicion_margin` 和真人被票出率配进相关性体检,确认沙盒代理没漂(`correlation_proxy_vs_real` 健康才放心继续);
- **失败回灌**:真人局里若 v6.1 仍在某新套路上被抓,抽象成新探测/场景入库——**就像 `sc_0123` 当初由 `hm_8842` 回灌而来一样**;
- **失败模式台账**:`fm_realtime_info` 的 `status` 从 `recurring/open` 更新为 `mitigated`,`real_world_rate` 记 `0.06→0.01`。

至此回到 §1 的起点:新的真实失败长成新场景,喂下一代优化。环闭合。

---

## 8. 把全链一句话串起来

> 一条场景 `sc_0123`(本身从真人失败 `hm_8842` 回灌而来)被引擎跑成 `MatchRecord`,裁判评成 `ScoreRecord`;**几百条这样的记录经聚合管线配对做差、压成"realtime_info 是弱点"的信号**;优化器据此单候选产出 `v6.1-realtime`(假设:通用回避反射能把通过率从 0.40 提到 ≥0.8);编排器配对评测确认可疑度显著降、通过率升到 0.88、留出集泛化成立,**晋升**;上线后真人校准验代理可信、真人新失败再回灌——**单局的失败是噪声,版本级的配对增益是信号,只有经三道闸的真增益沉淀为 champion。** 这条线,就是整套系统每一代在走的路。
