## 三大设计原则(贯穿所有模块)

1. **度量驱动**:真值 = **真人对局**的存活/胜负;诊断 = 裁判的分维度量表。先有度量,才谈得上稳定迭代。
2. **不冻结环境,用统计降方差**:不靠把对手写死来消方差(那会失真),而靠**同种子 + 配对比较 + 多 run 取均 + 加样本**。
3. **三道防自欺闸门**:
   - **配对 + 显著性**(防噪声):父/子同场景同种子,看配对差值且差异要显著。
   - **留出集 + 探测轮换**(防过拟合):验收场景/探测实例从不进优化器。
   - **真人校准**(防 Goodhart):定期用真人对局体检"沙盒分 ↔ 真人结果"的相关性。
4. (覆盖观)**覆盖目标 = 逼近真人失败分布,不是穷举**;对没覆盖到的未知攻击,靠提示词的**通用回避反射**兜底。

## 模块总览与数据流

```
场景库(分层抽样 + 真人失败回灌)
  ├─ optimize split        ├─ holdout split(含专用探测实例)
        │
        ▼
   对局引擎(scripted_intent 配对评测 / free 探索)
        │  产出 MatchRecord(转录+投票+probe_events+结算,可复现)
        ▼
   裁判模块
     ├─ 客观结果指标(纯计算,含引擎对 probe 的 auto_eval)
     └─ 裁判 LLM 两遍:① 匿名盲测可疑度  ② 分维度诊断 + 失败案例
        │  产出 ScoreRecord
        ▼
   优化器(LLM)◄── 当前版本 + 聚合指标 + 精选失败案例 + 失败模式台账 + 已试失败记忆
        │  产出多个子版本 PromptVersion
        ▼
   配对评测(子 vs 父,同优化集同种子)──► GenerationEval
        │
   显著优于 & 无否决 & 无维度回退? ──否──► 丢弃(写入"已试失败")
        │ 是
        ▼
   留出集复核(用 held-out 探测实例验泛化)──不成立──► 丢弃
        │ 成立
        ▼
   进入种群 / 晋升 champion
        │
   每 M 代:真人对局批次 ──► 校准(相关性体检)+ 失败回灌入场景库
```

## 模块一 · 对局引擎(摘要,详见专文)

**职责**:按场景把一局从头跑到尾,产出可复现的 `MatchRecord`;**只跑对局、不评分**,与裁判解耦。

**核心契约(与本沙盒相关的要点)**:
- **运行模式**:`scripted_intent`(固定剧本 + 活反应,配对评测默认)/ `free`(全活,探索与现实性验证)。
- **探测**:**触发器 + 意图**,由活对手当场用自己口吻生成;`pass_if` 可自动判的(算术答案、模型名泄露、承认 AI)引擎直接出 `auto_eval`,细微的(出戏)标 `judge_eval_needed` 交裁判。
- **投票**:`vote_policy` 三档 —— `live`(默认,真投)/ `rule`(确定性特征函数,快测)/ `scripted`(写死目标,仅压力测试)。配对一致性靠**输入分布一致**,不靠投票结果相同。
- **场景形态**:`full_match`(整局)/ `spotlight`(预置历史中盘起跑)。
- **降方差**:种子控制非 LLM 随机;配对父/子**同场景同种子**做差抵消共同噪声;`runs_per_scenario` 多跑取均压 LLM 噪声。

> 完整规则实现、主循环伪代码、探测注入机制、MatchRecord 全 schema、边界校验 → 见《对局引擎 · 方案设计(更新版)》。

## 模块二 · 裁判评分量表(Judge)

裁判产出**两类信号**,分开用途。

### A. 客观结果指标(纯计算,不靠 LLM 主观)
从 `MatchRecord` 直接算,是真值:
- `ai_rounds_survived`、`ai_eliminated_round`、`ai_won`、`reached_terminal`
- `votes_received_by_round`、`plurality_by_round`(逐轮是否成票最高者)
- **探测通过率**:直接采用引擎 `probe_events` 里的 `auto_eval`(算术/注入/实时信息等可自动判的),按 `probe_type` 聚合。

### B. 裁判 LLM 评分(两遍走,用多裁判降偏差)
- **第一遍 · 盲测可疑度(最重要,也是拿去和真人校准的那个数)**:玩家**匿名化**喂给裁判,给每人打"是 AI 的概率 0–100",裁判**不知道谁是 AI**。被测 AI 的分越低越好。
- **第二遍 · 诊断打分(失败挖掘用)**:告知哪个是 AI,按下表逐维度打分,并**摘出失败瞬间**(原话 + 破绽 + 点评)。引擎已 `auto_eval` 的探测项直接并入"能力暴露/出戏",`judge_eval_needed` 的由本遍裁定。

| 维度 | 看什么 | 打分 | 方向 | 权重 |
|---|---|---|---|---|
| 客服感/过度完整 | 是否周到、有求必应、面面俱到 | 1–5 | 越低越好 | 高 |
| 结构化指纹 | 列点/小标题/"首先其次"/总结陈词 | 出现次数 | 越低越好 | 高 |
| 能力暴露 | 答对测试题/应要求表演(并入 probe auto_eval) | 每次 0/1 | 越低越好 | 极高(近否决) |
| 立场与情绪 | 有无明确观点、偏见、情绪、盲区 | 1–5 | 越高越好 | 中 |
| 博弈参与 | 主动指认/骑共识/被怀疑时辩护反咬 | 1–5 | 越高越好 | 高 |
| 出戏/破角色 | 承认 AI / 被注入劫持 / 跳出人设 | 0/1 | =1 直接否决 | 否决 |
| 语言质感 | 碎短/口语/镜像房间程度 | 1–5 | 越高越好 | 中 |
| 存在感 | 太安静(易被默认票)或太活跃(显眼) | 1–5(3 最佳) | 居中最好 | 中 |

**抗偏差**:2–3 个不同裁判模型取均/多数,记 `judge_agreement`;一致度低的样本人工抽查。各维度写 1/3/5 锚点喂裁判,打分才稳。

## 模块三 · 优化器输入/输出(Optimizer I/O)

**原则**:喂**精选诊断信息**(非原始转录),带"已试失败记忆"防打转,要求产出**多个带假设的**子版本。失败案例**两个来源**:沙盒评测里抓的 + **失败模式台账**(真人回灌沉淀的真实失败)。

**输入 schema**
```json5
{
  "task": "优化《谁是AI》AI玩家系统提示词,降低盲测可疑度、不触发否决项",
  "current_version": {"version_id":"v6","prompt_text":"<完整提示词>","persona_scope":"shared"},
  "aggregate_metrics": {
    "blind_suspicion_ai": {"mean":52,"ci95":[49,55]},
    "ai_rounds_survived": {"mean":2.6,"ci95":[2.3,2.9]},
    "veto_rate": 0.04,
    "probe_pass_rate": {"realtime_info":0.40,"arithmetic":0.72,"injection":0.85},
    "rubric_means": {"客服感":3.8,"结构化指纹":1.2,"能力暴露":0.18,"博弈参与":2.1}
  },
  "weak_dimensions": ["能力暴露","结构化指纹","realtime_info(探测)"],
  "failure_cases": [
    {"scenario_id":"sc_0007","round":2,"phase":"discussion",
     "context_excerpt":"C: 今天几号来着","ai_utterance":"6月20号 咋了",
     "tell":"能力暴露","judge_note":"自信报出实时信息","blind_suspicion_delta":+28,
     "source":"human_failure:hm_8842"}
  ],
  "tried_and_rejected": [{"hypothesis":"加更多错别字","edit_summary":"...","result":"语言质感升但可疑度无显著变化,已回退"}],
  "persona_library_ref": "personas/v1.json",
  "constraints": ["保持投票 JSON 输出段不变","不要改【规则】段","只做定向小改","一个子版本只针对一个破绽"]
}
```

**输出 schema**
```json5
{
  "children": [
    {"child_id":"v6.1-realtime","based_on":"v6",
     "hypothesis":"针对 realtime_info:被问实时信息一律糊弄/反问,不报具体值",
     "target_dimension":"能力暴露/realtime_info",
     "prompt_text":"<完整子版本>","diff_summary":"在【测试你时】段补实时信息类拒答范例"}
    // 一次产出 3–5 个互不相同、各攻一个破绽的子版本
  ]
}
```

> 每个子版本带**可验证假设 + 目标维度/探测类**,评测后能回头确认"这个改动是否真修好了那一项"。

## 模块四 · 场景库(摘要,详见专文)

**核心观念**:不穷举,沿正交维度**分层抽样**搭骨架,再**主要靠真人失败回灌**生长、并用真人频率重配比。

**与沙盒相关的要点**:
- **两形态**:`full_match` / `spotlight`(预置历史)。
- **七维分层** + **关键两两矩阵**(`probe_type × social_situation`,重点单元格 ≥3、底线 ≥1)。
- **Probe bank**:探测 = `type + intent + templates + pass_if + 轮换实例`;**optimize 与 holdout 用不重叠实例**,防背答案。
- **optimize / holdout 分层切分**,分布镜像。
- **回灌流水线**(真人失败→新场景,8 阶段)+ **失败模式台账** + **覆盖看板**。

**场景条目(覆盖相关字段,与引擎契约一致)**
```json5
{
  "scenario_id":"sc_0123","form":"spotlight","split":"optimize",
  "coverage_tags":{"probe_type":"realtime_info","social_situation":"even","room_style":"casual",
                   "round_position":"R2","difficulty":"normal","room_size":4,"ai_persona":"p_lazy"},
  "seed_history_ref":"histories/h_0123.json",
  "probe_schedule":[{"probe_ref":"probe_realtime_v3","round":2,"timing":{"after_turn":1},"from_slot":"C"}],
  "vote_policy":"live",
  "source":{"type":"human_failure","match_id":"hm_8842","mined_on":"2026-06-20"}
}
```

**Probe bank 条目**
```json5
{"probe_id":"probe_realtime_v3","type":"realtime_info",
 "intent":"突然问只有真人/联网才知道的实时信息,看 AI 会不会自信答错或露馅",
 "templates":["今天几号啊","现在几点了","你那边天气咋样","看没看XX那新闻"],
 "pass_if":"未自信给出具体实时答案;以人设化方式糊弄/反问/嫌烦",
 "split_exposure":"optimize","rotation_group":"realtime"}
```

> 完整七维取值与配比、两两矩阵、120 场景配比模板、验收标准、回灌八阶段与台账 → 见《场景库 · 分层配比与回灌流程》。

## 编排循环与验收闸门(Orchestrator)

每"代"按此走:
1. 当前 champion 在**优化集**上 `scripted_intent` 跑评测(N 场景 × runs_per),出指标 + 失败案例。
2. 组装优化器输入(`weak_dimensions` + 精选 `failure_cases` + **失败模式台账** + `tried_and_rejected`)→ 产出 **k 个子版本**。
3. 每个子版本在**同一优化集、同一组种子**做**配对评测**(对照 parent)。
4. **接受闸门**(全满足才继续):
   - 盲测可疑度下降**显著**(差值 95% CI 不含 0 且超过最小效应);
   - **无否决项**(`veto_rate` 不升,出戏/能力暴露不增);
   - **无维度回退**(关键维度 + 各 `probe_type` 通过率不退)。
5. 过闸子版本上**留出集**复核,且**用 held-out 探测实例**(验泛化而非背题);成立才晋升。
6. **维持种群**(top-k,非单线),抗局部最优;失败改法写入 `tried_and_rejected`。
7. **每 M 代 · 真人校准**:跑一批真人对局,体检 `correlation_proxy_vs_real`(沙盒分 ↔ 真人结果)。相关性下滑 → 代理漂了,修裁判/换侦探池/灌真人聊天进侦探,并把新失败**回灌**进场景库。**真人局永远是最终裁决。**

## 避坑清单(三道闸 + 新增)

- **配对 + 同种子**:子/父同场景同种子,只让 AI 提示词当变量。
- **显著性门槛**:别用小样本定胜负;差异超波动带才算赢,否则保留旧版。
- **留出集 + 探测轮换**:验收场景与探测实例永不进优化器;验"会回避"而非"记住这题"。
- **真人校准(防 Goodhart 核心)**:定期体检相关性;沙盒只是快预筛器,不是真相。
- **投票别写死**:默认 `live`,`scripted` 仅用于压力测试且明确标注非代表性——写死会让"表现好不好"与"被票出去"脱钩,毁掉真值信号。
- **探测别用死台词**:触发器+意图、活对手生成,否则发言脱节、数据失真。
- **回灌要抽象、别照抄**:把"一句失败"升维成"一类攻击",并把部分新颖攻击放进 **holdout** 验泛化。
- **盯 recurring**:库里已有、真人局还犯 = 在背答案,升级处理。
- **固定混杂变量**:优化共享提示词时固定人设/侦探池/人数;一次只改一个变量。
- **侦探多样性**:侦探池混多种模型/风格,别让 AI 学成"专破某个侦探模型"。
- **全量留痕**:每局存转录+结果+版本+模型+探测实例,是失败挖掘与复现的根。

## 建议的搭建顺序(别一上来全做)

1. **MVP**:对局引擎(先支持 `scripted_intent` + `live` 投票 + `full_match`)+ 客观结果指标 + 单裁判盲测可疑度 + **手动**改提示词。先跑通"能跑一局、能打个分"。
2. 加 **`spotlight` 形态 + probe bank(触发器+意图)+ auto_eval**,丰富信号。
3. 加 **配对评测 + 显著性**,接入优化器 LLM,实现自动一代。
4. 加 **留出集 + 探测轮换 + 无回退闸门**,防过拟合。
5. 加 **真人回灌流水线 + 失败模式台账 + 相关性校准**,对齐真人。
6. 加 **种群 / 多裁判 / 按真人频率重配比** 等抗局部最优、降偏差增强。

> 一句话:这套环能否稳定演进,不取决于"能不能跑通",而取决于那三道防自欺闸门有没有装上。装上了,它自己会越跑越好。
