## CP-01 · 从"写好提示词"到"建可迭代优化系统"

- **触发**:在拿到第一版 AI 玩家提示词 + 人设卡后,你提出:这版不一定最优;拟人化和具体用哪个大模型有关;同一提示词不同模型效果不同;**甚至同模型同提示词、不同对局输出也不同**——如何保证提示词能持续稳定优化迭代?
- **改前状态**:项目目标是"调出一版尽量拟人的提示词",靠人工感觉评判。
- **关键洞察**:主观、无度量则只能凭感觉改来改去,无法稳定迭代。但这个游戏**自带客观真值**——真人投票就是一台免费的标注机。
- **决定**:整个项目从"调一版提示词"**转向"建一套度量驱动的优化系统"**。
- **波及**:催生了后续全部沙盒设计。
- **设计初衷**:把优化建立在**客观真值**上,而非感觉。这是所有后续工程的总起点。

## CP-02 · 提出离线自对弈沙盒(原创架构骨架)

- **触发**:你给出具体方案——让 LLM 自动对战、以初始提示词为种子版本、并行开 50 局、裁判大模型打分、再把这 50 局数据+提示词+人设喂裁判去优化出子版本、子版本再跑 50 局打分、对比前后决定是否晋升。
- **改前状态**:只确立了"要做优化系统",尚无具体架构。
- **关键洞察**:你的骨架正是成熟范式(self-play + LLM-as-optimizer / 进化式提示词搜索)的正确形态。但有三个洞:① 沙盒里的"对手"和"裁判"都是 LLM → 优化的是"骗过 LLM"而非"骗过真人"(Goodhart);② 50 局不足以把真进步和运气分开;③ 把 50 局原始数据一股脑喂优化器,不如喂精选诊断。
- **决定**:采纳骨架,补上**三道防自欺闸门**(配对+显著性、留出+轮换、真人校准)。
- **波及**:沙盒总体结构与编排循环成型。
- **设计初衷**:**闭环可行,但必须防自欺**——只让经得起重复检验的真增益沉淀。

## CP-03 · 质疑 semi_scripted → 重构为"固定剧本 + 活反应"

- **触发**:在配对评测里我提出 `semi_scripted` 模式(对手发言由场景库预置、按时点触发),你提出两个问题:① 预置发言可能与上下文毫无关联 → 对局数据失真,AI 基于失真场景的输出与真实对局偏差大;② 对手玩家的**投票**怎么通过场景库预置?
- **改前状态**:打算把对手"完全脚本化"以降低配对评测的方差。
- **关键洞察**:"把对手完全脚本化"和"对局要真实"无法同时成立。必须区分**可固定的**(与 AI 措辞无关的外部冲击)和**必须放活的**(对手对 AI 的反应、投票)。
- **决定**:① 探测从"死台词"改为**触发器 + 意图**,由活对手当场用自己口吻生成;② 投票改为 **`vote_policy` 三档**(默认 `live` 真投,绝不写死);③ 模式更名为"固定剧本 + 活反应"(scripted_intent)。
- **波及**:对局引擎、Schema 契约、场景库。
- **设计初衷**:**降方差不能以失真为代价**——只固定与 AI 措辞无关的外部冲击,反应放活;投票写死会让"AI 表现↔被票出"脱钩、毁掉真值。

## CP-04 · 追问覆盖与真人对齐 → 放弃穷举

- **触发**:你问——场景库要怎么写才能保证覆盖**所有**场景?怎么和真人的对局对齐?
- **改前状态**:隐含假设"场景库要尽量覆盖全"。
- **关键洞察**:对局状态空间组合爆炸,"覆盖所有"既不可达也不该追求;应换成"覆盖所有**重要失败模式**"+"用真人数据持续**校准**"。
- **决定**:沿正交维度**分层抽样**搭骨架;主要靠**真人失败回灌**生长、按真人频率重配比;用"代理指标 vs 真人结果"的**相关性**验收对齐;未覆盖的未知攻击靠提示词的**通用回避反射**兜底。
- **波及**:场景库专文(分层配比 + 回灌流程)。
- **设计初衷**:**覆盖度由真人真实失败分布背书,而非想象力**。

## CP-05 · 质疑并移除 `intent_schedule`(两连击,典型范例)

- **触发**:我在写契约时**新增**了 `intent_schedule`(逐轮给对手注入"本轮重点怀疑谁")。你先问:这字段之前设计里没提到、意义是什么、是不是和 `probe_schedule` 有点重复?我解释了"事件 vs 状态"的区别并为它补了"社交局势落地"的理由。你随即**第二次追问**:基于沙盒"降低方差、贴近真实"的总体原则,是不是 `probe_schedule` + `spotlight` 就够了?`intent_schedule` 反而不自然、可能增加方差。
- **改前状态**:契约里有 `intent_schedule` 和 `RosterSlot.base_intent`,用于人为控制社交局势。
- **关键洞察**:`probe`(突然甩测试)是**外生**冲击,与 AI 表现无关,可固定;但"怀疑谁"是对 **AI 表现的内生反应**,强行注入就**抹掉了"AI 演得好不好 → 会不会被怀疑"这条因果链**——而这正是评测要测的信号。它降的是表面方差,买来的却是**系统性偏差/失真**。
- **决定**:**删除** `intent_schedule` 与 `base_intent`;社交局势只经 `full_match` 自然涌现或 `spotlight` 预置历史实现。因该字段从未发布,直接当它没出现过移除。
- **波及**:Schema 契约、引擎(连带去掉 `injected_intent` 产物)、场景库。
- **设计初衷**:**不能抹掉"AI 表现→被怀疑"的因果链**;一个"看着方便"的旋钮若违背核心原则,就该拆掉。
- **备注**:这是你抓住"功能与原则不一致"的典型一击,也顺带修掉了我"夹带新增字段、未走升版程序"的流程问题。

## CP-06 · 推动把 Schema 抽成独立契约(并定范围"只收输入")

- **触发**:你指出当前引擎设计文档里没有 Scenario/Probe schema 的详细说明,问要不要补、补到单独文档还是直接放进引擎文档。
- **改前状态**:场景/探测的字段散在多份文档里,且略有出入(已在漂移)。
- **关键洞察**:契约是**多方共享的接口**(场景库生产、引擎解释、裁判读 pass_if),不该住进任何一方;塞进引擎文档会让 schema 被"引擎当前需要什么"带偏,且每改一个场景字段都在动引擎文档。
- **决定**:单独立《场景与探测 · Schema 契约》作**单一事实来源**;**你定范围"只收输入契约"**,MatchRecord 作为输出契约先留引擎(后又由你独立成《对局记录 · Schema 契约》)。其余文档的 schema 改为指针。
- **波及**:新增输入契约文档;引擎/场景库/总纲 schema 改摘要。
- **设计初衷**:**依赖只走契约;单一事实来源防漂移**。

## CP-07 · 质疑场景库与引擎是否该合并 → 维持拆分(强化依据)

- **触发**:你提出——场景库和对局引擎强相关、互相依赖,是不是不该分成两个模块?
- **改前状态**:二者已是独立模块,但耦合直觉让人怀疑该合并。
- **关键洞察**:**耦合 ≠ 该合并**。判据不是"相不相关",而是"是不是同质、会不会同步变、能不能独立替换与测试"——场景库是高频变的**数据**、引擎是低频变的**机制/解释器**,二者均不满足合并条件。真正强耦合的"探测触发器+意图"机制应**沉淀进契约层**,而非合并模块。
- **决定**:维持两模块,用稳定契约连接;把跨模块的共享约定收进 Schema 契约。
- **波及**:澄清而非改动;强化了契约层的定位。
- **设计初衷**:**用契约消化耦合,而不是把两个模块焊成大泥球**。

## CP-08 · 质疑"一次产 K 个候选"伤性能 → 单候选 + 外层派靶

- **触发**:你提出——优化器提示词要求一次产出 K 个候选(如 K=10),会不会影响大模型表现?它还能按要求准确生成这么多吗?
- **改前状态**:优化器一次调用产出 `children[]` 数组(K 个)。
- **关键洞察**:K 大时会在三维**同时塌**——格式(长输出退化、JSON 末尾断裂)、多样性(高价值破绽就那么几个,逼它凑数会换皮重复)、纪律(注意力摊薄,靠后候选开始整段重写/背答案);废品还白烧算力、拖累前几个好样本。
- **决定**:拆成 **K 次单候选调用**,多样性靠**外层枚举 K 个靶子**(破绽 × 算子)保证;你进一步明确:提示词短(约 50 行)就用**完整输出**、不上 diff 模式。`optimizer_temperature` 也随之从偏高调到中等(多样性已移到外层)。
- **波及**:优化器文档 §1 / §2.5 / §3 / §9 / §13。
- **设计初衷**:**单候选保质量,多样性靠可控的外层枚举而非模型即兴**。

## CP-09 · 追问"同一弱点配不同算子怎么选" → 类型约束 + 战绩排序

- **触发**:你指出 §2.5"同一弱点配不同算子、派 2 个用不同 `edit_type`"没说清**具体怎么选**,问是不是每个 `edit_type` 都选一次。
- **改前状态**:只说"配不同算子",未给选择规则。
- **关键洞察**:既不是穷举(7 个算子里多数与某破绽根本不搭),也不是随机;应是**两步**——① 破绽类型**约束**出 2–3 个对路算子;② 在对路算子里按**历史战绩**排序,取 **Top-2 做 A/B**。
- **决定**:新增"破绽类型 → 对路算子(默认优先级)"映射表 + `operator_scoreboard` 战绩表(Beta 平滑)+ 主弱点 Top-2 A/B、次要弱点 Top-1。
- **波及**:优化器文档 §2.6、战绩表记录、参数表。
- **设计初衷**:让"**哪类改法修哪类破绽最有效**"成为随迭代收敛的策略知识——把"假设验证回环"从单候选层抬升到**算子选择层**。

## CP-10 · 指出总纲缺两块 → 补齐跨模块接缝与参与方

- **触发**:你指出离线沙盒总纲是不是少了"对局记录 schema 契约"和"侦探玩家提示词模板"两部分。
- **改前状态**:MatchRecord 被有意留在引擎文档;侦探/填充玩家提示词一直挂账未成文。
- **关键洞察**:**侦探是真缺**(引擎、固定剧本+活反应、live 投票、难度档、裁判校准多处依赖,却从未成文);**MatchRecord** 是横跨引擎(产)与裁判(消)的接缝,和输入契约同质,应**提升为独立契约**而非留在生产方文档。
- **决定**:你自行补了《侦探玩家提示词》与《对局记录 · Schema 契约》;我补了《填充玩家提示词》,并把 filler 投票设计成**共识跟随(rule)**——因为你的侦探文档已指出"随机投票=噪声污染",共识跟随只放大已有怀疑、不注入噪声;总纲的文档地图与记录索引随之同步(MatchRecord 权威定义改指向输出契约)。

- **设计初衷**:**每个跨模块接缝都要有单一事实来源;每类对局参与方都要有提示词;filler 用共识跟随而非随机,以不污染信号**。

## 贯穿主线:一条反复出现的设计初衷

回看这 10 个变更点,**CP-03、CP-05、CP-08 是同一种介入**:面对一个"看着方便"的设计(脚本化对手 / 注入怀疑倾向 / 一次产 K 个),你都识别出它**违背了某条核心原则**(降方差不能失真 / 不能抹掉因果 / 不能牺牲候选质量),并要求砍掉或改掉。

CP-04、CP-07、CP-10 则是另一种介入:在**对齐真实**(覆盖真人失败)、**解耦**(契约消化耦合)、**完整性**(接缝有 SoT)上把设计推向更稳的形态。

把这两类合起来,你在整个过程中反复在守的,其实就是《总体设计》里那条信念——**只让经得起真人检验的真实改进沉淀下来,任何为了"方便"而引入失真、污染信号、降低质量的捷径都要被拦下**。这条主线,就是回溯本系统大多数关键设计时最该记住的设计初衷。
