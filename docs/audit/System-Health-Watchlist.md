## 1. 目的与范围

维护者的角色已从"执行迭代"上移到"监督闭环"。日常会盯的 5 件事——闭环是否顺利、迭代方向是否正确、胜率是否提高、拟人化指标是否提升、审计提案是否合逻辑——**必要但不充分**。本清单补齐这 5 件之外、**会让上述指标本身失真或失效**的系统级关注点，是维护者"更上层"那层的**固定看板**，与[审计章程](./CHARTER.md)并列。

- 章程管：design↔code 一致性的**自动闭环**。
- 本清单管：闭环本身**健不健康、指标可不可信**的**人工监督**。

## 2. 核心原则

> **你优化的全是代理（proxy）：LLM 裁判、脚本化对手、固定评测集。任何代理都会因为被优化而失真（Goodhart）。胜率与拟人化指标本身就是代理——所以监督的核心不是"指标涨没涨"，而是"指标还配不配被相信"。**

对照测试守的是"流水线机器没坏"，守不住"尺子在缩水"。把同样的怀疑延伸到**裁判、对手、评测集**，并定期拉真人校准，是本清单的主线。

> **当前快照（2026-07-01）**：闭环**已建好、但从未真跑过一代**——champion 仍是种子 `v0-baseline`、`sandbox_generation_evals` 0 行、真人校准一次没跑过。所以"自进化在迭代"目前是设想、不是现状；胜率曲线尚不存在，谈不上可不可信。先过 §4 的前置闸，再谈监督指标。证据见附录 §6。

## 3. 关注清单

三组，对应**度量有效性 / 退化模式 / 人与成本**。每条：**盯什么指标 · 触发什么动作 · 多久一次**。现状严重度：🔴 阻塞 / 🟠 重要 / 🟡 关注 / 🟢 已较好。

## 4. 开跑迭代前的前置闸（P0，先解决再信任任何胜率数字）

当前闭环尚未真跑过一代（§2 快照）。把"胜率 / 拟人化指标上升"当可信信号之前，**必须**先过这三关：

1. **跑通一代端到端**：确认 `run-generation → propose → 闸门 → 换 champion → 落 generation_eval` 全链真能完成（现有 0 证据）。
2. **插上反 Goodhart 保险丝**：建"真实对局 → `replay-backfill` → `calibration/run`"喂数据通道，**至少手动跑一次校准**让 `state.calibration` 有值。不插，闸门里那道冻结永不触发，之后每次"进步"都是凭信仰。
3. **定对手与裁判策略**：迭代前决定侦探多久增强 / 轮换（A2），并确保一次 run 内裁判冻结、换版有断代标记（A3）。
