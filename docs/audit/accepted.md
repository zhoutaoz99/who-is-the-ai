## 1. 概述

本文件登记**经人工裁定为故意设计或可接受**的审计发现。其作用见[章程](./CHARTER.md) §7.3：

- 审计 agent 每周期读取本清单；命中的 finding 标记 `accepted`，**不再当问题上报**，但应在周期报告概况中给出计数。
- 这是抑制 `confidence=low` 设计异味反复刷屏的主要手段。
- 仅**人**可向本清单增删条目；agent 只读。

## 3. 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `finding id` | string | 是 | 对应 `findings.jsonl` 中该发现的 `id`（签名），用于精确匹配 |
| `标题` | string | 是 | 该发现的一句话结论，便于人工辨认 |
| `接受理由` | string | 是 | 为什么这是故意设计 / 可接受；应能经得起复核 |
| `裁定人` | string | 是 | 做出接受裁定的个人或角色 |
| `裁定日期` | date | 是 | `YYYY-MM-DD` |
| `复核期限` | date | 否 | 到期后该接受失效、需重新裁定；留空表示长期接受 |

## 4. 维护规则

- 接受一条发现时，**应该**在 `接受理由` 中写清依据，避免日后无法追溯为何忽略。
- 当对应代码或设计发生变更时，相关接受条目**应该**重新复核；签名若因变更而改变，旧条目自然失效。
- 删除接受条目后，该 finding 会在下个周期重新按问题上报。
