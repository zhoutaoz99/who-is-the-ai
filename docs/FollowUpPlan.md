# 后续计划整理

本文统一整理项目后续计划，并按业务视角、实现视角和优先级归类。

## 优先级定义

- `P0`：影响核心可玩性、公平性或上线安全的必要事项。
- `P1`：补齐产品闭环、正式化运营和稳定性的关键事项。
- `P2`：提升留存、运营效率、AI 表现和复盘质量的增强事项。
- `P3`：中长期玩法形态、复杂能力或探索性方向。

## 开发阶段

| 阶段 | 目标 | 实现内容 | 优先级 |
| --- | --- | --- | --- |
| 第一阶段：无 AI MVP | 跑通完整游戏流程 | 用户登录、创建房间、加入房间、开始游戏、轮次倒计时、真人发言、投票出局、胜负判定。 | `P0` |
| 第二阶段：假 AI 接入 | 验证 AI 玩家接入点 | 后端创建 AI 玩家、AI 使用固定模板发言、AI 使用简单规则投票、真人与 AI 走同一套发言和投票流程。 | `P0` |
| 第三阶段：真实 AI Core | 让 AI 具备可玩性 | 接入真实模型、增加结构化输出、AI 发言策略、AI 投票策略、超时兜底和 AI 行为日志。 | `P1` |
| 第四阶段：积分与复盘 | 增强完整游戏闭环 | 真人获胜积分结算、积分流水、游戏复盘页、出局后上帝视角、每轮投票和发言记录回放。 | `P1` |
| 第五阶段：稳定性与扩展 | 支持正式上线 | Redis 存储实时房间状态、Socket.IO Redis Adapter 多实例、聊天限流、模型调用限流、异常重连、断线托管、日志和监控。 | `P1` |

## 业务视角

### P0：核心玩法和公平规则

| 事项 | 说明 |
| --- | --- |
| 明确平票处理规则 | 当前平票规则未定义；建议 30 秒重新投票，仍平票则本轮无人出局。 |
| 明确超时未投票规则 | 建议将超时未投票视为弃票，避免投票阶段卡死或产生争议。 |
| 明确掉线和异常行为处理 | 需要定义玩家掉线、异常重连、断线托管、恶意刷屏等规则。 |
| 增加无意义消息和作弊判定 | 需要判定系统限制作弊、刷屏或发送无意义内容，保护讨论质量。 |
| 提升 AI 对抗可信度 | 当前 AI 只获取当前轮次讨论记录，可能被真人玩家识别；需要补充跨轮次记忆和上下文能力。 |

### P1：完整游戏闭环

| 事项 | 说明 |
| --- | --- |
| 接入真实 AI Core | 从模板发言和规则投票升级到真实模型发言、投票、结构化输出和兜底策略。 |
| 增加账号和积分系统 | 接入持久化用户、积分余额和积分流水。 |
| 完善积分结算 | 真人获胜平分 2000 积分，并支持个人贡献奖励。 |
| 增加出局后体验 | 出局玩家进入上帝视角、继续观战，但不能影响在场玩家决策。 |
| 增加赛后复盘 | 展示每轮发言、投票、身份结果、出局顺序和积分变化。 |
| 支持房间规则配置 | 支持配置 AI 数量、真人数量、总轮数、单轮时长和奖励池。 |

### P2：运营和留存增强

| 事项 | 说明 |
| --- | --- |
| 增加排行榜 | 包括积分排行榜、周榜奖励等运营玩法。 |
| 增加每日游戏次数限制 | 限制每天游戏次数，防止无限刷局和积分通胀。 |
| 增加赛后推理评分 | 根据投票、发言和识别表现给真人玩家推理评分。 |
| 增加 AI 表现评分 | 对 AI 模拟玩家设置表现评分，例如成功存活到最后获得更高评分。 |
| 根据识别效率调整奖励 | 越早识别 AI，真人玩家获得更高奖励。 |
| 增加 AI 难度分级 | 让不同房间匹配不同强度的 AI。 |
| 增加 AI 人格系统 | 为 AI 设置不同人设和行为风格，提升局内差异。 |

### P3：玩法扩展

| 事项 | 说明 |
| --- | --- |
| 引入特殊行动或技能 | 例如查验、保护、沉默、禁言等有限次数能力。 |
| 增加语音发言模式 | 在纯文本 MVP 后探索语音发言。 |
| 扩展更复杂角色技能 | 在基础玩法稳定后加入更复杂的角色和技能体系。 |
| 扩展运营页面和 SEO 页面 | Next.js 方案中预留了运营页面、SEO 页面等完整产品形态。 |

## 实现视角

### P0：权威状态和基础安全

| 事项 | 说明 |
| --- | --- |
| 服务端作为唯一可信状态源 | 所有游戏状态、阶段切换、投票结算和胜负判定必须由后端完成。 |
| 服务端驱动倒计时和阶段切换 | 实时游戏中倒计时和讨论、投票阶段切换不能依赖前端。 |
| 完整校验 WebSocket 行为 | 校验登录、房间归属、玩家存活状态、当前阶段、发言冷却、投票目标和重复投票。 |
| 校验 AI 输出合法性 | 不能信任 AI 输出，必须校验结构、目标玩家、行动类型和内容长度。 |
| 增加 AI 超时兜底 | 模型超时时发言跳过，投票使用兜底策略，避免阻塞游戏流程。 |

### P1：持久化和正式运行能力

| 事项 | 说明 |
| --- | --- |
| 接入 PostgreSQL | 持久化用户、房间、对局、玩家、轮次、聊天、投票、AI 配置、AI 日志和积分流水。 |
| 接入 Redis | 存储实时房间状态、倒计时、限流信息，并为多实例 Socket.IO 做准备。 |
| 使用 Socket.IO Redis Adapter | 支持多实例 WebSocket 广播和房间扩展。 |
| 增加 JWT 和 WebSocket 鉴权 | 支持用户注册登录、JWT 签发校验和 Socket 连接鉴权。 |
| 积分结算使用事务 | 避免重复发奖、漏发和积分流水不一致。 |
| 持久化聊天和投票记录 | 支撑复盘、争议处理和日志分析。 |
| 增加异常重连和断线托管 | 提升实时游戏可用性，降低掉线对局影响。 |

### P1：真实 AI Core

| 事项 | 说明 |
| --- | --- |
| 接入真实模型调用 | 接入 OpenAI 兼容模型调用，并支持可替换模型配置。 |
| 强制结构化输出 | 发言、投票和跳过使用结构化 JSON，降低解析失败和非法输出风险。 |
| 拆分 AI Core 内部模块 | 建议拆分 `AiPlayerAgent`、`AiOrchestrator`、`PromptBuilder`、`ModelClient`、`ActionSchema`、`AiMemoryStore`。 |
| 增加 AI 行为日志 | 记录输入快照、输出动作、模型、耗时等信息，用于调试和复盘。 |
| 改进 AI 上下文 | AI 需要跨轮次聊天、投票历史、短期记忆和策略摘要，避免只依据当前轮次。 |
| 使用前缀缓存或提示缓存 | 当前大模型调用没有利用前缀缓存，会导致响应慢、Token 消耗高。 |

### P2：限流、监控和复盘分析

| 事项 | 说明 |
| --- | --- |
| 增加聊天限流 | 限制刷屏和异常消息频率。 |
| 增加模型调用限流 | 控制模型成本和并发压力。 |
| 增加日志和监控 | 支持正式上线后的问题排查和服务健康观测。 |
| 增加日志回放 | 用于复盘分析，并帮助提升 AI 胜率。 |
| 建设复盘页 | 前端展示身份、每轮发言、投票结果、出局顺序和积分变化。 |

### P3：复杂编排和多形态能力

| 事项 | 说明 |
| --- | --- |
| 评估 LangGraph | 当出现复杂多阶段推理、长期策略状态、AI 协作或多模型编排时再考虑引入。 |
| 支持语音或视频能力 | 基础纯文本玩法稳定后再考虑。 |
| 建设复杂角色和技能引擎 | 为查验、保护、沉默、禁言等技能提供可扩展规则模型。 |

## 建议落地顺序

1. `P0`：先补齐平票、弃票、掉线、刷屏、作弊判定等规则，并确保服务端权威状态和校验完整。
2. `P1`：接入真实 AI Core、PostgreSQL、Redis、账号鉴权、积分流水、复盘基础数据和断线恢复能力。
3. `P2`：完善排行榜、每日次数限制、AI 难度、人设、评分体系、日志回放、限流和监控。
4. `P3`：探索特殊技能、语音发言、复杂角色体系和 LangGraph 等高级 AI 编排。
