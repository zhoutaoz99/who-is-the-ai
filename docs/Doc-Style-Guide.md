# 文档规范

| 字段 | 内容 |
| --- | --- |
| 适用范围 | `docs/` 下所有正式文档；**不包括** `docs/ImplementationPlan/` 历史归档与 `docs/design/` 草稿 |
| 文档状态 | Active |
| 目标读者 | 项目维护者、开发、测试、评审者 |
| 最近核对日期 | 2026-06-30 |
| 目的 | 将当前文档统一到软件开发通用的、可审查、可追溯、可维护的标准格式 |

## 1. 现状诊断

本规范基于对当前 `docs/` 目录 8 篇正式文档（不含本文）的审阅得出，范围如下：

- `README.md`
- `Roadmap.md`
- `gameplay/*.md`

当前主要问题不是“内容不够多”，而是**文档类型、元数据和结构边界不够标准化**：

1. **文档类型未显式声明**  
   规则文档、设计文档、迭代记录、路线图和操作说明存在混写，读者进入文档前无法快速判断“这篇文档是讲规则、讲设计、讲现状，还是讲历史演进”。
2. **头部元数据不统一**  
   当前只有部分文档包含“最近核对日期”或“相关文档”，缺少文档状态、适用范围、责任人、目标读者、关联代码等标准字段。
3. **章节骨架不一致**  
   类似内容使用了不同章节名，例如“游戏简介 / 架构概览 / 一句话概览 / 背景与目标”，导致跨文档扫描成本高。
4. **“为什么、是什么、怎么做、做到哪一步”边界不清**  
   多篇文档同时包含设计动机、实现细节、现有限制、未来计划和历史迭代记录，缺少标准文档分型。
5. **可追溯性不足**  
   已有文档能链接代码，但多数没有形成“文档类型 -> 关键决策 -> 相关模块 -> 验证方式”的完整链路。
6. **专业文档格式不够稳定**  
   正式设计文档中仍大量使用口语式分段、非标准编号和临时性说明，不利于长期维护和评审。

本文件的目标不是要求所有文档写成同一种样子，而是要求它们**先归类，再按类型套模板**。

## 2. 规范级别

本规范使用以下术语：

- **必须**：不满足即视为不符合规范。
- **应该**：推荐做法；若不采用，文档中应说明原因。
- **可以**：可选做法，按文档类型和场景决定。

## 3. 编写原则

### 3.1 单文档单职责

一篇文档必须有且仅有一个**主类型**。允许补充次标签，但不能让一篇文档同时承担规则、设计、路线图和迭代记录四种职责。

### 3.2 面向读者

文档必须显式说明目标读者。面向开发者的设计文档、面向维护者的运维文档、面向产品/评审者的路线图，写法应不同。

### 3.3 事实与判断分离

- 事实：当前实现、当前配置、当前流程、当前数据结构。
- 判断：为什么这样设计、风险是什么、建议怎么改。

两者必须分开写，避免把“事实描述”和“作者观点”混成一段。

### 3.4 结构先于叙述

正式文档优先提供：

1. 文档元数据
2. 作用范围
3. 标准章节骨架
4. 表格、图、代码引用
5. 限制、风险、验证方式

长段叙述只能作为解释，不应取代结构。

### 3.5 可追溯

文档必须能回答以下问题中的至少一组：

- 这项规则对应哪个功能/模块？
- 这项设计落在哪些文件？
- 这个结论如何验证？
- 这个计划当前处于什么状态？

## 4. 文档分类

`docs/` 目录中的文档必须先归类，再按对应模板编写。推荐使用以下主类型：

| 主类型 | 目的 | 典型受众 | 当前目录示例 |
| --- | --- | --- | --- |
| `Overview` | 导航、索引、阅读路径 | 所有读者 | `README.md` |
| `Player Guide` | 面向玩家的玩法说明、教程、UI 帮助 | 玩家 | `gameplay/Player-Guide.md` |
| `Spec` | 规则、约束、功能定义 | 产品、开发、测试 | `gameplay/Gameplay.md` |
| `Design` | 架构、流程、数据、实现取舍 | 开发、评审者 | `gameplay/AI-Interaction-Flow.md`、`gameplay/AI-Human-Likeness-Design.md`、`gameplay/AI-Scheduling.md`、`gameplay/AI-Prompt-Cache-Optimization.md` |
| `Iteration Report` | 迭代记录、实验结论、经验沉淀 | 开发、评审者 | 当前暂无正式文档 |
| `Roadmap` | 分阶段目标、状态、依赖、优先级 | 维护者、协作者 | `Roadmap.md` |
| `Reference` | 接口、字段、配置、常量的查阅型资料 | 开发、测试、运维 | 当前暂无独立标准文档 |
| `Runbook` | 操作步骤、发布、排障、回滚 | 开发、运维 | 当前暂无独立标准文档 |
| `ADR` | 架构决策记录与取舍 | 开发、架构评审者 | 当前暂无独立标准文档 |

约束如下：

- 每篇文档**必须**声明一个主类型。
- 同一文档**不应该**混入大量不属于该类型的章节。
- 当文档内容跨类型时，**应该拆分**为多篇文档并互相链接。

## 5. 文件命名与标题

### 5.1 文件名

- 文件名**必须**稳定，避免频繁重命名造成链接失效。
- 文件名**应该**使用英文语义 slug，与代码和链接兼容。
- 文件名**不应**包含空格、中文、日期或“最终版/新版/临时”等临时后缀。

推荐形式：

```text
Topic-Subtopic.md
```

示例：

```text
AI-Interaction-Flow.md
AI-Human-Likeness-Design.md
AI-Prompt-Cache-Optimization.md
```

### 5.2 H1 标题

- 每篇文档**必须**只有一个 H1。
- 本仓库正文语言以中文为主；技术名词、协议名、代码标识符保留英文。
- 标题**应该**直接表达主题，不加“最终版”“详细说明文档”“实现方案设计”等冗余后缀。
- 成对文档**应该**使用“主题：子主题”的标准格式，而不是随意拼接。

推荐形式：

```text
# AI 玩家交互流程
# AI 拟人化设计
# AI 提示词缓存优化
```

## 6. 通用头部元数据

所有正式文档在 H1 下方都**必须**包含统一的元数据表，不再使用自由格式说明段替代标准头部。

### 6.1 必填字段

| 字段 | 说明 |
| --- | --- |
| `文档类型` | `Overview / Player Guide / Spec / Design / Iteration Report / Roadmap / Reference / Runbook / ADR` |
| `文档状态` | `Draft / Review / Active / Deprecated / Archived / Superseded` |
| `适用范围` | 该文档覆盖的功能、模块或目录 |
| `目标读者` | 谁应该读这篇文档 |
| `责任人` | 个人、团队或角色 |
| `最近核对日期` | `YYYY-MM-DD` |
| `关联代码` | 关键目录、模块、服务或页面 |
| `关联文档` | 上下游文档链接 |

### 6.2 推荐模板

```md
| 字段 | 内容 |
| --- | --- |
| 文档类型 | Design |
| 文档状态 | Active |
| 适用范围 | 普通对局中的 AI 发言与投票流程 |
| 目标读者 | 后端开发、评审者 |
| 责任人 | AI / Gameplay 维护者 |
| 最近核对日期 | 2026-06-15 |
| 关联代码 | `apps/api/src/ai/`、`apps/api/src/game/` |
| 关联文档 | [游戏玩法](./Gameplay.md)、[AI 发言调度](./AI-Scheduling.md) |
```

### 6.3 使用规则

- `最近核对日期` 只有在**实际按代码或事实重新核对后**才能更新。
- `文档状态` 变更时，**应该**同步更新文档正文中的表述。
- `Superseded` 文档**必须**指向替代文档。

## 7. 标准章节骨架

以下模板是 `docs/` 目录的标准格式。新文档必须从相应模板出发，而不是自由发挥。

### 7.1 Overview 模板

适用：索引、目录、阅读入口、文档总览。

必备章节：

1. `目的与范围`
2. `目录结构`
3. `阅读路径`
4. `文档清单`
5. `维护规则`

### 7.2 Spec 模板

适用：玩法规则、业务规则、功能规范、约束定义。

必备章节：

1. `背景`
2. `范围`
3. `术语与定义`
4. `规则或需求`
5. `边界条件与异常情况`
6. `非目标`
7. `验收标准`
8. `关联实现`

补充要求：

- 规则组或需要被单独追踪、单独变更、单独测试的规则项**应该**具备稳定标识，便于测试、评审和讨论。
- 正式规则或需求**应该**优先使用**语义化稳定 ID**，例如 `START-ACCESS`、`SPEAK-CONTENT`、`VOTE-BLIND-SIMULTANEOUS`。
- 若文档是产品规则或玩法规则，**应该优先给规则组编号**，不必为每个并列子句单独分配一个顺序号。
- 若某条内容不会被跨文档、测试、代码、PR 或评审单独引用，**不应该机械分配稳定 ID**。
- 若文档描述的是“当前产品规则”，则**不应**混入大量未来设想。

### 7.3 Player Guide 模板

适用：玩家可见的玩法说明、教程、UI 帮助文案。

必备章节：

1. `游戏目标`
2. `一局怎么进行`
3. `你可以做什么`
4. `怎么获胜`
5. `奖励规则`
6. `新手提示 / 常见问题`

补充要求：

- 文案**应该**面向玩家理解，而不是面向开发或实现。
- 文档**不应**出现内部模块名、代码路径、调试房术语和实现细节。
- 若需要同时提供“玩家版”和“内部版”，**应该**拆分为两篇文档。

### 7.4 Design 模板

适用：架构设计、流程设计、模块设计、评估流程设计、缓存策略设计。

必备章节：

1. `背景`
2. `目标`
3. `非目标`
4. `约束与假设`
5. `方案概览`
6. `详细设计`
7. `数据模型 / 接口 / 配置`
8. `备选方案与取舍`
9. `风险与失败模式`
10. `验证方式`
11. `已知限制`
12. `后续工作`

补充要求：

- Design 文档**必须**同时回答“为什么这样做”和“具体怎么做”。
- 涉及架构或流程时，**应该**配图；仅有文字描述通常不够。
- 已实现方案与设想方案**必须**明确区分。
- 关键设计决策**应该**具备可追踪标识，例如 `DEC-CONTEXT-INVALIDATION`、`DEC-BLIND-VOTE-HIDDEN`，或单独沉淀为 ADR。
- 假设、限制、背景说明默认**不需要**稳定 ID；只有在它们会被外部引用或独立追踪时才编号。

### 7.5 Iteration Report 模板

适用：提示词迭代记录、实验总结、拟人化改进过程、阶段性结论。

必备章节：

1. `背景`
2. `问题陈述`
3. `假设`
4. `改动项`
5. `实验设置或观察来源`
6. `结果`
7. `结论`
8. `下一步`

补充要求：

- 这类文档**应该**按“问题 -> 变更 -> 结果 -> 结论”组织，而不是混成设计文档。
- 历史迭代记录**可以**保留时间线，但应保证每轮记录结构一致。

### 7.6 Roadmap 模板

适用：阶段规划、优先级、交付路径。

必备章节：

1. `目标与范围`
2. `当前状态`
3. `优先级定义`
4. `分阶段计划`
5. `依赖与风险`
6. `里程碑 / 决策点`

补充要求：

- `Roadmap` **应该**只写未来或未完全完成的事项。
- 已完成事项**可以**保留，但应简写并链接到正式功能文档。

### 7.7 Reference 模板

适用：接口、字段、配置项、常量、状态码、数据结构。

必备章节：

1. `概述`
2. `实体 / 接口 / 配置清单`
3. `字段说明`
4. `示例`
5. `约束与兼容性`
6. `变更记录`

### 7.8 Runbook 模板

适用：部署、操作、发布、值班、故障处置、回滚。

必备章节：

1. `场景`
2. `前置条件`
3. `操作步骤`
4. `验证步骤`
5. `回滚方案`
6. `常见故障`
7. `升级 / 降级影响`

### 7.9 ADR 模板

适用：关键架构决策和取舍。

必备章节：

1. `背景`
2. `决策`
3. `备选方案`
4. `取舍理由`
5. `后果`
6. `状态`

补充要求：

- ADR **应该**一条决策一篇文档。
- 已废弃 ADR **不应**直接重写，应该新增新的 ADR 并标记 `Superseded`。

## 8. 章节与编号规则

- `Overview` 和短索引文档**可以**不编号。
- `Spec`、`Design`、`Runbook`、`ADR` **应该**使用阿拉伯数字层级编号，便于评审和引用。
- 同一篇文档内**必须**保持一种编号体系，不混用 `一、二、三` 与 `1. 1.1 1.2`。
- 标题层级**不应**超过 H4。
- 章节层级编号与规则/决策标识是两套体系：前者可用 `1. / 1.1`，后者只在**确有引用需求**时使用语义化稳定 ID，如 `START-SETUP`、`WIN-CONDITIONS`、`DEC-DOUBLE-STAGE-SPEECH`。

推荐形式：

```text
## 1. 背景
## 2. 目标
### 2.1 业务目标
### 2.2 技术目标
```

## 9. 内容表达规范

### 9.1 语言与术语

- 本仓库文档以中文为主。
- 英文术语首次出现时，**应该**给出中文语义或上下文。
- 缩写首次出现时，**应该**写全称或给出定义。

### 9.2 叙述方式

- 文档**应该**优先用结论句，不用聊天式表述。
- 规则、结论、限制、风险**应该**拆成列表或表格。
- 不要用“感觉上”“大概”“先这么做”作为正式文档表达，除非明确标记为假设。

### 9.3 事实来源

- 当前实现描述**必须**可在代码或运行行为中验证。
- 未落地方案**必须**明确标记为“计划 / 设想 / 待实现”。
- 环境变量、配置值、常量值**应该**区分“代码默认值”“示例值”“线上实际值”。
- 日期、版本号、超时时间、容量等量化信息**必须**使用明确格式和单位。

## 10. 图、表、代码与引用规范

### 10.1 图

- 流程、架构、时序、ER 关系**应该**优先使用 Mermaid 或可维护的文本图。
- 每张图后都**应该**补一段文字解释图的边界、关键节点和注意事项。
- 图不能替代表格和定义；字段、状态、约束仍要落回正文。

### 10.2 表格

以下内容优先使用表格：

- 配置项
- 常量
- 状态
- 数据字段
- 优先级
- 风险

推荐表头：

- 配置：`名称 | 默认值 | 说明 | 来源`
- 字段：`字段 | 类型 | 必填 | 说明`
- 风险：`风险 | 触发条件 | 影响 | 缓解措施`

### 10.3 代码块

- 代码块**必须**标明语言类型。
- JSON、HTTP、SQL、TypeScript、Bash 等分别使用对应语言标记。
- 示例代码**应该**尽量最小化，只保留说明问题所需内容。

### 10.4 代码引用

- 文件路径、模块名、类型名、环境变量、常量名一律使用反引号。
- 路径**应该**相对仓库根，例如 `apps/api/src/ai/ai.service.ts`。
- 描述代码时，**应该**同时说明“它负责什么”，不要只堆路径。

### 10.5 文档链接

- 站内文档**必须**使用相对路径链接。
- 新增、重命名或拆分文档后，**必须**回写 `docs/README.md`。

| --- | --- | --- |

## 12. 维护流程

### 12.1 何时必须更新文档

以下变更发生时，相关文档**必须**同步检查：

- 玩法规则变化
- 核心流程变化
- 模块职责变化
- API / 数据结构变化
- 配置项变化
- 评估逻辑或评分标准变化
- 路线图状态变化

### 12.2 更新要求

- 改代码时，**应该**同时检查是否影响现有文档。
- 只改措辞不改事实时，`最近核对日期` 可以不变。
- 改了事实、流程、配置、接口或结论时，核对后**必须**更新日期。

### 12.3 过期文档处理

- 过期但仍保留参考价值的文档，标记 `Archived`。
- 被新文档取代的文档，标记 `Superseded` 并链接替代文档。
- 不再适用且无保留价值的文档，按仓库管理策略删除。

## 13. 发布前检查清单

提交或大幅修订文档前，至少检查以下项目：

1. 是否声明了主类型和文档状态。
2. 是否补齐了统一元数据表。
3. 是否使用了对应类型的标准章节骨架。
4. 是否明确区分了已实现、计划中、历史结论。
5. 是否给出了关联代码和关联文档。
6. 是否列出了限制、风险或边界条件。
7. 是否能让目标读者在 30 秒内判断这篇文档是否与自己相关。
8. 是否更新了 `docs/README.md` 中的索引。

## 14. 最小模板

新文档可直接从以下模板开始：

```md
# <标题>

| 字段 | 内容 |
| --- | --- |
| 文档类型 | <Overview / Player Guide / Spec / Design / Iteration Report / Roadmap / Reference / Runbook / ADR> |
| 文档状态 | <Draft / Review / Active / Deprecated / Archived / Superseded> |
| 适用范围 | <模块、流程、功能或目录> |
| 目标读者 | <读者角色> |
| 责任人 | <个人 / 团队 / 角色> |
| 最近核对日期 | YYYY-MM-DD |
| 关联代码 | `<path>` |
| 关联文档 | [X](./X.md) |

## 1. 背景

## 2. 目标

## 3. 非目标

## 4. 详细内容

## 5. 风险与限制

## 6. 验证方式

## 7. 后续工作
```

本模板是最低要求。实际章节应根据文档主类型替换为第 7 节对应模板，而不是机械复用同一骨架。
